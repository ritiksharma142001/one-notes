Introduction to Kafka

Basic Terminology​

Definitions​

  1. What is a Source (in simple terms)
  • A source is any system or app that sends data.

Examples:

  • A website sending user click data
  • A sensor sending temperature readings
  • An app sending payment events
  1. What is a Target (or Sink)
  • A target is any system that receives and uses data.

Examples:

  • A database storing the data
  • A dashboard showing the data
  • A file system writing logs
  1. Problem Statement (Before Kafka)
  • Say you have:
    • 3 sources (App, Website, IoT Device)
    • 3 targets (Database, Dashboard, Alerting System)
    • If each source connects directly to each target, the system looks like this:
Source A ─┬─> Target X
          ├─> Target Y
          └─> Target Z

Source B ─┬─> Target X
          ├─> Target Y
          └─> Target Z

Source C ─┬─> Target X
          ├─> Target Y
          └─> Target Z
That’s 9 connections, and it becomes worse as you grow.
  1. What is a Distributed System
  • A distributed system is a system where:
    • Components (like sources/targets) are spread across multiple machines
    • They work together to complete a task
    • They can fail independently

Examples:

  • A cloud app where database, frontend, backend are on different servers
  • Kafka itself is a distributed system
  1. Problems Without Kafka (Tight Coupling)

  • Issue 1: Source Fails

    • If Source A goes down, all targets expecting its data break or miss data.
    • Targets don’t know why — they just stop getting data.
    • You would need to manually notify or code in failure detection.

  • If you add Source D, you must update every target to listen to it.

    • More code, more testing, more risk.

  • Issue 3: New Target Added

    • Every source now needs to send data to this new target.

  • Again, more changes in multiple places.

  • This Creates Tight Coupling

  • Tightly Coupled = Everything depends directly on everything else

  • You can’t add, remove, or update systems easily.

  • One change = many changes everywhere else.

Scaling Problems in Distributed Systems — and Why Request-Response Breaks Down​

As distributed systems grow, several key challenges emerge:

1. Handling Large-Scale Data Streams

The Problem: In modern systems, millions of events per second may come from multiple sources (apps, sensors, services). Traditional request-response communication (e.g., HTTP calls) can’t handle this volume efficiently.

Why Request-Response Fails:

  • Each sender waits for a reply — this slows things down.
  • It’s synchronous, blocking performance.
  • Doesn’t support high-throughput data ingestion.

Example: Imagine a mobile app with 10 million users sending logs:

  • With HTTP: Each event = 1 request + 1 response = system overload.
  • With Kafka (event streaming): App sends logs asynchronously, no need to wait.

2. Preventing Tight Coupling Between Services

The Problem: As more services are added (data sources, consumers, analytics, etc.), connecting them directly to each other creates a web of dependencies.

Why Request-Response Fails:

  • Services must know about each other’s existence, endpoints, formats.
  • Any change in one service may break others.
  • Adding a new consumer = update every producer = chaos.

Example:

  • Without Kafka:
    • Service A sends data to Service B, C, and D directly.
    • Add Service E? Now A must update to send to E too.
  • With Kafka:
    • Service A sends to Kafka.
    • B, C, D, E independently read from Kafka. No need to touch A.

3. Ensuring Fault Tolerance and Preventing Data Loss

The Problem: In a distributed system, machines fail. If a system goes down, we can’t afford to lose important data (transactions, logs, sensor readings, etc.).

Why Request-Response Fails:

  • If the target service is down, the request fails.
  • Data is lost unless you retry manually.
  • No built-in retry, buffering, or data durability.

Example:

  • Without Kafka:
    • Service A sends data to Service B.
    • If B is offline, data is lost.
  • With Kafka:
    • Service A sends to Kafka (which stores data).
    • When B comes back online, it can replay and catch up.

Summary: Why Kafka-like Messaging Systems Work Better​

ProblemRequest-Response LimitationKafka/Event Streaming Advantage
High-throughput data ingestionSlow, synchronousFast, async, scalable
Service evolution & flexibilityTight couplingLoose coupling through central broker
Fault tolerance & durabilityData loss on failureData stored, replayable, resilient

Final Thought​

Traditional request-response (like REST APIs) is great for small, interactive systems. But as you scale to large, event-driven, real-time distributed systems, you need tools like Kafka to:

  • Decouple producers and consumers
  • Handle huge volumes of data
  • Make your system resilient to failures

Kafka​

Apache Kafka — A Scalable, Fast, and Fault-Tolerant Messaging System​

Apache Kafka is a distributed event streaming platform used to build real-time data pipelines and streaming applications. It serves as a middle layer between data-producing applications (sources) and data-consuming applications (targets).

Kafka is based on the publish-subscribe (pub-sub) model, where producers publish messages and consumers subscribe to receive them.

Kafka is designed to be:

  • Horizontally Scalable
  • Fault Tolerant
  • High Throughput and Low Latency

Publish-Subscribe Model in Kafka

Kafka uses a pub-sub messaging pattern:

  • Producers send (publish) messages to Kafka topics.
  • Consumers read (subscribe) messages from those topics.
  • Producers and consumers are fully decoupled.

Example: Consider a system that processes online food orders.

  • The ordering app publishes an order event (Producer).
  • The kitchen display, delivery app, and billing system all subscribe to the order events (Consumers).
  • Each consumer processes the message in its own way, independently of the others.

Decoupling Sources and Targets

Kafka decouples systems by acting as a central data broker:

Without Kafka:

  • Source systems must directly connect to each target.
  • This leads to tight coupling, where every source must know about each target.
  • Difficult to manage as the number of services increases.

With Kafka:

  • Source systems send messages to Kafka (not directly to targets).
  • Target systems independently read from Kafka.
  • No direct dependency between producers and consumers.
  • Systems can evolve independently.

Kafka Is Horizontally Scalable

Horizontal scalability means you can add more Kafka servers (called brokers) to handle more data or traffic.

Kafka handles data in topics, which are split into partitions. Partitions are distributed across multiple brokers, enabling:

  • Parallel processing
  • Load distribution
  • Easy scaling by adding more brokers

Example: If one Kafka broker is overloaded, you can add another broker. Kafka automatically balances the load by distributing topic partitions to the new broker.

Fault Tolerance in Kafka

Kafka replicates each partition across multiple brokers. If one broker fails:

  • Other brokers with replicas take over.
  • Consumers and producers can continue operating without data loss.
  • This ensures high availability and durability.

Kafka Performance Features

1. Scalability to 100s of Nodes:

  • Kafka clusters can contain hundreds of brokers.
  • Easily handles enterprise-level data pipelines.

2. High Throughput:

  • Kafka can handle hundreds of megabytes per second.
  • Can process hundreds of thousands to millions of messages per second.

3. Low Latency:

  • Kafka is optimized for fast message delivery.
  • Typical latency is as low as 2 milliseconds from producer to consumer.

Summary​

Horizontal ScalabilityAdd brokers to handle more load and partitions

Apache Kafka is widely used in modern distributed systems for building scalable, real-time, fault-tolerant data pipelines and event-driven architectures.

When to Use and NOT Use Apache Kafka

Apache Kafka is a powerful distributed streaming platform, but it is not the right fit for every use case. Below is a clear guide on where Kafka excels and where it should be avoided.

Use Kafka For:​

  1. Real-Time Data Streaming Kafka is ideal for streaming real-time data from multiple sources to consumers.

Example: Streaming click data from a website to a real-time analytics dashboard.

  1. Log Aggregation Kafka can collect logs from various services and centralize them for monitoring or analysis.

Example: Microservices send logs to Kafka, and a monitoring system reads from it to detect issues.

  1. Event Sourcing Kafka stores a series of events that represent changes in state. These events can be replayed to reconstruct the system state.

Example: An e-commerce system records all order events (created, shipped, cancelled) into Kafka to build an order history.

  1. Messaging Kafka acts as a message broker between loosely coupled systems, offering durability and scalability.

Example: A payment service sends a transaction event to Kafka. Multiple services (billing, inventory, notification) consume that message.

  1. Batch Data Processing Kafka supports storing and replaying data, which is useful for periodic processing in batches.

Example: A data pipeline reads Kafka messages every hour, transforms them, and loads into a data warehouse.

Do NOT Use Kafka For:​

  1. Simple Request-Response Communication Kafka is not designed for synchronous interactions or direct responses.

Example to Avoid: Calling a Kafka topic instead of an API to get a user’s profile details on button click.

  1. Small-Scale Projects Kafka requires infrastructure, setup, and maintenance that may be overkill for small projects.

Example to Avoid: Using Kafka in a simple web app with one form submission per day.

  1. High Latency Tolerance Kafka is optimized for low latency, real-time streaming. If your use case is tolerant of long delays, simpler systems may be better.

Example to Avoid: Sending monthly reports where a simple batch job would work fine.

  1. Monolithic Applications Kafka fits well with microservices and event-driven architecture. Monoliths with in-process communication don’t benefit much from Kafka.

Example to Avoid: A single-server app using Kafka internally instead of function calls.

Real-World Example

LinkedIn uses Kafka to process 1 billion messages per day.

They use it to handle activity streams, log collection, and real-time analytics across their platform with massive scale and reliability.

Kafka is a powerful tool when used in the right scenarios. It shines in systems that require real-time, scalable, and decoupled data processing across multiple services.

Common Message Queuing Systems​

Message queues allow systems to communicate asynchronously by sending messages that are stored until the receiver is ready to process them. Below are some widely used message queuing systems:

  1. RabbitMQ
  • Written in Erlang
  • Acts as a traditional Message Queue Server
  • Stores messages (jobs) in memory
  • Based on the Advanced Message Queuing Protocol (AMQP)
  • Supports features like message acknowledgments, routing, retries, and more

Use Case: Reliable message delivery between microservices with strong delivery guarantees.

  1. Apache ActiveMQ
  • Open-source message broker written in Java
  • Also based on AMQP, MQTT, and STOMP protocols
  • Supports both queue-based and publish-subscribe messaging models
  • Integrates well with Java-based systems

Use Case: Enterprise-level applications requiring robust message delivery and Java ecosystem integration.

  1. Apache Kafka
  • Originally developed at LinkedIn
  • Written in Scala and Java
  • Designed for high-throughput, real-time data streaming
  • Uses persistent storage by default and leverages OS disk cache for hot data
  • Typically used for event streaming, not just traditional queuing
  • Supports distributed processing and massive scalability

Use Case: Real-time analytics, log aggregation, stream processing, and large-scale message ingestion.

  1. Amazon SQS (Simple Queue Service)
  • A fully managed message queuing service provided by AWS
  • Supports standard queues (high throughput means process, receive data ie. more data in less time) and FIFO queues (ordered delivery)
  • Handles scaling, availability, and fault tolerance automatically
  • Suitable for serverless, microservices, and distributed applications

Use Case: Cloud-native applications needing a scalable, easy-to-use queuing system without managing infrastructure.

Kafka Components and Architecture​

What is a Kafka Broker?

A Kafka broker is a single Kafka server.

  • It receives data from producers, stores it on disk, and serves data to consumers.
  • Each broker handles a portion of the total data using topics and partitions.

Example:

If you have a topic with 3 partitions and 3 brokers, each broker may store one partition.

What is a Kafka Cluster?

A Kafka cluster is a group of Kafka brokers working together.

  • The cluster distributes data (partitions) across the brokers.
  • One broker becomes the controller, which manages cluster metadata (e.g., who is leader for each partition).

Why have a cluster?

  • Scalability: Add more brokers to handle more data.
  • Fault Tolerance: If one broker fails, others keep working.

What Was ZooKeeper’s Role?

Historically, Kafka used Apache ZooKeeper to:

  • Manage broker metadata (e.g., which broker is controller)
  • Perform leader election
  • Track the health/status of brokers
  • Store configuration and cluster info

But ZooKeeper had some issues:

  • Added complexity: Kafka + ZooKeeper = more moving parts
  • Harder to scale and upgrade
  • ZooKeeper could become a bottleneck or single point of failure

What is Kafka KRaft Mode (Kafka Raft Metadata Mode)?

Kafka is now moving away from ZooKeeper and using its own built-in consensus system, called KRaft (Kafka Raft).

KRaft = Kafka + Raft protocol, a distributed consensus algorithm.

In KRaft Mode:

  • Kafka maintains metadata internally (no ZooKeeper needed)
  • Elects a controller broker via the Raft protocol
  • Is simpler to deploy and manage
  • Has better performance and stability

Key Benefit:

Kafka becomes self-managed, easier to operate, and more reliable at scale.

Key Terminologies​

Kafka Core Concepts

  1. Producer

A Producer is an application or service that sends data (messages/events) to Kafka.

  • It publishes messages to a Kafka topic.

Example:
A payment service sends a message every time a transaction is made.

  1. Consumer

A Consumer is an application or service that reads data from Kafka topics.

  • It subscribes to a topic and processes incoming messages.

Example:
An analytics service consumes payment events to generate real-time reports.

  1. Event

An Event (also called a message or record) is the unit of data in Kafka.

It contains:

  • Key (optional): Used for partitioning
  • Value: The main data
  • Timestamp: When the event was produced

Example:

{
  "key": "user123",
  "value": "payment of $100",
  "timestamp": "2025-05-13T10:00:00"
}
  1. Broker

A Broker is a single Kafka server.

  • It receives, stores, and serves messages.
  • Multiple brokers work together in a cluster.

Example:
You may run 3 Kafka brokers, each storing different partitions of data.

  1. Cluster

A Kafka Cluster is a group of brokers working together.

  • It distributes messages across brokers using topics and partitions.
  • One broker is chosen as the controller to manage cluster-level tasks like leadership election.

Why use a cluster?​

  • Scalability: Add more brokers to handle more traffic.
  • Fault tolerance: If one broker fails, others take over.

Kafka Topic?​

A Topic in Kafka is like a category or stream where data (messages) is stored and organized.

Think of it like a table in a NoSQL database — just like you store data in different tables (like users, orders, etc.), in Kafka you store events in different topics.

How Topics Work

  • Each topic is split into smaller parts called partitions.
  • This allows the topic's data to be spread across multiple servers (called brokers).
  • This helps with scaling and parallel processing.

What is a Partition?

A partition is like a slice of the topic's data.

  • It helps Kafka distribute and store data more efficiently.
  • Each message in a partition gets a unique number called an offset, which acts like a row number or ID.

Offsets – Like Row Numbers

  • Kafka assigns every message a unique number called an offset.
  • The offset tracks the message’s position within a partition.
  • Together, the topic + partition + offset uniquely identify any message in Kafka.

Example: E-commerce Application

Let's say you are building an e-commerce app. You can create topics for different types of events:

  • order-placed: whenever a customer places an order
  • order-shipped: whenever an order is shipped
  • inventory-updated: whenever stock levels change
  • payment-processed: whenever a payment is completed

Each of these topics can be split into multiple partitions and processed independently by different services.

Why Use Topics and Partitions?

  • Topics keep your data organized by category.
  • Partitions make your system faster and more scalable.
  • Offsets let you keep track of what was processed and resume from where you left off.

In Simple Words:

  • Topic = folder or category of messages (like “orders” or “payments”)
  • Partition = a slice of that folder (used for parallel work)
  • Offset = line number or index of each message in that slice

Kafka Concepts Explained Using the Example​

  1. Producer

Forbes, Vogue, and Fortune India are like producers.
They publish articles (events/messages) under the "sports" topic.

  1. Topic

The "Sports" floor represents a Kafka Topic named sports.
It contains all content related to sports.

  1. Partitions

Each room under the "Sports" floor is a partition — a slice of the topic.

This allows parallel processing.

#Example:

  • Room 1 → Partition 0
  • Room 2 → Partition 1
  • Room 3 → Partition 2
  • Room 4 → Partition 3

Each room has its own order of messages, and each message inside has a unique offset.

  1. Brokers

The magazine stores (Broker 1 and Broker 2) store and manage these rooms.

  • Broker 1 has: Room 1, Room 2
  • Broker 2 has: Room 3, Room 4

They work together in a cluster to manage the full topic.

  1. Cluster

All brokers (stores) are part of one Kafka Cluster.

The cluster:

  • Distributes partitions of the topic across different brokers
  • Enables scaling and fault tolerance
  1. Consumers

Sanuj, Ram, and Bob are consumers.

They visit different rooms (partitions) to read articles.

Kafka can assign each consumer to a partition, so they can read in parallel.

Example:

  • Sanuj reads from Room 1 (Partition 0)
  • Ram reads from Room 2 (Partition 1)
  • Bob reads from Room 3 (Partition 2)

If there's more readers or more data, Kafka can spread it further across more partitions and brokers.

Why This Architecture?​

  • Parallelism: Multiple rooms (partitions) allow multiple consumers to read at the same time.

  • Scalability: You can add more rooms (partitions) or stores (brokers) as your magazine grows.

  • Efficiency: Each room has an ordered set of magazines (messages), so nothing gets mixed up.

  • Resilience: If one store (broker) goes down, the others can still serve data.

Kafka Partitions Explained (In Simple Terms)

What is a Kafka Partition?​

Kafka topics are split into smaller pieces called partitions.

A partition is the basic unit of parallelism in Kafka.
Each partition stores a sequence of messages in the exact order they were received.

Key Characteristics of a Partition​

  • Ordered:
    Messages inside a partition are always stored in the order they arrive.

  • Immutable:
    Once a message is written to a partition, it cannot be changed.

  • Offset-Based:
    Each message in a partition is given a unique offset (a number like 0, 1, 2...).
    This offset helps Kafka track the position of each message, like a line number.

Why Use Partitions?​

  1. Scalability
    Kafka can spread partitions across multiple brokers (servers).
    This allows you to scale horizontally — simply add more brokers when needed.

  2. Parallel Processing
    Multiple consumers can read from different partitions at the same time.
    This improves speed and efficiency, especially with high message volume.

Example: E-commerce Order Events​

Let's say we have a Kafka topic called order-events.

We can divide this topic into 4 partitions:

  • Partition 0 → stores order events 0, 4, 8, ...
  • Partition 1 → stores order events 1, 5, 9, ...
  • Partition 2 → stores order events 2, 6, 10, ...
  • Partition 3 → stores order events 3, 7, 11, ...

If we have 4 consumers, each one can read from a different partition at the same time — making the system fast and efficient.

Summary​

  • Partition = slice of a Kafka topic
  • Offset = position of a message in a partition
  • Parallelism = multiple consumers process partitions independently
  • Scalability = partitions can be spread across multiple Kafka brokers

What is an Offset in Kafka?

In Kafka, messages are stored in partitions, and each message has a unique number called an offset.

  • Offset is a number that shows the position of a message in a partition.
  • The first message has offset 0, the next is 1, then 2, and so on.
  • Offsets help Kafka keep track of where each message is.

How Consumers Use Offsets

When a consumer wants to read messages, it can choose where to start by selecting an offset:

  • Offset 0 → start from the beginning of the partition.
  • Latest offset → start from the most recent message.

Offsets allow consumers to remember what they’ve read and continue from the correct position.

How Kafka Handles Partitions and Message Consumption

  • When a topic has multiple partitions, Kafka assigns messages to partitions in a round-robin way by default.
  • Users can also create custom partitioning logic, like assigning messages based on a message key (e.g., user ID).

How Consumers Work

  • Consumers read messages from one or more partitions.
  • They keep track of the last offset they read, so they know where to continue next time.

Message Retention in Kafka

  • Kafka does not delete messages right after they are read.
  • Instead, it keeps messages based on a retention policy:
    • For example: 7 days or 100 GB per partition.
  • This means consumers can re-read old messages by going back to an older offset — helpful for reprocessing data.

Kafka Event Flow: From Producer to Consumer Group

  1. Producer Sends Events
  • Producers send messages (events) to a Kafka topic.
  • Each message can include:
    • A key (optional)
    • A value (the actual content)
  1. Topic and Partitions
  • A Kafka topic is split into partitions.
  • Events are stored in partitions in append-only logs.
  1. Partitioning Logic
  • No key: Kafka assigns partitions in round-robin.
  • With key: Kafka uses a hash of the key to assign the partition — this ensures ordering per key.
  1. Offsets
  • Each message in a partition has a unique offset (starting from 0).
  • Offsets increase as new messages are added.
  • Offsets are per partition, not per topic.
  1. Consumer Groups
  • A consumer group is a set of consumers working together.
  • Kafka assigns partitions to consumers.
    • Each partition is read by only one consumer in the group.
  • Consumers track the last offset they read.
  1. Retention and Reprocessing
  • Kafka keeps messages for a configured time (e.g., 7 days), even after consumption.
  • Consumers can:
    • Re-read messages from older offsets
    • Recover from failures
    • Reprocess for analytics
