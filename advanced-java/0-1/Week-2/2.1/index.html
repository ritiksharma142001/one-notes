Spring Boot Web

Introduction​

  • Notes

Introduction to Spring Boot Web​

  • This is the introductory session for Spring Boot Web.
  • Focus areas include:
    • Spring Boot Web dependency
    • Creating web services
    • Developing RESTful APIs

Why Web Services Are Important​

  • In earlier days, data and services were accessed physically at a specific location.
  • With the advent of the internet, data can now be accessed remotely, even from thousands of miles away.
  • Example:
    • When you open Instagram, the feed/reels you see are not stored on your device.
    • They are fetched from a remote database via web services (APIs).

How Data Is Accessed​

  • A client makes a request to a web service.
  • The web service communicates with the database.
  • The response (data) is sent back to the client.
  • This process happens over the internet, without any physical wire connection.

Role of APIs​

  • APIs (Application Programming Interfaces) act as bridges between:
    • The frontend (client)
    • The backend (server & database)
  • As a backend developer, your responsibilities include:
    • Building APIs and web services
    • Managing data access and business logic

Backend Developer Responsibilities​

  • You are responsible for ensuring that:
    • APIs are highly available (no single point of failure)
    • APIs are robust and scalable
  • Core tasks:
    • Designing and building APIs
    • Managing and securing data
    • Ensuring reliability and performance

Prerequisites​

  • Make sure you understand key Spring Boot fundamentals:
    • Dependency Injection (DI)
    • Bean creation and management
    • Spring Boot application lifecycle
  • These basics form the foundation for advanced topics like:
    • Kafka integration
    • Microservices deployment
    • High-scale application development

Backend-Only Focus​

  • This course/module focuses solely on the backend:
    • No frontend development (HTML, CSS, JS, etc.)
    • Tasks include managing databases, building web services, and exposing RESTful APIs

RESTful API Development​

  • You will start by building REST APIs to handle HTTP requests.
  • In the upcoming module, you will focus on:
    • The data layer
    • Using Spring Data JPA to interact with databases
    • Performing complex data

REST APIs​

What Are APIs?​

  • API stands for Application Programming Interface.
  • APIs are specifications that define how data should be transferred between:
    • A web service and a client (e.g., browser, mobile app)
    • Or between two web services (service-to-service communication)

Types of APIs​

  • APIs come in various types and standards, including:
    • REST (Representational State Transfer) – most widely used
    • SOAP
    • WebSockets – used for real-time, two-way communication
  • Each type has its own:
    • Methodologies
    • Terminologies
    • Documentation and standards (e.g., as defined in Spring Boot)

Why Learn REST APIs First?​

  • REST is the most commonly used API type.
  • Learning REST provides a strong foundation:
    • Makes it easier to understand and implement other API types.
    • Once REST is clear, working with other APIs becomes simpler.

What is REST?​

  • REST stands for Representational State Transfer.
  • REST APIs follow a set of rules and conventions to:
    • Build
    • Interact with
    • Consume web services

Key Characteristics of REST APIs​

  • Stateless:
    • Each API request contains all necessary data.
    • The backend does not maintain any session state.
    • No tracking of user session or history is needed on the server side.
    • This makes REST services more scalable and easier to manage.
  • Self-contained Requests:
    • Each API call is independent.
    • It includes all required information to process the request.

Introduction to HTTP Request Methods​

  • In REST APIs, different HTTP request methods are used to perform specific tasks.
  • Each method maps to a CRUD operation: Create, Read, Update, and Delete.

Types of Request Methods​

  • GET

    • Used to retrieve data from the server.
    • Example: Fetching a list of posts, users, or products.

  • POST

    • Used to create a new resource on the server.
    • Example: Uploading a file, submitting a form, adding a new user.
    • When you upload a reel on a platform like Instagram, you're creating a resource via a POST request.

  • PUT

    • Used to update an existing resource completely.
    • Example: Updating the content or metadata of an uploaded video.
    • If you want to replace the entire resource (like the whole reel), use PUT.

  • PATCH

    • Used to partially update a resource.
    • Example: Editing just the caption of a reel without touching the video itself.
    • PATCH is efficient when you only want to modify a small part of a resource.

  • DELETE

    • Used to delete a resource from the server.
    • Example: Deleting a user, comment, or a post.

CRUD Operations​

  • The above methods allow you to perform CRUD operations:
    • C – Create → POST
    • R – Read → GET
    • U – Update → PUT / PATCH
    • D – Delete → DELETE
  • As backend developers, it's our responsibility to implement all CRUD operations using these REST APIs.

spring-boot-starter-web​

Spring Boot Starter Dependencies​

  • Spring provides starter dependencies that are pre-configured for developers.
  • One such dependency is:
    • spring-boot-starter-web, which we will use to build REST APIs.

Project Setup Using Spring Initializr​

  • Open Spring Initializr to bootstrap your project.
  • Set up project configuration:
    • Build Tool: Maven
    • Language: Java
    • Spring Boot Version: 3.3.0 (or latest stable)
    • Group ID: Define your coding structure (e.g., com.example)
    • Artifact: spring-boot-web-tutorial
    • Name: spring-boot-web-tutorial
    • Packaging: JAR
    • Java Version: 22 (or your preferred version)
  • Add the dependency:
    • spring-boot-starter-web
  • Other dependencies like Lombok, Spring Data JPA, etc., can be added later from Maven Central.

What’s Inside spring-boot-starter-web​

Once the project is generated and opened in IntelliJ:

  • Navigate to pom.xml and observe the dependencies brought in:
    • spring-boot-starter
    • spring-boot-starter-json
    • spring-boot-starter-tomcat (embedded web server)
    • spring-web
    • spring-webmvc (for MVC architecture)

Included Core Dependencies Explained​

  • Spring Boot Starter
    • Contains basic dependencies like IOC container, dependency injection, and bean management.
  • Jackson
    • Used to automatically convert JSON/XML ↔ Java Objects.
    • Handles serialization/deserialization so you don’t have to manually parse JSON.
  • Spring Core
    • Provides essential components of the Spring Framework.
  • Spring MVC (Model-View-Controller)
    • Supports scalable architecture.
    • Ensures code is organized and maintainable.
  • Spring Boot Starter Tomcat
    • Embedded Tomcat server.
    • No external server configuration is needed.
    • Default configurations are provided but can be customized.

Running the Application​

  • Go to:
src/main/java/com/example/springbootwebtutorial/SpringBootWebTutorialApplication.java
  • Run the application.
  • Console output will show:
    • Embedded Tomcat server started on port 8080 (default).
    • We didn’t define this port—it's auto-configured by Spring Boot.

Accessing the Local Server​

  • Your Spring Boot application is now running on:
http://localhost:8080

  • Since no route is defined yet, visiting this URL shows an error related to /error.

  • This is expected and will be fixed once we define endpoints.

Key Benefits of Spring Boot Starter Web​

  • No need for external server setup.
  • Auto-configuration for most common use cases.
  • Enables rapid development of RESTful services.
  • Pre-configured dependencies for JSON handling, web server, and MVC architecture.

Spring Boot MVC Architecture​

Understanding Spring Boot MVC Architecture​

  • Introduction to Dependencies

    • We previously looked at the embedded Tomcat server that comes with the spring-boot-starter-web.
    • Another important dependency included is Spring Boot MVC, which follows the MVC architecture.

  • What is MVC?

    • MVC stands for Model, View, Controller.
    • It is an architectural pattern, not specific to Spring Boot.
      • You can implement MVC in Node.js, Android, or any other framework.

  • Purpose of MVC:

    • Promote separation of concerns.
    • Improve code maintainability and scalability.

  • MVC Layers Explained

    • Model: Represents the data layer (e.g., entities, repositories).
    • View: The output or response sent to the client (UI, JSON, etc.).
    • Controller: Handles incoming requests, interacts with the model, and returns the appropriate view.

  • Request Flow in MVC Architecture

    • Clients (e.g., browser, Postman) send HTTP requests.
    • The Tomcat server processes these HTTP requests.
    • Controllers are the entry point for all requests.
      • They handle different data types like JSON or XML.
      • Spring Boot uses Jackson to convert JSON/XML to Java objects.

  • Controller Responsibilities

    • Extract data from requests.
    • Perform logging.
    • Implement security/authentication checks.
    • Pass valid data to the Service Layer.

  • Service Layer Responsibilities

    • Contains the business logic.
    • Example: If a user likes a post (reel), the controller passes the reelId and userId to the service.
    • Service layer checks if the operation is allowed and performs the action.
    • Communicates with the Repository Layer.

  • Repository Layer Responsibilities

    • Represents the data access layer.
    • Contains repositories that interact with the database using entities (mapped to DB tables).
    • Performs CRUD operations.

  • Entities and Database

    • Entities are Java classes that represent database tables.
    • Operations like create, read, update, delete are handled via repositories.
    • Best practice: Do not access the database directly from the service layer; use repositories instead.

  • Why Layered Architecture?

    • Ensures each layer has a well-defined responsibility.

      • Controller: Handles requests, logging, security, data transformation (DTO ↔ Entity).
      • Service: Contains the business logic.
      • Repository: Performs data operations.

    • Promotes clean, scalable, and testable code.

Layered Architecture​

A scalable and maintainable Spring Boot application is typically divided into three main layers:

1. Presentation Layer​

  • This is the first point of contact for incoming HTTP requests.

  • In Spring Boot, this is typically represented by the Controller classes.

  • Common responsibilities:

    • Handle HTTP requests/responses.
    • Perform authentication checks to ensure requests are legitimate.
    • Convert incoming JSON into Java objects (done automatically using the Jackson library).
    • Prevent unauthenticated requests from reaching deeper layers.

  • Note: In some companies, this is referred to as the “presentation layer” instead of “controller layer.”

2. Business Layer​

  • Also known as the service layer.

  • This layer contains the business logic of the application.

  • Common responsibilities:

    • Validate the incoming data.
    • Apply authorization rules (e.g., only admins can delete users).
    • Handle business use cases (e.g., calculating discounts, processing orders).

  • Authentication vs Authorization

    • Authentication (checking if a user is logged in) happens in the presentation layer.
    • Authorization (checking if the user has the right to perform an action) happens in the business layer.

3. Persistence Layer​

  • Also known as the data access layer (DAO or repository layer).
  • Responsible for interacting with the database.
  • Common tools and technologies:
    • Uses JDBC drivers, Hibernate, and Spring Data JPA.
    • Implements CRUD operations (Create, Read, Update, Delete).
    • Defines queries using JPQL (Java Persistence Query Language), which is translated to SQL by Hibernate.

Benefits of Layered Architecture​

  • Separation of Concerns: Each layer handles a specific responsibility, making the code modular.
  • Easier Debugging: Errors can be traced to a specific layer quickly.
  • DRY Principle (Don’t Repeat Yourself):
    • Business logic is reusable across multiple controllers.
    • Persistence logic is reusable across multiple service methods.
  • Loose Coupling:
    • Achieved through dependency injection.
    • Changes in one layer have minimal impact on others.
  • Scalability:
    • Allows the application to scale efficiently in large production environments.
    • Easy to maintain, test, and extend.

Why to use MVC Architecture​

The Model-View-Controller (MVC) architecture is widely used in modern application development because it offers several key benefits:

1. Separation of Concerns​

  • Each layer is responsible for a specific task:
    • Model: Handles data and business logic.
    • View: Manages the presentation and user interface.
    • Controller: Connects the model and view; handles user input.
  • Separation ensures that different teams can work independently:
    • One team can work on the presentation layer (UI).
    • Another team can focus on business logic.
    • A third team can manage database access (persistence layer).

2. Reusability​

  • Business logic (Service layer) can be reused across multiple controllers or views.
  • For example:
    • An Employee controller and a Department controller can both use the same EmployeeService.
  • This is only possible because of the layered structure provided by MVC.

3. Improved Testability​

  • Since layers are loosely coupled, each can be tested independently:
    • You can perform unit testing on services, controllers, and repositories separately.
  • Helps in writing effective and maintainable test cases.

4. Scalability​

  • MVC makes it easier to scale both the codebase and the development process:
    • As the application grows, the modular structure allows new features to be added with minimal conflict.
    • Multiple teams can work on different layers simultaneously without interference.

5. Widespread Adoption​

  • MVC is not limited to Spring Boot or Java.

  • It is used in many frameworks and technologies like:

    • ASP.NET MVC
    • Django (Python)
    • Ruby on Rails
    • Angular, React (MV* pattern)

  • Its structure is universally applicable for building web, desktop, or mobile applications.

6. Maintainability​

  • Easier to locate and fix bugs, add new features, or refactor code.
  • The clear separation makes the application easier to understand and maintain over time.

Spring Boot Web Project Structure​

Complete Flow of a Web Application​

  • Client Interaction:

    • The client sends an HTTP request to the Controller.
    • Data is usually sent in JSON format.

  • Controller to Service:

    • The JSON data is converted into a DTO (Data Transfer Object).
    • The controller passes this DTO to the Service Layer.

  • Service to Repository:

    • The Service Layer interacts with the Repository Layer.
    • Data may be converted from DTO to Entity before performing CRUD operations.

  • Repository to Database:

    • The Repository Layer interacts with the actual database (e.g., SQL or NoSQL).
    • It performs required operations like save, fetch, update, or delete.

  • Returning Data to Client:

    • Data from the database is transformed back from Entity → DTO → JSON.
    • The response travels back: Repository → Service → Controller → Client.

Importance of DTO and Entity Separation​

  • DTO (Data Transfer Object):
    • Used for communication between Controller and Service.
    • Keeps the internal structure of entities hidden from the outside world.
    • Prevents exposing the database schema to the clients.
  • Entity:
    • Used for communication between Service and Repository.
    • Represents the actual database model (tables).
  • Why Hide Entities?
    • Prevents security risks like SQL Injection.
    • Reduces coupling between client and database structure.
    • Promotes better encapsulation and abstraction.

Standard Spring Boot Project Structure (Packages)​

com.yourcompany.projectname
│
├── controller       // For handling client requests (Presentation layer)
├── service          // For business logic
├── repository       // For database interactions (DAO layer)
├── dto              // For data transfer between controller and service
├── entity           // For database table mappings (JPA Entities)
├── config           // For configuration files (e.g., security, DB configs)
└── utils            // For utility/helper classes (e.g., date converter, unit conversion)

Alternative Project Structure (Used by Companies)​

In microservices or large-scale apps, companies may organize code feature-wise.

├── employee
│   ├── controller
│   ├── service
│   ├── repository
│   ├── dto
│   └── entity
├── department
│   ├── controller
│   ├── service
│   ├── repository
│   ├── dto
│   └── entity
