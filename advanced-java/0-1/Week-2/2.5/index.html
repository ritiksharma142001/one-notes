PUT, PATCH and DELETE Mappings

Additional Mappings and Code Optimization​

  • We will now:

Implement the remaining HTTP methods:​

  • @PutMapping
  • @PatchMapping
  • @DeleteMapping
  • Perform code optimizations to make the application more aligned with industry standards.

Key Concepts​

  • @PutMapping (Full Update)

    • Used when updating the entire resource.
    • Example: Changing all details of an employee with ID 1 from "Ram" to "Shyam" — the ID remains the same, but all other details are replaced.
    • Ideal when replacing the whole object on the backend.

  • @PatchMapping (Partial Update)

    • Used when updating specific fields of an existing resource.
    • Example: Only updating the employee's name or department, not the entire employee record.
    • Suitable for modifying only a subset of fields without affecting the rest.

Implementing PUT Mapping for Employee Update by ID​

  • We aim to return an EmployeeDto after updating an employee by their ID.
  • The function name updateEmployeeById is descriptive and follows good naming conventions.

Setting Up the PUT Mapping​

  • We use @PutMapping and append the employeeId to the path.
  • Example path: /api/employees/{employeeId}
  • The employeeId is extracted using @PathVariable of type Long.

Receiving Data for Update​

  • The updated employee data is passed in the request body using @RequestBody.
  • The data is received as an EmployeeDto object.

Calling the Service Method​

  • We call a service method: updateEmployeeById(employeeId, employeeDto).
  • If not already defined, we create this method in the EmployeeService class.

Handling the Update Logic in Service Layer​

  • Inside the service method:
    • Convert EmployeeDto to EmployeeEntity using ModelMapper.
    • Set the id of the entity manually in case it's not present in the DTO.
    • Save the entity using employeeRepository.save(employeeEntity).

Behavior Based on Employee Presence​

  • If the employee exists:
    • The record will be updated.
  • If the employee does not exist:
    • A new employee record will be created with the given ID and data.

Explanation with HashMap Analogy​

  • Similar to how a HashMap works:
    • If the key (employee ID) exists, the value is updated.
    • If the key doesn't exist, a new entry is created.

Returning the Response​

  • Convert the saved entity back to EmployeeDto using ModelMapper.
  • Return the DTO as the response to the client.

Delete Employee by ID – Explanation Summary​

Delete Mapping in REST API​

  • To delete an employee, a @DeleteMapping is used.
  • The employee ID is passed as a path variable (e.g., /employee/{id}).
  • No RequestBody is required since only the ID is sufficient.

Method Signature​

  • Initially, a void method is used for deletion.
  • Later, it is changed to return a boolean to indicate success or failure of the delete operation.

Deleting the Employee​

  • mployeeRepository.deleteById(id) is called to remove the employee from the database.
  • If the employee exists, it is deleted successfully.
  • If the employee does not exist, no action is taken and no error is shown.

Lack of Feedback​

  • Initially, no response is returned to indicate whether the employee was found or deleted.
  • This makes it unclear if the delete operation actually succeeded.

Improved Approach with Boolean Return​

  • The method return type is updated to boolean to provide better feedback.
  • employeeRepository.existsById(id) is used:
    • If the employee exists, it is deleted and true is returned.
    • If the employee does not exist, false is returned.

esting the API​

  • Example:
    • Employee with ID 54 exists → delete returns true.
    • Employee with ID 55 does not exist → delete returns false.

Conclusion​

  • The final version provides a better user experience by indicating the result of the delete action.
  • The API now supports full CRUD operations:
    • Get all employees
    • Get employee by ID
    • Create employee
    • Update employee
    • Delete employee by ID (with feedback)

PATCH Mapping in Spring Boot: Partial Update of Employee Entity​

Why PATCH Mapping Is Needed​

  • Partial updates (e.g., updating only name) using @PutMapping may result in other fields being overwritten with nul`l.
  • Using PUT with incomplete data causes the remaining fields (email, age, dateOfJoining, etc.) to be lost.
  • This is undesirable—we only want to update specific fields without affecting the rest.

Problem with PUT Mapping Example​

  • Suppose we update employee with `id=1`and send only:
{
  "name": "XYZ"
}
  • Result: Only name is updated, but email, age, and dateOfJoining become null.

Solution: Use @PatchMapping with Map Input​

  • Instead of sending the full entity, send a map containing only the fields to update
@PatchMapping("/employees/{id}")
public EmployeeDTO updatePartialEmployeeById(
    @PathVariable Long id,
    @RequestBody Map<String, Object> updates
)
  • Use a Map<String, Object>:
    • key = field name (e.g., "name")
    • value = new value for that field (e.g., "XYZ")

Handling Null Fields Safely​

  • This approach avoids passing fields with null values.
  • Only specified fields will be updated.

Service Method Outline​

  • Check if employee exists:
if (!employeeRepository.existsById(id)) {
    return null; // or throw custom NotFoundException
}
  • Refactor Repeated Code:
  • Create a reusable method:
public boolean existsByEmployeeId(Long id) {
    return employeeRepository.existsById(id);
}
  • Fetch the employee:
Employee employee = employeeRepository.findById(id).orElse(null);
  • Apply updates:
    • Iterate through the Map and update only those fields.
    • Use reflection or a manual if condition per field.

Reflection-Based Partial Update in Java (Spring Boot)​

Entity Presence Check​

  • Before performing updates, we ensure the employee already exists in the database.
  • Hence, there's no need to re-check for null here.

Error Handling (Optional)​

  • We could wrap the logic inside a try-catch block to gracefully handle exceptions.
  • However, error handling is not the focus at this stage.

Updating Fields Dynamically​

  • We want to update only the fields provided in the updates map.
  • Fields not included in the updates map should remain unchanged.

Use of Reflection​

  • Java Reflection allows dynamic access and modification of object fields at runtime.
  • It enables us to update fields without writing boilerplate if-else or switch logic.

Iterating Through Updates​

  • Loop through each entry in the updates map (Map<String, Object>).
  • key represents the field name, and value is the new value to be set.

Getting the Field Object​

  • Use ReflectionUtils.findField(Employee.class, key) to get the corresponding field.

Store the result in a Field object (e.g., fieldToUpdate).

Making Field Accessible​

  • Most fields in the entity are private.
  • Use fieldToUpdate.setAccessible(true) to allow modification.

Setting Field Value​

  • Use ReflectionUtils.setField(fieldToUpdate, employeeEntity, value) to set the new value.
  • This updates the in-memory entity object with the provided changes.

Saving the Updated Entity​

  • After applying all updates, call the save() method of the repository.
  • This persists the changes to the database.

Returning the Updated DTO​

  • Use ModelMapper to convert the updated entity back to a DTO.
  • Return this DTO as the response.

Why Reflection is Useful​

  • It reduces repetitive code when updating dynamic fields.
  • Allows updating objects using field names (as strings) at runtime.
  • No need to hard-code field access for each possible property.

Testing the PATCH Endpoint​

  • Created a PATCH endpoint that accepts partial updates.
  • Tested by updating only the name field of employee with id=2.
  • Result: Only name was updated; other fields remained unchanged.

package com.example.springBootWeb.springBootWeb.services;

import com.example.springBootWeb.springBootWeb.dto.EmployeeDTO;
import com.example.springBootWeb.springBootWeb.entities.EmployeeEntity;
import com.example.springBootWeb.springBootWeb.repositories.EmployeeRepository;
import org.apache.el.util.ReflectionUtil;
import org.modelmapper.ModelMapper;
import org.springframework.data.util.ReflectionUtils;
import org.springframework.stereotype.Service;

import java.lang.reflect.Field;
import java.util.List;
import java.util.Map;
import java.util.stream.Collectors;

@Service
public class EmployeeService {

    private final EmployeeRepository employeeRepository;

    private final ModelMapper modelMapper;

    public EmployeeService(EmployeeRepository employeeRepository, ModelMapper modelMapper) {
        this.employeeRepository = employeeRepository;
        this.modelMapper = modelMapper;
    }

    public EmployeeDTO getEmployeeById(Long id) {
        EmployeeEntity employeeEntity =  employeeRepository.findById(id).orElse(null);
        return modelMapper.map(employeeEntity, EmployeeDTO.class);
    }

    public List<EmployeeDTO> getAllEmployees() {
        List<EmployeeEntity> employeeEntities =  employeeRepository.findAll();
        return employeeEntities
                .stream()
                .map(employeeEntity -> modelMapper.map(employeeEntity, EmployeeDTO.class))
                .collect(Collectors.toList());
    }

    public EmployeeDTO createNewEmployee(EmployeeDTO inputEmployee) {
        EmployeeEntity toSaveEntity = modelMapper.map(inputEmployee, EmployeeEntity.class);
        EmployeeEntity savedEmployeeEntities = employeeRepository.save(toSaveEntity);
        return modelMapper.map(savedEmployeeEntities, EmployeeDTO.class);
    }

    public EmployeeDTO updateEmployeeById(Long employeeId, EmployeeDTO employeeDTO) {
        EmployeeEntity employeeEntity = modelMapper.map(employeeDTO, EmployeeEntity.class);
        employeeEntity.setId(employeeId);
        EmployeeEntity savedEmployeeEntity = employeeRepository.save(employeeEntity);
        return modelMapper.map(savedEmployeeEntity, EmployeeDTO.class);
    }

    public boolean isExistsEmployeeId(Long employeeId){
        return employeeRepository.existsById(employeeId);
    }
    public boolean deleteEmployeeById(Long employeeId) {
        boolean exists = isExistsEmployeeId(employeeId);
        if(!exists) return false;
        employeeRepository.deleteById(employeeId);
        return true;
    }

    public EmployeeDTO updatePartialEmployeeById(Long employeeId, Map<String, Object> updates) {
        boolean exists = isExistsEmployeeId(employeeId);
        if(!exists) return null;
        EmployeeEntity employeeEntity = employeeRepository.findById(employeeId).get();
        updates.forEach((field, value)->{
            Field fieldToBeUpdated = ReflectionUtils.getRequiredField(EmployeeEntity.class, field);
            fieldToBeUpdated.setAccessible(true);
            ReflectionUtils.setField(fieldToBeUpdated, employeeEntity, value);
        });
        return modelMapper.map(employeeRepository.save(employeeEntity), EmployeeDTO.class);
    }
}

Fixes for isActive Field in JSON Serialization with Jackson​

Observation of PATCH Behavior:​

  • When updating a single field (e.g., name), all other fields remain intact.
  • Demonstrated that updating fields like age via PATCH works as intended.

Issue with Boolean Field isActive:​

  • The field isActive was returning null even when passed correctly in the JSON.
  • This was due to Jackson's handling of boolean field naming conventions.

Root Cause of the Issue:​

  • Jackson tries to map Java boolean fields using getter methods.
  • For boolean isActive, Jackson looks for a method named getIsActive() but the auto-generated method is typically isActive(), causing confusion during deserialization.
  • This mismatch results in null values or additional unintended fields like active.

Fix with @JsonProperty:​

  • Use @JsonProperty("isActive") annotation on the isActive field to explicitly define the property name in JSON.
  • This helps Jackson map the JSON field isActive correctly to the Java property.
@JsonProperty("isActive")
private Boolean isActive;

Consistency Required Across DTOs and Entities:​

  • Ensure the same @JsonProperty("isActive") annotation is used in both DTO and entity classes wherever the isActive field is present.
  • This ensures Jackson consistently serializes and deserializes the field across all layers.

Handling Getters and Setters:​

  • The issue persisted because the default getter/setter names caused Jackson to create conflicting fields like active and isActive.
  • For example:
    • Getter: getActive() instead of getIsActive()
    • Setter: setActive() instead of setIsActive()

Solution with Lombok:​

  • Removed manually written getters and setters.
  • Used Lombok annotations to auto-generate them cleanly, avoiding naming issues:
@Getter
@Setter
@AllArgsConstructor
@NoArgsConstructor
public class EmployeeDTO {
    @JsonProperty("isActive")
    private Boolean isActive;
    // other fields
}

Final Result:​

  • After these changes, JSON now contains only one isActive field.
  • Values are correctly passed from Postman into Java objects and persisted as expected.
  • The GET and PATCH operations now work properly without duplicating or misnaming fields.

Using ResponseEntity to Return Meaningful HTTP Status Codes​

Purpose of ResponseEntity:​

  • In a REST controller, we use ResponseEntity to return appropriate HTTP status codes.
  • For example:
    • When a resource is created → return 201 Created.
    • When a resource is not found → return 404 Not Found.

How to Use ResponseEntity:​

  • It is part of the org.springframework.http package.
  • It can encapsulate any object, such as EmployeeDTO.

Returning ResponseEntity for Get by ID:​

  • Instead of returning the EmployeeDTO directly, wrap it in a ResponseEntity.
if (employeeDto == null) {
    return ResponseEntity.notFound().build(); // Returns 404
} else {
    return ResponseEntity.ok(employeeDto);     // Returns 200
}

Handling Null Issues with ModelMapper:​

  • If the entity is null, ModelMapper throws a "Source cannot be null" error.
  • To handle this safely, use Optional<Employee>:
Optional<Employee> employee = employeeRepository.findById(id);

Mapping Optional Entity to DTO:​

  • Convert the optional entity to DTO using .map():
return employeeRepository.findById(id)
    .map(emp -> modelMapper.map(emp, EmployeeDTO.class))
    .map(ResponseEntity::ok)
    .orElse(ResponseEntity.notFound().build());
  • This returns:
    • 200 OK with data if present.
    • 404 Not Found if the employee is not found.

    }

    @PostMapping
    public ResponseEntity<EmployeeDTO> createNewEmployee(@RequestBody EmployeeDTO inputEmployee) {
        EmployeeDTO savedEmployee = employeeService.createNewEmployee(inputEmployee);
        return new ResponseEntity<>(savedEmployee, HttpStatus.CREATED);
    }


    @PutMapping(path = "/{employeeId}")
    public ResponseEntity<EmployeeDTO> updateEmployeeById(@RequestBody EmployeeDTO employeeDTO, @PathVariable Long employeeId) {
        return ResponseEntity.ok(employeeService.updateEmployeeById(employeeId, employeeDTO));
    }

    @DeleteMapping(path = "/{employeeId}")
    public ResponseEntity<Boolean> deleteEmployeeById(@PathVariable Long employeeId) {
        boolean gotDeleted = employeeService.deleteEmployeeById(employeeId);
        if (gotDeleted) return ResponseEntity.ok(true);
        return ResponseEntity.notFound().build();
    }

    @PatchMapping(path = "/{employeeId}")
    public ResponseEntity<EmployeeDTO> updatePartialEmployeeById(@RequestBody Map<String, Object> updates,
                                                 @PathVariable Long employeeId) {
        EmployeeDTO employeeDTO = employeeService.updatePartialEmployeeById(employeeId, updates);
        if (employeeDTO == null) return ResponseEntity.notFound().build();
        return ResponseEntity.ok(employeeDTO);
    }

}
