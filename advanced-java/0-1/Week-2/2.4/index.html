Service Layer

Connecting the Persistence Layer and Presentation Layer Using the Service Layer​

Introduction​

  • In today's class, we will learn how to connect the Persistence Layer and the Presentation Layer with the help of the Service Layer.
  • The Service Layer acts as a bridge between the Persistence Layer and the Presentation Layer.
  • It helps improve the modularity, scalability, and testability of the application.

Current Project Structure​

  • So far, we have seen how Controllers, Repositories, DTOs, and Entities work.
  • Currently, we are directly passing the instance of the repository into the controller.
  • This practice is not recommended, as it tightly couples the controller with the repository.

Why We Need the Service Layer​

  • There should be a middle layer between the controller and the repository – this is the Service Layer.
  • It ensures that your code remains modular and separated by concerns.

Key Responsibilities of the Service Layer​

  • Acts as a Bridge
    • Connects the Persistence Layer (Repository) with the Presentation Layer (Controller).
  • Encapsulates Business Logic
    • Contains all business rules and operations, such as conditional logic, data manipulation, and validation.
  • Coordinates Component Interaction
    • Manages and orchestrates the interactions between different components in a clean and maintainable way.
  • Provides a Unified Interface
    • Offers a centralized and consistent way for external clients (like controllers) to interact with business functionality.

Separation of Concerns​

  • The Controller Layer handles client interactions (e.g., HTTP requests).
  • The Persistence Layer handles interactions with the database.
  • The Service Layer handles all the business logic, including:
    • If-else conditions
    • Custom rules provided by product managers
    • Logging
    • Security checks
    • Authentication and authorization

Real-World Scenario​

  • Suppose we have two controllers:
    • EmployeeController
    • DepartmentController
  • The DepartmentController needs access to data from the EmployeeRepository.
  • Instead of injecting the EmployeeRepository directly into both controllers:
    • We create an EmployeeService class.
    • This service:
      • Calls the EmployeeRepository
      • Performs logging
      • Handles authentication
      • Implements additional business logic
  • Now, instead of repeating the same logic (like logging or authentication) in every controller, we centralize it in the service.

Advantages of Using the Service Layer​

  • Avoids Code Duplication
    • Logic like logging or security checks is written once and reused across multiple controllers.
  • Increases Modularity
    • Services are modular and reusable in different parts of the application.
  • Follows DRY Principle
    • "Don't Repeat Yourself" – encourages clean and maintainable code.
  • Improves Scalability
    • As the application grows, services can easily be extended without affecting controllers or repositories.

Steps to Implement the Service Layer​

Navigate to the Service Package​

  • Go to the previously created service package in your project structure.

Create an Employee Service Class​

  • Create a new class named EmployeeService.
  • This class will contain all the business logic related to employee operations.

nnotate the Service Class​

  • Annotate the class with @Service to mark it as a Spring-managed component (bean).
  • This enables Spring to auto-detect and inject this service into other components like controllers.

Move Business Logic from Controller to Service​

  • Previously, methods such as findEmployeeById were present in the controller using the EmployeeRepository directly.
  • This is not a recommended practice as it tightly couples the controller to the persistence layer.

Refactor Controller to Use Service​

  • Remove EmployeeRepository from the controller.
  • Inject EmployeeService into the controller using constructor injection.
private final EmployeeService employeeService;

public EmployeeController(EmployeeService employeeService) {
    this.employeeService = employeeService;
}

Create Method in Service Layer​

  • Move the repository call employeeRepository.findById(id) to the service layer.
  • Create a method like getEmployeeById(Long id) inside EmployeeService.
public Employee getEmployeeById(Long id) {
    return employeeRepository.findById(id).orElse(null);
}

Call Service Method from Controller​

  • In the controller, call employeeService.getEmployeeById(id) instead of directly accessing the repository.

Inject Repository in Service Layer​

  • Use constructor injection to inject EmployeeRepository in the EmployeeService.
private final EmployeeRepository employeeRepository;

public EmployeeService(EmployeeRepository employeeRepository) {
    this.employeeRepository = employeeRepository;
}

Handle Optional Return Type​

  • Handle the case where the employee may not be found:
    • Either return null (temporarily).
    • Or use orElseThrow() to throw a custom exception (exception handling will be added later).

Avoid Returning Entity from Controller​

  • Best practice: do not return entities from controller directly.
  • Instead, convert Entity to DTO inside the service layer.
  • DTO conversion will be handled in a later section.

package com.example.springBootWeb.springBootWeb.services;

import com.example.springBootWeb.springBootWeb.entities.EmployeeEntity;
import com.example.springBootWeb.springBootWeb.repositories.EmployeeRepository;

import java.util.List;

public class EmployeeService {

    private final EmployeeRepository employeeRepository;

    public EmployeeService(EmployeeRepository employeeRepository) {
        this.employeeRepository = employeeRepository;
    }

    public EmployeeEntity getEmployeeById(Long id) {
        return employeeRepository.findById(id).orElse(null);
    }

    public List<EmployeeEntity> getAllEmployees() {
        return employeeRepository.findAll();
    }

    public EmployeeEntity createNewEmployee(EmployeeEntity inputEmployee) {
        return employeeRepository.save(inputEmployee);
    }
}

DTO Mapping and ModelMapper in Service Layer​

Identifying the Problem​

Repository Removed, But Entities Still in Use​

  • Although we have successfully removed the use of EmployeeRepository from the controller and introduced EmployeeService, there is still an issue.
  • The controller is still using the Employee entity directly.

Best Practice: No Entity Exposure in Controller​

  • According to clean architecture principles:
    • The controller (presentation layer) must not deal with Entity classes.
    • It should interact only with DTOs (Data Transfer Objects).

Why This Is a Problem​

  • Entities Still Returned by Service
    • Service methods such as getEmployeeById() return an Employee entity.
    • The controller receives the entity and returns it in the API response — this violates encapsulation.
  • Need for DTO Conversion
    • We should convert the Employee entity into an EmployeeDTO within the service layer, and then return the DTO to the controller.

Manual Conversion Approach​

  • Example: Manual Mapping from Entity to DTO
public EmployeeDTO getEmployeeById(Long id) {
    Employee employee = employeeRepository.findById(id).orElse(null);
    return new EmployeeDTO(employee.getId(), employee.getName(), employee.getEmail());
}

Drawbacks of Manual Mapping​

  • Manual mapping works, but it becomes repetitive and error-prone:
    • You must manually map each field from entity to DTO and vice versa.
    • With multiple entities and DTOs, this logic becomes redundant and scattered.
  • For example, similar code will be required:
    • In getAllEmployees()
    • In createEmployee()
    • And for every future entity you add

Solution: Use a Mapping Library​

  • Use of ModelMapper
    • To solve the problem of repeated conversion logic, we can use a popular mapping library called ModelMapper.
    • ModelMapper automatically maps fields between Entity and DTO classes based on matching field names.

Step 1: Adding the ModelMapper Dependency​

  • Locate the Dependency

    • Search for ModelMapper Maven in your browser.
    • You can find the official website or the Maven repository listing.

  • Add to pom.xml

    • Copy the ModelMapper dependency (e.g., version 3.0.0).
    • Paste it inside your pom.xml file.

  • Refresh Maven

    • After adding the dependency, refresh your pom.xml in the IDE.
    • This downloads the required ModelMapper library.

Step 2: Understanding How ModelMapper Works​

  • Purpose of ModelMapper
    • It helps convert one type of object to another, e.g., Entity ↔ DTO.
  • Matching Fields Automatically
    • ModelMapper scans the fields in both classes.
    • If fields with the same name and compatible types exist, it automatically maps them.
    • Example:
      • Employee entity has: id, name, email, age.
      • EmployeeDTO has matching fields.
      • ModelMapper will map values from entity to DTO automatically.

Step 3: Basic Usage of ModelMapper​

  • Initialize the Mapper
ModelMapper mapper = new ModelMapper();
  • Perform Mapping
EmployeeDTO dto = mapper.map(employeeEntity, EmployeeDTO.class);
return dto;
  • Outcome
    • This will return a DTO object instead of an entity.
    • The controller now receives a DTO, not an entity, which aligns with best practices.

Reusing ModelMapper in Multiple Places​

  • Why Reuse Is Needed
    • You may need to convert:
      • A single entity → DTO
      • A list of entities → list of DTOs
    • Example:

In getAllEmployees(), convert List<Employee> to List<EmployeeDTO>.

  • Avoid Repeating new ModelMapper()
    • Reinitializing the object every time is not recommended in Spring.

Step 4: Spring-Style ModelMapper Configuration​

  • Spring Best Practice
    • Instead of manually creating objects, define a ModelMapper bean.
    • This allows Spring to manage its lifecycle and inject it wherever needed.

Next Steps​

  • Go to the configuration class (usually named AppConfig, MapperConfig, etc.).
  • Define a bean for ModelMapper.
@Configuration
public class AppConfig {

    @Bean
    public ModelMapper modelMapper() {
        return new ModelMapper();
    }
}

Now You Can Autowire It​

@Autowired
private ModelMapper modelMapper;
  • Use it wherever needed without creating a new instance each time.

Inject the Bean into the Service Class​

  • Use constructor injection for better immutability and testability.
private final ModelMapper modelMapper;

@Autowired
public EmployeeService(ModelMapper modelMapper) {
    this.modelMapper = modelMapper;
}

Why Constructor Injection?​

  • Ensures ModelMapper is available at runtime.
  • Avoids repeatedly creating new objects.
  • Lets Spring handle the lifecycle of the dependency.

Converting Entity List to DTO List Using Stream API​

  • Fetch Entity List
List<Employee> employeeEntities = employeeRepository.findAll();
List<EmployeeDTO> employeeDTOs = employeeEntities.stream()
    .map(employee -> modelMapper.map(employee, EmployeeDTO.class))
    .collect(Collectors.toList());

Explanation​

  • stream() converts the list to a stream.
  • map() applies the mapping function to each entity.
  • collect(Collectors.toList()) gathers the mapped DTOs into a list.

package com.example.springBootWeb.springBootWeb.services;

import com.example.springBootWeb.springBootWeb.dto.EmployeeDTO;
import com.example.springBootWeb.springBootWeb.entities.EmployeeEntity;
import com.example.springBootWeb.springBootWeb.repositories.EmployeeRepository;
import org.modelmapper.ModelMapper;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.stream.Collectors;

@Service
public class EmployeeService {

    private final EmployeeRepository employeeRepository;

    private final ModelMapper modelMapper;

    public EmployeeService(EmployeeRepository employeeRepository, ModelMapper modelMapper) {
        this.employeeRepository = employeeRepository;
        this.modelMapper = modelMapper;
    }

    public EmployeeDTO getEmployeeById(Long id) {
        EmployeeEntity employeeEntity =  employeeRepository.findById(id).orElse(null);
        return modelMapper.map(employeeEntity, EmployeeDTO.class);
    }

    public List<EmployeeDTO> getAllEmployees() {
        List<EmployeeEntity> employeeEntities =  employeeRepository.findAll();
        return employeeEntities
                .stream()
                .map(employeeEntity -> modelMapper.map(employeeEntity, EmployeeDTO.class))
                .collect(Collectors.toList());
    }

    public EmployeeDTO createNewEmployee(EmployeeDTO inputEmployee) {
        EmployeeEntity toSaveEntity = modelMapper.map(inputEmployee, EmployeeEntity.class);
        EmployeeEntity savedEmployeeEntities = employeeRepository.save(toSaveEntity);
        return modelMapper.map(savedEmployeeEntities, EmployeeDTO.class);
    }
}

package com.example.springBootWeb.springBootWeb.entities;

import jakarta.persistence.*;
import lombok.*;

import java.time.LocalDate;

@Entity
@Getter
@Setter
@AllArgsConstructor
@NoArgsConstructor
@Table(name = "employees")
public class EmployeeEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Long id;
    private String name;
    private String email;
    private Integer age;
    private LocalDate dateOfJoining;
    private Boolean isActive;
}


package com.example.springBootWeb.springBootWeb.dto;

import java.time.LocalDate;

public class EmployeeDTO {

    private Long id;
    private String name;
    private String email;
    private Integer age;
    private LocalDate dateOfJoining;
    private Boolean isActive;

    public EmployeeDTO(){

    }
    public EmployeeDTO(Long id, String name, String email, Integer age, LocalDate dateOfJoining, Boolean isActive) {
        this.id = id;
        this.name = name;
        this.email = email;
        this.age = age;
        this.dateOfJoining = dateOfJoining;
        this.isActive = isActive;
    }

    public Long getId() {
        return id;
    }

    public void setId(Long id) {
        this.id = id;
    }

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    public String getEmail() {
        return email;
    }

    public void setEmail(String email) {
        this.email = email;
    }

    public Integer getAge() {
        return age;
    }

    public void setAge(Integer age) {
        this.age = age;
    }

    public LocalDate getDateOfJoining() {
        return dateOfJoining;
    }

    public void setDateOfJoining(LocalDate dateOfJoining) {
        this.dateOfJoining = dateOfJoining;
    }

    public Boolean getIsActive() {
        return isActive;
    }

    public void setIsActive(Boolean active) {
        isActive = active;
    }
}
