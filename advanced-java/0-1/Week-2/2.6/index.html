Input Validation

Introduction to Input Validation in Spring Boot​

Overview of CRUD Operations​

  • So far, we have covered how CRUD (Create, Read, Update, Delete) operations work in REST APIs.
  • We learned how to create, read, update, and delete data in a database through REST endpoints.

Need for Input Validation​

  • Next, we focus on validating input data.
  • Input validation ensures that users send correct and meaningful data to the server.
  • For example:
    • The email field must contain a valid email address.
    • The date of birth must be a past date.
    • Other custom validations may be required depending on business logic.

Why Validation is Important​

  • Without validation, users can send any data.
  • For instance, a random string in the email field would still get accepted and stored in the database.
  • We want to prevent such invalid inputs from reaching the database.

Interception at Controller Level​

  • Ideally, invalid data should be caught before reaching the service layer.
  • The controller should intercept and validate incoming requests.

Validation Strategy​

  • The controller receives the request body as a DTO (Data Transfer Object).
  • We want to validate this DTO before the controller passes it to the service layer.

Spring Boot Validation Support​

-Spring Boot provides an in-built library for input validation: spring-boot-starter-validation.

Steps to Enable Validation​

  • Go to your pom.xml file.
  • Add the dependency:
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
  • Refresh your Maven project to download the necessary jars.

Validating the DTO​

  • The controller accepts input as an EmployeeDTO.
  • We need to add validation annotations (like @Email, @NotNull, @Past, etc.) on DTO fields.
  • This ensures that invalid DTOs are rejected automatically, even before they reach the service layer.

Input Validation in Spring Boot​

Common Validation Annotations​

  • Spring Boot (via Jakarta Validation) provides multiple annotations to validate input fields:
    • @NotNull
    • @NotEmpty
    • @NotBlank
    • @Size
    • @Max, @Min
    • @Email
    • @Pattern
  • These annotations help enforce constraints on incoming request data.

Applying Validation on DTO Fields​

  • For example, if you want to ensure that the name field is mandatory:
@NotNull(message = "Required field in employee: name")
private String name;
  • This annotation ensures the field is not null, and returns a custom error message if validation fails.

Dependency for Validation​

  • Make sure the following dependency is added in pom.xml:
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-validation</artifactId>
</dependency>

Enabling Validation in Controller​

  • Merely annotating the DTO fields isn’t enough.
  • In the controller, annotate the DTO parameter with @Valid to trigger validation:
@PostMapping("/employees")
public ResponseEntity<EmployeeDTO> createEmployee(@Valid @RequestBody EmployeeDTO employeeDTO) {
    ...
}
  • @Valid ensures Spring Boot validates the incoming request before executing the method logic.

Behavior Before and After Validation​

  • Before adding @Valid:
    • Requests with invalid or missing fields (like name = null) are accepted and persisted.
  • After adding @Valid:
    • Such requests are rejected with HTTP 400 (Bad Request) and appropriate validation messages.

Validation Output​

  • When validation fails:
    • The controller method is not executed.
    • A validation error response is returned, typically with HTTP status 400.
    • The response includes the message defined in the annotation (e.g., "Required field in employee: name").

Handling Error Messages​

  • As of now, the error message is visible in the response body but not structured clearly.
  • Proper formatting and customization of validation error responses (e.g., showing field names and messages) will be addressed in the next session on exception handling.

Validation Success Example​

  • When a valid name is provided, the request proceeds successfully, and the employee is created.
  • Example:
{
  "name": "John Doe",
  "email": "john@example.com"
}
  • This would pass validation and proceed to the service layer.

Java Bean Validation Annotations​

@NotNull​

  • Ensures the annotated field is not null.
  • Commonly used when a field must not be missing in the request.
  • Example:
@NotNull(message = "Name is required")
private String name;
  • Without adding @Valid in the controller, the validation won't work.

Enable Validation in Controller​

  • Use @Valid on the request body to activate validation:
@PostMapping("/employee")
public ResponseEntity<EmployeeDto> createEmployee(@Valid @RequestBody EmployeeDto employeeDto) {
    ...
}

Validation Error Response​

  • If a field is missing or null (with @NotNull), the API returns HTTP 400 - Bad Request.
  • Example message: Required field in Employee: name.

@NotEmpty​

  • Ensures the field is not null and not empty (size > 0).
  • Useful for Collections, Strings, Arrays.
  • Example:
@NotEmpty(message = "Employee name cannot be empty")
private String name;

@NotBlank​

  • Ensures the string is not null and not blank after trimming.
  • Best suited for String fields.
  • Example:
@NotBlank(message = "Employee name cannot be blank")
private String name;
  • Prevents cases where user submits " " as input.

@Size​

  • Used to validate the length of strings, arrays, collections, etc.
  • You can define min and max length constraints.
  • Example:
@Size(min = 3, max = 10, message = "Number of characters in name should be in the range 3 to 10")
private String name;
  • If a name like "An" is passed, it will fail this validation.

@Email​

  • Ensures the field is a valid email format.
  • Example:
@Email(message = "Email should be a valid email")
private String email;
  • Valid example: user@example.com.
  • Invalid: anu@ or just anu.

@Max​

  • Ensures the annotated element does not exceed a maximum value.
  • Example:
@Max(value = 80, message = "Age cannot be greater than 80")
private int age;
  • If age exceeds 80, validation fails with the custom message.

@Min​

  • Ensures the annotated element is not less than a specified value.
  • Example:
@Min(value = 18, message = "Age of employee cannot be less than 18")
private int age;
  • Prevents input like age = 2 (e.g., a toddler) from being accepted.

@Size​

  • Used with String, arrays, or collections to specify minimum and/or maximum length/size.
  • Example:
@Size(min = 3, max = 20, message = "Name must be between 3 and 20 characters")
private String name;

@Pattern​

  • Validates a String field against a regular expression.
  • Example (allow only "admin" or "user"):
@Pattern(regexp = "^(admin|user)$", message = "Role of employee can either be 'admin' or 'user'")
private String role;
  • Starts with ^ and ends with $ to ensure exact match.
  • Can test and verify regex in IntelliJ using the Check Regular Expression utility.
  • If input is "users", "Admin1", etc., validation will fail.

@NotNull / @NotBlank​

  • Ensures the field is not null or not empty (not just whitespace).
  • Examples:
@NotBlank(message = "Role of employee cannot be blank")
private String role;

@NotBlank(message = "Email cannot be blank")
private String email;

@NotNull(message = "Age must be provided")
private Integer age;

@Email​

  • Validates that a given String is a valid email address.
  • Can be combined with @NotBlank to ensure it’s provided.
@Email(message = "Invalid email format")
@NotBlank(message = "Email cannot be blank")
private String email;

@Positive​

  • Ensures the field holds a positive number (greater than 0).
  • Suitable for fields like salary, quantity, etc.
@Positive(message = "Salary must be a positive number")
private Double salary;

@NotNull​

  • Used on: Fields like Integer salary
  • Purpose: Ensures the field is not null.
  • Custom Message: "Salary of employee should not be null"
@NotNull(message = "Salary of employee should not be null")
private Integer salary;

@Positive / @PositiveOrZero​

  • Used on: Numeric fields
  • Purpose:
    • @Positive: Must be > 0
    • @PositiveOrZero: Must be ≥ 0
  • Custom Message: "Salary of employee should be positive"
@Positive(message = "Salary of employee should be positive")
private Integer salary;

@Past, @PastOrPresent, @Future, @FutureOrPresent​

  • Used on: LocalDate, Date, or Calendar fields like dateOfJoining
  • Purpose:
    • @Past: Date must be in the past
    • @PastOrPresent: Date must be in the past or today
    • @Future: Date must be in the future
    • @FutureOrPresent: Date must be today or in the future
  • Custom Message: "Date of joining field in employee cannot be in the future"
@PastOrPresent(message = "Date of joining field in employee cannot be in the future")
private LocalDate dateOfJoining;

@Digits(integer = X, fraction = Y)​

  • Used on: Double, BigDecimal, or numeric types
  • Purpose: Restricts number of digits before and after the decimal
    • Example: @Digits(integer = 6, fraction = 2) → allows up to 999999.99
  • Custom Message: "Salary can be in the form XXXXXX.XX only"
@Digits(integer = 6, fraction = 2, message = "Salary must be in the format XXXXXX.XX")
private Double salary;
public class EmployeeDTO {

    @NotNull(message = "Salary of employee should not be null")
    @Positive(message = "Salary of employee should be positive")
    @Digits(integer = 6, fraction = 2, message = "Salary must be in the format XXXXXX.XX")
    private Double salary;

    @PastOrPresent(message = "Date of joining cannot be in the future")
    private LocalDate dateOfJoining;

    // Other fields, getters, setters
}

@DecimalMin and @DecimalMax – For Decimal Values​

  • Use these when validating floating-point numbers, like double, BigDecimal, etc.
  • You must pass values as Strings.
  • Example:
@DecimalMin(value = "100.50", message = "Salary must be at least ₹100.50")
@DecimalMax(value = "99999.99", message = "Salary must be less than ₹99999.99")
private double salary;

This ensures that:

  • Minimum salary allowed: 100.50
  • Maximum salary allowed: 99999.99
  • Common mistake:
    • You shouldn't use @Min/@Max with double values because they only accept long values.

@AssertTrue and @AssertFalse – For Booleans​

  • Used to enforce conditions on boolean fields.
  • Example:
@AssertTrue(message = "Employee should be active")
private boolean isActive;
  • This means isActive must be true, else validation will fail.

@Valid – For Nested Objects​

  • If you have objects inside objects (like DTOs containing other DTOs), and you want to validate their fields too:
  • Example:
public class EmployeeDTO {
    @Valid
    private DepartmentDTO department;
}
  • This triggers validation on DepartmentDTO fields recursively.

Why and How to Create Custom Validation Annotations in Java (Spring Boot)​

Why Custom Annotations?​

  • Custom validation annotations are useful when:
    • Built-in annotations (@Min, @Max, @Pattern, etc.) are not sufficient.
    • You want to apply complex or domain-specific validation logic (e.g., checking if a number is prime).
    • You want to reuse validation logic across multiple entities without duplicating code

How Built-in Annotations Are Defined​

  • All standard validation annotations are defined as:
@interface AnnotationName { ... }

They include:​

  • @Constraint: Indicates this is a validation annotation.
  • @Target: Specifies where the annotation can be applied (e.g., FIELD, METHOD, PARAMETER).
  • @Retention: Specifies how long the annotation is retained (e.g., SOURCE, CLASS, RUNTIME).

Retention Policies:​

  • SOURCE: Discarded by the compiler; used only in source code (like comments).
  • CLASS: Recorded in .class files by the compiler but ignored at runtime by the JVM.
  • RUNTIME: Retained at runtime and can be accessed via reflection (used for validations).
  • For validations, use @Retention(RetentionPolicy.RUNTIME) so that annotations can be processed during runtime.

Anatomy of a Validation Annotation:​

  • A custom annotation includes:
String message() default "default error message";
Class<?>[] groups() default {};
Class<? extends Payload>[] payload() default {};
  • message: Default message if validation fails.
  • groups: Used for grouping constraints.
  • payload: Carries metadata information (rarely used).

Example Use Case for Custom Annotation:​

  • Suppose you want to validate a field like role where only "ADMIN" or "USER" are valid values.
  • You could use:
@Pattern(regexp = "ADMIN|USER")
  • This works for simple cases.
  • But for more complex validations (e.g., external DB lookup, business logic), define a custom annotation and validator.

Creating a Custom Annotation to Validate Employee Role​

Commenting Existing Annotation​

  • Comment out the default @Pattern annotation used for validating the role field.

Create Custom Annotation Package​

  • Create a new package named annotation to organize all custom annotations.

Define the Custom Annotation Interface​

  • Create an interface named EmployeeRoleValidation.
  • Use @Retention(RetentionPolicy.RUNTIME) to retain the annotation at runtime.
  • Use @Target({ElementType.FIELD, ElementType.PARAMETER}) to allow the annotation to be used on fields and parameters.

Add Annotation Metadata (Message, Groups, Payload)​

  • Add the following attributes:
String message() default "Role must be either USER or ADMIN";
Class<?>[] groups() default {};
Class<? extends Payload>[] payload() default {};
  • These are standard components required for validation annotations. You can explore their purpose in detail, but for now, they can be reused as-is.

Create the Validator Class​

  • Create a class named EmployeeRoleValidator.
  • Implement the ConstraintValidator<EmployeeRoleValidation, String> interface.
  • Implement the isValid() method which checks whether the input role is valid.

Implement the Validation Logic​

public boolean isValid(String inputRole, ConstraintValidatorContext context) {
    List<String> roles = List.of("USER", "ADMIN");
    return roles.contains(inputRole);
}
  • Extendable: You can add more roles like MANAGER, HR, etc., to make it scalable without modifying regex patterns.

Link Annotation with Validator​

  • Use the @Constraint(validatedBy = EmployeeRoleValidator.class) annotation on your custom annotation interface to connect it with the validator logic.

Usage​

  • Now, you can use your custom @EmployeeRoleValidation annotation directly on any String field, like:
@EmployeeRoleValidation
private String role;

Custom Validation in Spring Boot​

Homework Task Given:​

  • Implement a custom annotation that checks whether a given integer is a prime number.

Custom Annotation Usage:​

  • A custom annotation @EmployeeRoleValidation is applied on the role field instead of the standard @Pattern.
  • The message inside the annotation defines: "Role of employee can either be USER or ADMIN."

Validation Behavior:​

  • If invalid data is passed (e.g., role = "users"), a 400 Bad Request is returned.
  • All validation errors are shown together. For example:
    • Invalid role
    • isActive = false triggers another error
  • Fixing one invalid field (e.g., setting isActive = true) narrows down the errors to just the remaining invalid fields.

Successful Validation:​

  • If all values are valid (e.g., role = admin and isActive = true), the request is successfully processed.

How Custom Validation Works:​

  • Create a custom annotation with @Constraint and @Target, @Retention.
  • Implement the ConstraintValidator interface.
  • Override the isValid() method to implement custom logic.
  • If isValid() returns true, input is accepted; otherwise, it's rejected.

Using @Valid:​

  • Add @Valid on fields or method parameters to trigger validation.
  • Example: Apply @Valid on the Employee DTO in POST and PUT endpoints.
  • For PATCH (partial updates), validation is handled manually since the input is a map of fields.

Limitation Noted:​

When multiple fields are invalid, all errors are returned together. The error messages can be overwhelming or unclear to the end user.

package com.example.springBootWeb.springBootWeb.annotations;

import jakarta.validation.Constraint;
import jakarta.validation.Payload;

import java.lang.annotation.ElementType;
import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;
import java.lang.annotation.Target;

@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.FIELD, ElementType.PARAMETER})
@Constraint(validatedBy = {EmployeeRoleValidator.class})
public @interface EmployeeRoleValidation {
    String message() default "Role of Employee can either be USER or ADMIN";

    Class<?>[] groups() default {};

    Class<? extends Payload>[] payload() default {};
}


package com.example.springBootWeb.springBootWeb.annotations;

import jakarta.validation.ConstraintValidator;
import jakarta.validation.ConstraintValidatorContext;

import java.util.List;

public class EmployeeRoleValidator implements ConstraintValidator<EmployeeRoleValidation, String> {

    @Override
    public boolean isValid(String inputRole, ConstraintValidatorContext constraintValidatorContext) {
        List<String> roles = List.of("USER", "ADMIN");
        return roles.contains(inputRole);
    }
}

package com.example.springBootWeb.springBootWeb.controllers;

import com.example.springBootWeb.springBootWeb.dto.EmployeeDTO;
import com.example.springBootWeb.springBootWeb.entities.EmployeeEntity;
import com.example.springBootWeb.springBootWeb.repositories.EmployeeRepository;
import com.example.springBootWeb.springBootWeb.services.EmployeeService;
import jakarta.validation.Valid;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PathVariable;
import org.springframework.web.bind.annotation.RestController;

import org.springframework.web.bind.annotation.*;

import java.time.LocalDate;
import java.util.List;
import java.util.Map;
import java.util.Optional;

@RestController
@RequestMapping(path = "/employees")
public class EmployeeController {

//    @GetMapping(path = "/getSecretMessage")
//    public String getMySuperSecretMessage() {
//        return "Secret message: asdfal@#$DASD";
//    }


    private final EmployeeService employeeService;

    public EmployeeController(EmployeeService employeeService) {
        this.employeeService = employeeService;
    }

    @GetMapping(path = "/{employeeId}")
    public ResponseEntity<EmployeeDTO> getEmployeeById(@PathVariable(name = "employeeId") Long id) {
        Optional<EmployeeDTO> employeeDTO = employeeService.getEmployeeById(id);
        return employeeDTO
                .map(employeeDTO1 -> ResponseEntity.ok(employeeDTO1))
                .orElse(ResponseEntity.notFound().build());
    }

    @GetMapping
    public ResponseEntity<List<EmployeeDTO>> getAllEmployees(@RequestParam(required = false, name = "inputAge") Integer age,
                                                             @RequestParam(required = false) String sortBy) {
        return ResponseEntity.ok(employeeService.getAllEmployees());
    }

    @PostMapping
    public ResponseEntity<EmployeeDTO> createNewEmployee(@RequestBody @Valid EmployeeDTO inputEmployee) {
        EmployeeDTO savedEmployee = employeeService.createNewEmployee(inputEmployee);
        return new ResponseEntity<>(savedEmployee, HttpStatus.CREATED);
    }


    @PutMapping(path = "/{employeeId}")
    public ResponseEntity<EmployeeDTO> updateEmployeeById(@RequestBody @Valid EmployeeDTO employeeDTO, @PathVariable Long employeeId) {
        return ResponseEntity.ok(employeeService.updateEmployeeById(employeeId, employeeDTO));
    }

    @DeleteMapping(path = "/{employeeId}")
    public ResponseEntity<Boolean> deleteEmployeeById(@PathVariable Long employeeId) {
        boolean gotDeleted = employeeService.deleteEmployeeById(employeeId);
        if (gotDeleted) return ResponseEntity.ok(true);
        return ResponseEntity.notFound().build();
    }

    @PatchMapping(path = "/{employeeId}")
    public ResponseEntity<EmployeeDTO> updatePartialEmployeeById(@RequestBody Map<String, Object> updates,
                                                                 @PathVariable Long employeeId) {
        EmployeeDTO employeeDTO = employeeService.updatePartialEmployeeById(employeeId, updates);
        if (employeeDTO == null) return ResponseEntity.notFound().build();
        return ResponseEntity.ok(employeeDTO);
    }

}

package com.example.springBootWeb.springBootWeb.dto;

import com.example.springBootWeb.springBootWeb.annotations.EmployeeRoleValidation;
import com.fasterxml.jackson.annotation.JsonProperty;
import jakarta.validation.constraints.NotNull;


import java.time.LocalDate;

import jakarta.validation.constraints.*;
import lombok.*;

@Getter
@Setter
@AllArgsConstructor
@NoArgsConstructor
public class EmployeeDTO {

    private Long id;

    @NotBlank(message = "Name of the employee cannot be blank")
    @Size(min = 3, max = 10, message = "Number of characters in name should be in the range: [3, 10]")
    private String name;

    @NotBlank(message = "Email of the employee cannot be blank")
    @Email(message = "Email should be a valid email")
    private String email;

    @NotNull(message = "Age of the employee cannot be blank")
    @Max(value = 80, message = "Age of Employee cannot be greater than 80")
    @Min(value = 18, message = "Age of Employee cannot be less than 18")
    private Integer age;

    @NotBlank(message = "Role of the employee cannot be blank")
//    @Pattern(regexp = "^(ADMIN|USER)$", message = "Role of Employee can either be USER or ADMIN")
    @EmployeeRoleValidation
    private String role; //ADMIN, USER

    @NotNull(message = "Salary of Employee should be not null")
    @Positive(message = "Salary of Employee should be positive")
    @Digits(integer = 6, fraction = 2, message = "The salary can be in the form XXXXX.YY")
    @DecimalMax(value = "100000.99")
    @DecimalMin(value = "100.50")
    private Double salary;

    @PastOrPresent(message = "DateOfJoining field in Employee cannot be in the future")
    private LocalDate dateOfJoining;

    @AssertTrue(message = "Employee should be active")
    @JsonProperty("isActive")
    private Boolean isActive;
}
