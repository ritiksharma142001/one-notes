Transforming API Response

Current Issue:​

  • Currently, we are returning the raw response from our service layer directly via the controller.
  • The objects we receive from services are sent to the frontend without any transformation.
  • There is no consistent response format, which can cause problems for the frontend while handling data.

Why Response Transformation Is Important:​

  • In production-ready applications, each API typically follows a standardized response format.
  • A consistent response structure helps the frontend:
    • Parse the response easily.
    • Handle errors gracefully.
    • Render data efficiently.

Goal:​

  • To implement a global response transformation mechanism.
  • So that we don’t need to manually format the response in every controller method.

Solution:​

  • Use @RestControllerAdvice in Spring Boot.
  • This allows us to apply global changes to all API responses, enforcing a uniform structure automatically

Objective​

  • To create a global API response handler using @RestControllerAdvice and ResponseBodyAdvice in Spring Boot.
  • To return uniform API responses for all controller methods across the application.

Why Use a Global Response Handler?​

  • Currently, each controller returns raw objects directly from the service layer.
  • This leads to inconsistent API response formats, which are hard to manage in the frontend.
  • A standardized response format improves readability, maintainability, and error handling on the frontend

Setting Up the Global Response Handler​

  • Create a new class called GlobalResponseHandler.
  • Annotate it with @RestControllerAdvice to apply globally.
  • Implement the ResponseBodyAdvice<Object> interface instead of extending any class.
@RestControllerAdvice
public class GlobalResponseHandler implements ResponseBodyAdvice<Object> {
    // Methods will go here
}

Override Required Methods​

  • supports()
    • This method determines which responses should be intercepted.
    • Return true to apply it globally to all controller methods.
@Override
public boolean supports(MethodParameter returnType, Class<? extends HttpMessageConverter<?>> converterType) {
    return true;
}
  • beforeBodyWrite()
    • This method allows you to modify the response body before it is sent to the client.
    • Wrap the response inside a custom ApiResponse<T> class.
@Override
public Object beforeBodyWrite(Object body, MethodParameter returnType, MediaType selectedContentType,
                              Class<? extends HttpMessageConverter<?>> selectedConverterType,
                              ServerHttpRequest request, ServerHttpResponse response) {
    if (body instanceof ApiResponse) {
        return body; // Already wrapped
    }
    return new ApiResponse<>(body);
}

Define the ApiResponse<T> Class​

  • A generic class that can hold:
    • data: for successful responses
    • error: for error details (optional)
    • timestamp: for response time
public class ApiResponse<T> {
    private T data;
    private ApiError error;
    private LocalDateTime timestamp;

    // Default constructor
    public ApiResponse() {
        this.timestamp = LocalDateTime.now();
    }

    // Constructor for success
    public ApiResponse(T data) {
        this();
        this.data = data;
    }

    // Constructor for error
    public ApiResponse(ApiError error) {
        this();
        this.error = error;
    }

    // Getters (or use Lombok @Getter)
}

Result of Implementation​

  • Every API response will now return a consistent structure:
{
  "timestamp": "2025-06-18T07:30:00",
  "data": {
    "id": 1,
    "name": "John Doe"
  },
  "error": null
}

In case of an error:​

{
  "timestamp": "2025-06-18T07:30:00",
  "data": null,
  "error": {
    "code": "USER_NOT_FOUND",
    "message": "User with ID 10 not found"
  }
}

Benefits:​

  • Frontend developers know exactly where to look for data, error, and timestamp.
  • Ensures a uniform structure across all APIs.
  • Easy to extend in the future for metadata, pagination, etc.

Enhancing Error Handling Using Global Exception Handler in Spring Boot​

Current Approach:​

  • Errors are currently returned as a ResponseEntity<APIError>, which is not ideal or clean for a production-grade application.

Improvement Suggestion:​

  • Instead of directly returning APIError, we should wrap it inside a custom APIResponse object.
  • This way, all responses—success or error—can follow a consistent structure.

Implementation Steps:​

  • Create a helper method called buildErrorResponseEntity(APIError error) in your global exception handler.
  • This method should return ResponseEntity<APIResponse<?>>.

Inside the Method:​

public ResponseEntity<APIResponse<?>> buildErrorResponseEntity(APIError error) {
    return new ResponseEntity<>(new APIResponse<>(error), error.getStatus());
}
  • Here, APIResponse wraps the APIError.
  • Status code is extracted from the error object using error.getStatus().

Handling Generic Type Warning:​

  • To resolve warnings related to raw types in generics, use wildcards
ResponseEntity<APIResponse<?>> response = ...

Refactor Existing Code:​

  • Replace all direct error responses with:
return buildErrorResponseEntity(apiError);

Code Organization:​

  • Keep internal/helper methods like buildErrorResponseEntity() at the bottom of the class.
  • Keep external/public methods at the top for better code readability.

Benefits of the New Structure​

Consistent API Responses:​

  • Whether the response contains data or an error, the structure remains uniform:
{
  "timestamp": "...",
  "data": null,
  "error": {
    "message": "...",
    "status": "BAD_REQUEST"
  }
}

Testing the Error Response:​

  • When an error occurs, the data is null, and the error field contains detailed error information including sub-errors and status.

Formatting Timestamp:​

  • You can customize the timestamp format using Jackson's @JsonFormat annotation:
@JsonFormat(pattern = "HH:mm:ss dd-MM-yyyy")
private LocalDateTime timestamp;
  • Ensure correct pattern usage:
    • HH = hour
    • mm = minute
    • ss = second
    • dd = day
    • MM = month (capital M)
    • yyyy = year

Final Result​

Professional Output:​

  • Clean and consistent response format.
  • Easily consumable by frontend developers.
  • Production-ready and scalable error handling mechanism.

Frontend Developer-Friendly:​

  • Developers will appreciate this structured format as it makes frontend error handling and UI display much easier.
