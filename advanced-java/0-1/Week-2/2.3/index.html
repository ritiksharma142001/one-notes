Persistence Layer & JPA

  • Understand entities and repositories.
  • Learn how to connect a Spring Boot application to a database.
  • Get introduced to JPA (Java Persistence API), now called Jakarta Persistence API.

Java Persistence API - JPA​

Understanding JPA and Databases​

  • JPA is a standard specification used to manage relational data in Java applications.
  • It's now referred to as Jakarta Persistence API, but still abbreviated as JPA.
  • JPA helps in object-relational mapping (ORM) — converting Java objects to database tables.

What is a Database?​

  • A database is simply a server that stores data on secondary memory.
  • It can be connected to your application using drivers.
  • Popular databases (e.g., MySQL, PostgreSQL) provide drivers for various programming languages like Java, Python, and JavaScript.

How Connection Works​

  • Traditionally, to connect Java to a database:
    • Use JDBC (Java Database Connectivity).
    • Provide URL, username, and password to connect to the server.
    • Write raw SQL queries to fetch or store data.

Role of Spring Data JPA​

  • Spring Boot Starter Data JPA simplifies this process by:
    • Auto-configuring JDBC and the database driver.
    • Providing a JPA provider (commonly Hibernate).
    • Eliminating the need to manually configure connections or write boilerplate SQL.

What is Hibernate?​

  • Hibernate is the default JPA provider in Spring Boot.
  • Other providers include EclipseLink, OpenJPA, etc.
  • Hibernate handles:
    • ORM (Object-Relational Mapping).
    • Conversion of Java objects to database tables.
    • Generating optimized SQL queries.
    • Executing JPQL (Java Persistence Query Language) which is later converted to SQL.

Benefits of Using Hibernate​

  • Developers only need to work with Java objects.
  • Hibernate handles:
    • Object-to-table conversion.
    • Executing SQL on the database.
    • Fetching results and mapping them back to Java objects.

Layered Architecture Summary​

  • You can directly use JDBC to interact with databases — but that involves manual query handling.
  • Using JPA + Hibernate, the process is abstracted and automated.
  • On top of Hibernate, Spring Data JPA adds support for repositories (interfaces) to simplify CRUD operations.

Practical Setup for the Course​

  • To implement all the above concepts, we need:
    • Spring Data JPA dependency – to perform persistence operations.
    • A database – for now, we’ll use an in-memory database called H2.

Why Use H2 Database?​

  • H2 is a lightweight, in-memory database used for development and testing.
  • It:
    • Emulates a real database.
    • Requires no installation or configuration.
    • Lets us practice all JPA operations without needing to set up MySQL or PostgreSQL.

Setting Up Spring Boot with JPA and H2 Database​

Adding Dependencies in pom.xml​

  • Open the pom.xml file of your Spring Boot project.
  • Add the following dependencies:
    • spring-boot-starter-web
    • spring-boot-starter-data-jpa
    • com.h2database:h2
  • Remove the version and scope from the H2 dependency since they are auto-configured via the Spring Boot parent dependency.

Dependency Management​

  • Spring Boot’s parent POM manages versions for commonly used libraries.
  • Therefore, no need to manually set the version for H2.

Creating the Entity Class​

Defining the Entity​

  • Create a class named EmployeeEntity.
  • Annotate the class with @Entity to indicate it should be mapped to a database table.
  • Optionally use @Table(name = "employees") to specify a custom table name.
  • If @Table is omitted, the table name will default to the class name (employee_entity).

Field Definitions​

  • Copy all necessary fields from the DTO class to the Entity class.
  • This includes fields like id, name, email, etc.
  • Though it may seem like code duplication, DTO and Entity serve different purposes:
    • DTO: Used for request/response and input validation.
    • Entity: Represents the table schema and includes all persistable fields (including sensitive data like passwords).

DTO vs Entity Clarification​

Purpose of DTO​

  • Used to send and receive only necessary and safe data.
  • Can include validation annotations like @NotNull, @Email, etc.

Purpose of Entity​

  • Represents the database table.
  • Defines how data is stored, including constraints like uniqueness, indexes, and data types.

Primary Key and ID Generation​

Defining the Primary Key​

  • Use @Id to mark the primary key field (usually id).
  • Annotate with @GeneratedValue to auto-generate the value.

Choosing ID Generation Strategy​

  • Use strategy = GenerationType.AUTO for automatic strategy selection.
  • This allows Hibernate to choose between SEQUENCE and IDENTITY based on the database type.
  • Example:
@Id
@GeneratedValue(strategy = GenerationType.AUTO)
private Long id;

Repository in Spring Data JPA​

Why Not Use Entity Directly for Operations?​

  • Performing operations directly on the entity class is not advisable.
  • A separate layer is needed to handle data access operations — this is where a repository comes in.

What Does the Repository Do?​

  • It handles CRUD operations (Create, Read, Update, Delete) for the entity.
  • It abstracts the data access layer and interacts with the database.

Creating the Repository​

  • Create an interface named EmployeeRepository.
  • Instead of writing a class and manually implementing methods, define an interface.
  • This interface will extend JpaRepository, provided by Spring Data JPA.

Syntax Example:​

public interface EmployeeRepository extends JpaRepository<Employee, Long> {
}
  • Employee is the entity class.
  • Long is the type of the entity's primary key (id field).

Before Spring Data JPA:​

  • Developers manually created repository classes.
  • They defined each method for querying the database.
  • Implementing queries like findById(), save(), delete() required custom code.

After Spring Data JPA:​

  • All basic CRUD methods are auto-generated.
  • No implementation required — just extend the JpaRepository.

Behind the Scenes:​

  • JpaRepository is implemented by SimpleJpaRepository.
  • It provides default implementations for all basic operations.
  • All methods are transactional by default.

Inherited Methods Include:​

  • save(), saveAll()
  • findById(), findAll()
  • deleteById(), deleteAll()
  • count(), etc.

Custom Query Methods Using Method Naming Convention:​

  • You can define custom queries by method name, without writing SQL or JPQL.
  • Example:
List<Employee> findByName(String name);
  • Spring Data JPA will auto-generate the query.

Complex Queries (To Be Covered Later):​

  • Queries like findBySalaryGreaterThan(double salary) can be defined similarly.
  • These will be covered in the next module (custom queries).

Best Practices - Layered Architecture:​

  • Do not inject the repository directly into the controller.
  • Instead, use a service layer in between:
    • Controller → Service → Repository
  • This maintains clean separation of concerns and improves testability.

Controller Layer – Dependency Injection and Repository Use​

Setting Up the Controller​

  • We navigate to the EmployeeController.
  • The controller depends on EmployeeRepository.
  • We declare:
private final EmployeeRepository employeeRepository;
  • We use constructor injection:
public EmployeeController(EmployeeRepository employeeRepository) {
    this.employeeRepository = employeeRepository;
}

Enabling Dependency Injection​

  • This setup will not work unless EmployeeRepository is registered as a Spring Bean.
  • To do that, annotate the repository interface with:
@Repository
  • Since @Repository is a specialization of @Component, it will be automatically detected and injected.

Using Repository Methods in Controller​

Get Employee by ID​

  • Use:
employeeRepository.findById(id)
  • This method returns:
Optional<Employee>
  • Why Optional?
    • Prevents null pointer exceptions.
    • Wraps the result to handle absent values gracefully.
    • Example:
Optional<Employee> optionalEmployee = employeeRepository.findById(id);
return optionalEmployee.orElse(null);

DTO vs Entity in Controller​

  • The repository returns Entity, but controller should return DTO.
  • For now, we're using the entity directly to simplify.
  • Note: This is not a recommended practice. You should:
    • Use DTOs in the controller.
    • Map entities to DTOs using a mapping layer (e.g., ModelMapper or manual mapping).
  • Later, when the service layer is added, entities and repositories will be removed from the controller.

Get All Employees​

public List<Employee> getAllEmployees() {
    return employeeRepository.findAll();
}

Create a New Employee​

public Employee createEmployee(@RequestBody Employee employee) {
    return employeeRepository.save(employee);
}
  • This method:
    • Saves the employee to the database.
    • Returns the saved employee.
    • Does not return Optional since saving is expected to succeed.

Spring Boot with H2 Database Configuration and Lombok Integration​

Basic CRUD Operations​

  • Three basic methods are implemented:
    • createEmployee() – to add an employee.
    • getAllEmployees() – to fetch all employees.
    • getEmployeeById() – to fetch an employee by ID

Using the H2 Database​

  • Spring Boot automatically configures the H2 in-memory database by default.
  • Default configurations (URL, username, password) are auto-assigned.
  • You can run the application directly without any explicit DB configuration, and Spring Boot will set up the H2 database automatically.

Understanding In-Memory Database​

  • H2 (in-memory) database is volatile:
  • It resets every time the application stops.
  • Not suitable for production environments.
  • Ideal for learning and development purposes.

Creating and Fetching Data​

  • An employee named Ram is created with a joining date and active status.
  • GET request is made to retrieve data.
  • Data appears incomplete due to missing getters and setters.

Fix: Missing Getters, Setters, and Constructors​

  • Problem: Entity fields are private and have no accessors.
  • Solution:
    • Instead of writing them manually, use Lombok annotations.

Lombok Integration​

  • Add Lombok dependency in pom.xml:
<dependency>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
</dependency>
  • Spring Boot’s parent dependency already manages the version, so you can omit version and scope.

Useful Lombok Annotations​

  • @Getter – Generates getters for all fields.
  • @Setter – Generates setters.
  • @AllArgsConstructor – Generates constructor with all fields.
  • @NoArgsConstructor – Generates a no-argument constructor.
  • Avoid using @EqualsAndHashCode directly in entities if you're working with bidirectional relationships.

Verifying the Data Flow​

  • After using Lombok, the application successfully:
    • Saves employee data.
    • Retrieves all employees (e.g., id=1, id=2).
  • Data is now persisted properly in the in-memory DB.

Switching to a File-Based H2 Database​

  • In-memory database data is lost on shutdown.
  • File-based H2 database retains data between application restarts.

Configuration in application.properties​

# Use file-based H2 database instead of in-memory
spring.datasource.url=jdbc:h2:file:/path/to/your/database/spring-boot-web-tutorial-db

# Set username and password
spring.datasource.username=anuDB
spring.datasource.password=Anu123

# Enable H2 Console
spring.h2.console.enabled=true

Replace /path/to/your/database/ with your actual directory path.​

  • Make sure the database name (spring-boot-web-tutorial-db) is meaningful and relevant to your project.

Accessing the H2 Console​

  • Run the application and visit:
http://localhost:8080/h2-console
  • In the console:
    • Enter the same JDBC URL you defined in application.properties.
    • Enter the username and password (anuDB, Anu123).
    • Click Connect to view and manage the database tables.

Spring Boot - Data Persistence with H2 Database​

Connecting to H2 Database:​

  • Copied the JDBC URL from H2 console and pasted it into the connection setup.
  • Used the credentials:
    • Username: anuDB
    • Password: Anu123
  • Successfully connected to the database.

Issue on First POST Call:​

  • Encountered an internal server error: Error: employee_sequence not found.
  • Reason: The @GeneratedValue strategy for the id field in the Employee entity expects a sequence table which doesn’t exist.
  • This issue occurs because the database is pre-created and doesn’t allow Spring Boot to auto-generate schema.

Solution – Changing DDL Auto Configuration:​

  • In application.properties, added:
spring.jpa.hibernate.ddl-auto=create
  • This instructs Spring Boot to create the schema every time the application starts.

Observation:​

  • After updating the DDL property and restarting the app, the employee_sequence table and other schema were generated.
  • Successfully inserted data into the table (e.g., employee named "Ram").

Viewing Data in H2 Console:​

  • After re-connecting with the credentials, verified:
    • employee table contains inserted records.
    • employee_sequence table was created.
    • Used SELECT * FROM employee; to view the data.

Changing DDL Strategy for Data Persistence:​

  • After the initial creation, changed the DDL property to:
spring.jpa.hibernate.ddl-auto=update
  • This prevents the table from being dropped on application restart.
  • Now, even after closing and re-running the app, existing data is retained.

Persistence Location:​

  • Data is stored in the file system.
  • The H2 database file path can be checked locally where the data persists.

Entity Annotation Recap:​

  • @Entity: Marks a class as a JPA entity representing a database table.
  • Hibernate handles object-relational mapping (ORM) behind the scenes.

JPA Repository Overview:​

  • Interface provides built-in CRUD methods like:
    • save(), findAll(), findById(), deleteById() etc.
  • Supports writing custom queries, which will be covered in the next module.

package com.example.springBootWeb.springBootWeb.entities;

import jakarta.persistence.*;
import lombok.*;

import java.time.LocalDate;

@Entity
@Getter
@Setter
@AllArgsConstructor
@NoArgsConstructor
@Table(name = "employees")
public class EmployeeEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Long id;
    private String name;
    private String email;
    private Integer age;
    private LocalDate dateOfJoining;
    private Boolean isActive;
}

import org.springframework.web.bind.annotation.PathVariable;
import org.springframework.web.bind.annotation.RestController;

import org.springframework.web.bind.annotation.*;

import java.time.LocalDate;
import java.util.List;

@RestController

public class EmployeeController {

    private final EmployeeRepository employeeRepository;

        this.employeeRepository = employeeRepository;
    }

    @GetMapping(path = "/{employeeId}")
    public EmployeeEntity getEmployeeById(@PathVariable(name = "employeeId") Long id) {
        return employeeRepository.findById(id).orElse(null);

        return employeeRepository.save(inputEmployee);

}
