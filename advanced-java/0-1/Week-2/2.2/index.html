Presentation Layer

Presentation Layer and Controller​

Introduction to Web Layer Architecture​

  • Now that we understand how the web layer architecture works, let's begin exploring its components.
  • We will start with the first layer: the Presentation Layer.

Purpose of the Presentation Layer​

  • It defines the logic required for communication between the client and the Spring Boot application.
  • This layer serves as the first point of contact for incoming client requests via HTTP connections.

Role of the Presentation Layer​

  • It receives input from the client and returns output back to the client.
  • The controller is responsible for handling communication with external clients over the internet.

Scope of Today’s Class​

  • Focus on the Presentation Layer, specifically:
    • Controller
    • DTO (Data Transfer Object)
  • We will explore:
    • Different types of requests handled by the controller
    • How the controller interacts with clients

Upcoming Topics​

  • In future sessions, we will cover:
    • Persistence Layer: Repository and Entity
    • Service Layer: Business logic

Understanding Controller Behavior​

  • Before creating our first custom controller and API logic, it’s important to understand how the client interacts with the controller.
  • When you run the Spring Boot application, the server starts and does the following:
    • Scans all defined controllers in the code.
    • Informs the DispatcherServlet (Spring’s front controller) about each mapping.

Role of the DispatcherServlet​

  • It maps each incoming HTTP request to the appropriate controller method.
  • For example:
    • A GET request to a specific URL will be routed to the matching controller method as defined by annotations (like @GetMapping).
    • This mapping is critical for the functioning of the presentation layer.

Creating the Controller​

  • Go to the controllers package/folder in your project.

  • Create a new controller class, e.g., EmployeeController.

├── SpringBootWebApplication.java
├── configs
├── controllers
│   └── EmployeeController.java
├── dto
├── entities
├── repositories.java
├── services
└── utils

Defining the Controller​

  • This controller will handle logic for the Employee entity.
  • Annotate the class with @RestController:
@RestController
public class EmployeeController {
    // endpoint methods go here
}
  • The @RestController annotation is a combination of:
  • @Controller
  • @ResponseBody

Purpose of Annotations​

  • @RestController ensures all HTTP request methods inside the class return data directly as JSON or XML (commonly JSON).
  • If using only @Controller, you would need to annotate each method with @ResponseBody to ensure data is returned properly.
  • Using @RestController removes the need to manually add @ResponseBody on each method.

Annotated Controller​

Understanding @RestController and Request Mapping in Spring Boot

Purpose of @RestController​

  • By annotating a class with @RestController, we confirm that:
    • This controller will handle RESTful web services.
    • It will conform to REST API conventions. -The @RestController annotation simplifies the development of REST APIs in Spring Boot.

Defining Request Mappings​

  • Inside a @RestController, we can define various HTTP method mappings, such as:
    • @GetMapping
    • @PostMapping
    • @PutMapping
    • @DeleteMapping
  • These mappings allow us to handle different types of client requests (GET, POST, etc.).

Annotation-Based Programming Model​

  • Spring Web MVC provides an annotation-based programming model.
  • Controllers (@Controller or @RestController) use annotations to:
    • Map HTTP requests to handler methods.
    • Handle request input (e.g., parameters, request bodies).
    • Manage exception handling (covered later in the course).

Difference Between @Controller and @RestController​

  • @RestController is a specialized version of @Controller.
  • It is a shorthand that combines:
    • @Controller
    • @ResponseBody
  • This means:
    • Every method in a @RestController automatically returns the response as JSON or XML, directly to the client

Role of @ResponseBody​

  • If we use @Controller, we must add @ResponseBody to each method to ensure it returns a serialized response.
  • With @RestController, this is not necessary—all methods automatically act as if @ResponseBody is present.

Automatic JSON and XML Conversion​

  • When we added the spring-boot-starter-web dependency, it included a library called Jackson.
  • Jackson's responsibility is to:
    • Automatically convert Java objects to JSON/XML in the response.
    • Automatically convert JSON/XML in the request to Java objects.

Understanding Request Mapping in Spring Boot​

Introduction to Request Mapping​

  • In this lecture, we will explore how to define various types of request mappings in a Spring Boot REST controller.
  • The @RequestMapping annotation is used to map an incoming HTTP request to a controller method.

Features of @RequestMapping​

  • The @RequestMapping annotation provides several attributes to match:
    • URL paths
    • HTTP methods (GET, POST, PUT, DELETE)
    • Request parameters
    • Headers
    • Media types (e.g., JSON, XML)

Observing REST APIs in Action​

  • On any modern website, multiple REST APIs are constantly in use.
  • Using browser tools (e.g., Chrome DevTools), you can:
    • Reload the page
    • Inspect the Network tab
    • Observe the details such as:
      • Request method (e.g., GET)
      • Request URL
      • Response status code
      • Remote address
      • Response headers
      • Response body in JSON format

Defining a GET Mapping in Spring Boot​

  • The @GetMapping annotation is a shortcut for @RequestMapping(method = RequestMethod.GET).
  • It is used to map HTTP GET requests to a specific handler method.

Example Usage​

  • In a Spring Boot controller, we can define a method like this:
@GetMapping("/get-secret-message")
public String getSecretMessage() {
    return "This is your secret message: xyz123";
}
  • This defines:
    • A resource at the URL path /get-secret-message
    • The method that handles GET requests to this URL
    • A return value (in this case, a secret string) sent back to the client

Understanding URL Structure​

  • The URL acts as the Uniform Resource Locator (URL) for the resource.
  • Once mapped, this URL becomes accessible over the web.
  • For example:
    • If the application is hosted at http://localhost:8080
    • Then, accessing http://localhost:8080/get-secret-message will return the defined string.

Role of Component Scanning​

  • Spring Boot automatically detects and configures components using component scanning.
  • This is enabled by the @SpringBootApplication annotation, which includes @ComponentScan.
  • What it does:
    • Scans the base package and its sub-packages
    • Detects classes annotated with @RestController, @Service, @Component, etc.
    • Registers them as beans in the Spring container

Internal Working of Request Mapping​

  • When the application starts:
    • The component scan detects the controller and its annotations.
    • The DispatcherServlet is notified of the mappings.
    • When a request is received:
      • The DispatcherServlet matches the URL to the appropriate handler method.
      • It then calls the method and returns the response.

Testing the Endpoint​

  • After running the application:
    • Navigating to /get-secret-message in the browser or Postman shows:
      • The message returned by the controller method.
    • This demonstrates that the resource is now publicly accessible (if deployed online).

Building a GET Mapping in Spring Boot to Return Employee Data​

  • Now, I will define a new @GetMapping method that will allow us to retrieve employee data.

Creating the EmployeeDTO Class (Data Transfer Object)​

  • We will define an object of type EmployeeDTO.
  • DTO (Data Transfer Object) is used to transfer data:
    • From the client to the controller.
    • From the controller to the service layer.
    • It acts as a middle layer and does not interact with the persistence layer (e.g., database).

Defining Fields for the EmployeeDTO Class​

  • The EmployeeDTO class will contain the following fields:
    • private Long id; – Represents the employee's unique ID.
    • private String name; – Represents the employee's name.
    • private String email; – Represents the employee's email.
    • private Integer age; – Represents the employee's age.
    • private LocalDate dateOfJoining; – Represents the joining date of the employee.
    • private Boolean isActive; – Indicates if the employee is currently active.

DTO Structure​

  • This EmployeeDTO class is a POJO (Plain Old Java Object).
  • It:
    • Does not contain any business logic.
    • Is solely used for transferring structured data in the application.

Implementing the GET Method to Return DTO​

  • In the controller, we will write a method to return an object of EmployeeDTO.
  • This method will be:
public EmployeeDTO getEmployeeById(){

}
  • We will later accept the id as an input parameter (e.g., via @PathVariable or @RequestParam) to fetch the specific employee.

Dynamic URLs Paths​

Using @GetMapping and Path Variables in Spring Boot

GetMapping Setup:​

  • Initially, a @GetMapping endpoint is created to fetch data.
  • A DTO (Data Transfer Object) named EmployeeDTO is defined to structure the response data.
  • The EmployeeDTO includes fields such as id, name, email, age, dateOfJoining, and isActive.

Purpose of DTO:​

  • DTOs are used to transfer data between the client and the controller/service layer.
  • They are not used for database persistence; that responsibility lies with the entity classes.

Field Definitions in DTO:​

  • All fields are marked as private.
  • Getters, setters, a default constructor, and a parameterized constructor are defined for data access and object creation.
  • This class is a POJO (Plain Old Java Object), with no business logic.

Endpoint Method Design:​

  • A controller method is created to return an employee based on their ID.
  • The method accepts an employee ID as input and returns an EmployeeDTO.

Types of Mappings in Spring:​

  • Spring supports various mappings: @GetMapping, @PostMapping, @PutMapping, @DeleteMapping, and @PatchMapping.
  • This example focuses on @GetMapping for demonstration purposes.

Passing Data Through URL:​

  • There are multiple ways to pass data from the client to the server, such as:
  • @PathVariable
  • @RequestParam
  • In this case, @PathVariable is used because the employee ID is essential to fetch specific data.

Why Use PathVariable:​

  • @PathVariable is ideal when the parameter is a mandatory part of the URL path.
  • It ensures that a specific resource is identified directly via the URL (e.g., /employees/id).

Method Parameter Setup:​

  • The employee ID is received as a parameter using @PathVariable.
  • The method signature becomes:
public EmployeeDTO getEmployeeById(@PathVariable("id") Long employeeId)
  • Ensure the path variable name in the annotation matches the name in the URL mapping.

Returning a Sample DTO Object:​

  • A hardcoded EmployeeDTO object is returned for demonstration.
  • Example values are assigned to each field, including a LocalDate for date of joining and a Boolean for active status.

Automatic JSON Conversion:​

  • Spring Boot automatically converts Java objects to JSON using the Jackson library.
  • Developers do not need to manually convert the object; Jackson handles the serialization.

Testing the Endpoint:​

  • When accessing the endpoint (e.g., /employees/1), a JSON response is returned with the hardcoded employee details.
  • Changing the ID in the URL (e.g., /employees/100) reflects the new value in the response.

Mandatory Nature of PathVariable:​

  • Since the employee ID is part of the URL, omitting it results in an error.
  • This reinforces that the ID is a required field for the operation getEmployeeById.

Why Not Use RequestParam Here:​

  • @RequestParam is typically used for optional or query string parameters (e.g., /employees?id=1).
  • Since the ID is essential and identifies the resource, @PathVariable is the more appropriate choice.

urpose of Request Parameters:​

  • Request parameters are used to pass optional inputs like filtering, sorting, and tracking (e.g., UTM parameters).
  • These do not break the application if omitted.

Real-World Example:​

  • Visiting a sponsored Amazon link shows additional info (e.g., tag, ref, id) in the URL.
  • Even without these parameters, the main site (amazon.in) works fine.

Use Case in a Controller:​

  • Define a method to retrieve all employees and optionally sort/filter them.
  • Use @RequestParam to capture optional query parameters.

Example Controller Method:​

  • Example Controller Method:
@GetMapping("/employees")
public String getAllEmployees(@RequestParam(required = false) Integer age) {
    return "Hi, age: " + age;
}
  • URL: /employees?age=12 → Output: Hi, age: 12

Passing Multiple Parameters:​

  • You can add multiple optional parameters:
public String getAllEmployees(@RequestParam(required = false) Integer age,
                              @RequestParam(required = false) String sortBy)
  • URL: /employees?age=12&sortBy=name → Output: Hi, age: 12, sortBy: name

Handling Missing Parameters:​

  • By default, @RequestParam is required.
  • To make it optional, use: @RequestParam(required = false)

Improving Code Structure:​

  • Instead of repeating /employees in each endpoint, use @RequestMapping("/employees") at the class level.
  • Now, all endpoints inside the class inherit this base path.

Customizing Path Variable Names:​

  • Example:
@GetMapping("/{employeeId}")
public String getEmployee(@PathVariable("employeeId") Long id)
  • Even if the URL uses employeeId, you can map it to a variable named id in code.

Customizing Request Parameter Names:​

  • You can rename request parameters:
@RequestParam(name = "inputAge", required = false) Integer age
  • Now, the request must use ?inputAge=30 instead of ?age=30.

Types of Request Mappings in Spring Boot​

  1. @GetMapping Used to retrieve resources from the server.

Example: Fetching a list of employees.

  1. @PostMapping Used to create a new resource on the server.

While it's technically possible to use @GetMapping to create data, this is not recommended.

Use @PostMapping for resource creation to follow RESTful conventions.

  1. @PutMapping Used to update an entire existing resource.

Example: Update an employee record completely.

  1. @DeleteMapping Used to delete a resource by ID or any identifier.

  2. @PatchMapping Used to partially update a resource, e.g., updating only the email field of an employee.

Important Notes on HTTP Methods​

  • It is possible to assign the same URL path to different request mappings as long as they have different HTTP methods (GET, POST, etc.).
  • Browsers by default send GET requests, not POST or others.
  • To test POST, PUT, or DELETE mappings, you need to use tools like Postman or build a front-end client.

Using Postman to Test APIs​

  • Postman is a popular tool for testing APIs during development.

  • To test a POST endpoint:

    • Open Postman.
    • Set the method to POST.
    • URL: http://localhost:8080/employees

  • In the Body section, choose raw and set type to JSON.

  • Send your JSON data, e.g.:

{
  "name": "John",
  "department": "HR"
}
  • You’ll receive the response as defined in your controller.

Example: Handling POST Request with @RequestBody​

  • Suppose you want to create a new employee with a POST request:
@PostMapping("/employees")
public EmployeeDTO createEmployee(@RequestBody EmployeeDTO inputEmployee) {
    inputEmployee.setId(100L); // Set ID manually
    return inputEmployee;
}
  • Key points:
  • Use @RequestBody to bind the incoming JSON to a Java object.
  • You can modify the object (e.g., assign an ID) before returning or saving it.
  • In real-world apps, you would save the data to a database, which will be covered later.

public class EmployeeDTO {

    private Long id;

    }

    }

    public void setName(String name) {
        this.name = name;
    }

    public String getEmail() {
        return email;
    }

    public void setEmail(String email) {
        this.email = email;
    }

    public Integer getAge() {
        return age;
    }

    public void setAge(Integer age) {
        this.age = age;
    }

    public LocalDate getDateOfJoining() {
        return dateOfJoining;
    }

    public void setDateOfJoining(LocalDate dateOfJoining) {
        this.dateOfJoining = dateOfJoining;
    }

    public Boolean getActive() {
        return isActive;
    }

    public void setActive(Boolean active) {
        isActive = active;
    }
}

Passing Data to Controller in Spring Boot​

  • Demonstrated how to pass complete data using @RequestBody in a @PostMapping method.
  • If no request body is passed, it results in a "Bad Request" error because the required employee data is missing.
  • To resolve this:
    • Go to Postman > Body tab.
    • Select "raw" and choose "JSON" as the data format.
    • Provide employee details in JSON format. Example:
{
  "name": "Ram",
  "email": "ram@codingshuttle.com",
  "age": 23
}
  • Even if some fields are omitted (e.g., dateOfJoining, active), the backend handles them appropriately (e.g., sets null).
  • On submitting this request:
    • A new employee (Ram) is created and stored (currently in a mock system, not a real database).
    • The id is auto-generated.
    • The JSON payload is automatically converted to a Java object.
    • No need for manual mapping — handled automatically using Lombok and Jackson.

Summary of Data Passing Techniques​

The three main ways to pass data into a Spring Boot controller:

  • @PathVariable – Data passed in the URL path (e.g., /employee/{id})
  • @RequestParam – Data passed as query parameters (e.g., /employee?id=1)
  • @RequestBody – Data passed in the request body as JSON
