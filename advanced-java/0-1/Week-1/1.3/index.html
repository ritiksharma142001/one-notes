Beans

What are Beans?​

Beans are the basic building blocks of a Spring application. They are classes whose objects, lifecycles, and dependencies are managed by the Spring framework, so we don't have to create objects or handle their dependencies ourselves. In short, beans are the core components that make up Spring Boot applications.

Why beans are important?​

  • Beans help us add dependencies (things your class needs) into our code, so the dependency injection system can work smoothly.
  • Now, you’ve already used Java before, right? For example, if you have a class called Car, you would normally create an object like this:
Car car = new Car(); // using 'new' keyword
  • This uses the new keyword to call the constructor, and then Java creates the object in heap memory.
  • But in Spring, this process is different. We don't manually create the object — instead, Spring does it for us. This object is now called a bean.
  • So, you don’t need to use new anymore to create objects. Spring takes care of that automatically.
  • In Spring, you rarely write new Car() yourself. Most of the time, Spring will give you the object directly using dependency injection.
  • A bean is a managed object that is created, assembled, and managed by the Spring IoC container.
  • For example, suppose your Car class needs some values like color, model, engine name, etc. Spring will provide all of that automatically.
  • Spring will create the object, assemble it with what it needs, and manage it (like reusing or deleting it when not needed).
  • You don’t need to manually create even a single object. Spring and dependency injection will handle it all.
  • Beans are the backbone of Spring applications. They are the core building blocks that are connected (wired) together.
  • In Java, we use multiple classes wired together — and in Spring, these are beans wired to form a full application.
  • All the large applications made by companies like Google or Amazon are basically made of many beans working together, based on how they’re configured.

What Happened Behind the Scenes above?

  • Spring automatically created the object of Car class.
  • Spring injected it into the Person class using @Autowired.
  • You did not write new Car() — Spring did it for you.
  • The Car object is now a Spring Bean.
  • ApplicationContext — a special object that stores and manages all your beans (classes annotated with @Component, @Service, etc.).

Spring Annotations​

  • Earlier, we used XML files to create beans in Spring Framework.
  • But with Spring Boot, we now use Java-based annotations instead of XML.
  • Java developers are more comfortable with Java, so annotations made it easier.
  • Writing XML files for configuration was complex and repetitive.
  • Spring Boot provides annotations like @Bean, @Component, and @Service.
  • These annotations help us define beans and configurations in Java code.
  • You’ve probably already seen annotations like @Override, @Deprecated, etc.
  • Annotations provide metadata to the compiler or Spring framework.
  • This metadata helps Spring know how to create, manage, or inject objects.
  • Spring annotations make it easy to define and manage beans programmatically, without XML.

Stereotype​

What’s the same?

  • All stereotype annotations tell Spring to treat the class as a Bean — so Spring creates and manages objects of those classes automatically.
  • They all make the class a candidate for dependency injection (Spring can give that object wherever needed).

What’s different?

  • They give different meanings or roles to Spring internally. For example:

  • @Service means the class holds business logic — Spring may apply some service-specific logic or tools.

  • @Repository means the class is related to database or data access — Spring can automatically translate database exceptions into Spring’s DataAccessException.

  • @Controller means the class handles web requests — Spring treats it as a web controller.

  • @Component is generic — it just tells Spring to manage this class as a bean but doesn’t specify any special role.

Why does this matter?

  • It helps Spring apply special behaviors depending on the stereotype.
  • It helps organize your code and make it clear what each class does.
  • It helps tools and developers understand your code better.

Basics annotations​

  • main starting point of the entire application.
  • A Spring Boot application should have only one class annotated with @SpringBootApplication to avoid confusion about where the main configuration resides.
  • You can think of this single entry point as the root from which the entire application is bootstrapped.
  • If you want to explore this annotation further, you can hold down the Command (Mac) or Control (Windows/Linux) key and click on the annotation in your IDE. This will take you to the compiled class definition of @SpringBootApplication.
  • There, you’ll see that it is a meta-annotation that itself is composed of multiple other annotations. Don’t worry about understanding all of those right now — we’ll cover them one by one as we go further in the course.

Inside @SpringBootApplication annotations​

  • some of the annotations that make up @SpringBootApplication are:-
  • The @Configuration annotation tells Spring that this class contains configuration details.
  • The @EnableAutoConfiguration annotation enables Spring Boot’s auto-configuration mechanism, which attempts to automatically configure your Spring application based on the dependencies present in the classpath. It tells Spring Boot to start auto-configuring the project starting from this package.
  • The @ComponentScan annotation tells Spring Boot to start scanning the package (and sub-packages) from this class location for any components, services, repositories, or controllers annotated with stereotype annotations.
  • The @Documented annotation indicates that this annotation should be included in the generated JavaDoc.
  • The @Target annotation defines the types of Java elements the annotation can be applied to (e.g., class, method, field, etc.).
  • The @Retention annotation defines how long annotations are retained — whether only in the source code, class files, or at runtime.

Important things SpringBootApplication​

  • If you navigate to the @SpringBootApplication annotation and drill down into the @Configuration annotation within it, you’ll notice that @Configuration itself is annotated with @Component.
  • This means that the class annotated with @SpringBootApplication is implicitly treated as a Spring component — in other words, it's managed by the Spring container as a bean.
  • So essentially, the class that serves as the entry point of your Spring Boot application is itself a bean.
  • Why is that the case?
  • Because @SpringBootApplication is a meta-annotation that ultimately inherits from @Component. And as we discussed earlier, any class annotated with @Component (or any annotation that inherits from it) is automatically registered as a bean in the Spring context.
  • This demonstrates a core principle in Spring Boot — everything is treated as a bean, including the main class.

code​

  • First, we create a simple class called Apple with a method eatApple() that prints "I am eating the apple."
  • Normally in Java, to use this class, we create an object using new Apple().
  • But in Spring Boot, we don't use new to create objects (beans). Instead, Spring manages those objects for us.
  • To tell Spring to manage the Apple class, we add @Component annotation on it. This makes it a Bean.
  • To use this bean inside our main application class, we use @Autowired to inject the Apple bean.
  • Since main is static, we cannot use the bean there directly. So we implement CommandLineRunner which runs after Spring initializes beans.
  • Inside the run() method (which is non-static), we use the injected Apple bean to call eatApple().
  • This way, Spring creates and injects the Apple object automatically, and we don't get errors.
import org.springframework.boot.CommandLineRunner;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.stereotype.Component;
import org.springframework.beans.factory.annotation.Autowired;

@Component
class Apple {
    public void eatApple() {
        System.out.println("I am eating the apple.");
    }
}

@SpringBootApplication
public class MySpringApp implements CommandLineRunner {

    @Autowired
    Apple apple;  // Inject Apple bean here

    public static void main(String[] args) {
        SpringApplication.run(MySpringApp.class, args);
    }

    @Override
    public void run(String... args) {
        apple.eatApple();  // Use the injected Apple bean here
    }
}

What is CommandLineRunner?​

  • magine your app starts like turning on a machine.
  • CommandLineRunner is a place where you can put code that you want to run right after the machine turns on.
  • It’s like a button that gets pressed automatically when your app starts.

Why do we use it?

  • The normal start part (main method) cannot use some special things in Spring easily.
  • So, Spring gives you CommandLineRunner to run code after everything is ready.
import org.springframework.boot.CommandLineRunner;

@Component
public class MyRunner implements CommandLineRunner {

    @Override

What happens?

  • When your Spring Boot app starts, it will automatically print: App started! Now I am running this code.

Why NOT use new keyword to create objects in Spring?​

  • When you do new Person(), you create a new object every time, but:
  • Spring apps can serve billions of requests.
  • Creating objects manually every time can lead to:
  • High memory usage (lots of objects created and discarded).
  • Difficult lifecycle management (who controls when objects are created/destroyed?).
  • Mixing business logic with object management, which complicates code.
  • So, Spring creates and manages objects (beans) for you, including their entire lifecycle.
  • This is done by the IoC (Inversion of Control) container, which uses Dependency Injection (DI).
  • You just declare what you need (like @Component, @Service annotations) and Spring handles the rest.

What is Dependency Injection?​

  • Spring injects (provides) dependencies (objects) wherever they are needed.
  • ou just mark the places where you want dependencies injected, for example:
@Autowired
private Person person;
  • Spring gives you the same shared bean or creates it once, and injects it automatically.
  • This is efficient, reusable, and clean.

What about the annotations like @Service, @Component, @Repository?​

  • These are stereotype annotations — special labels to tell Spring what role a class plays:
  • @Component: generic bean
  • @Service: service layer bean
  • @Repository: data access bean
  • @Controller: web controller
  • Internally, these annotations extend @Component, so they behave the same but make the code more readable and organized.
  • They don't add extra behavior, just clarify the purpose.

another way to create Spring beans​

  • another way to create Spring beans explicitly using configuration classes and @Bean methods, instead of relying on stereotype annotations like @Component or @Service.

What is explicit bean declaration?

  • You create a special configuration class (like AppConfig) and mark it with @Configuration.
  • Inside this class, you write factory methods that create and return beans.
  • Each factory method is annotated with @Bean.
  • Spring calls these methods to create and manage beans.

Why use explicit bean declaration?

  • You want more control over how and when beans are created.
  • You want to create beans for different use cases or configurations.
  • You want to conditionally load certain beans depending on environment or config.
@Configuration
public class AppConfig {

    @Bean
    public Apple getApple() {
        return new Apple();  // You manually create the object here with new keyword.
    }
}
  • Here, you manually create the Apple object in the method.
  • But Spring will manage this bean's lifecycle and inject it wherever needed.
  • You don't call getApple() yourself; Spring does it.

Important: Avoid duplicate beans

  • If Apple class itself has @Component annotation and you also create a bean explicitly in AppConfig, Spring will create two beans of the same type.
  • This causes an error: "required a single bean but found two."
  • So you need to either remove @Component from the class or remove the explicit @Bean method.

How to think about this:

  • @Component approach: Class is automatically scanned and registered as bean.
  • @Configuration + @Bean approach: You explicitly define how to create beans.
  • Both ways lead to Spring managing the bean lifecycle and dependency injection.

Why is this useful?

  • Explicit bean declaration lets you define beans from third-party libraries or classes you can't modify (no @Component there).
  • You can customize bean creation logic, for example:
  • Pass constructor parameters,
  • Set properties,
  • Create proxies or wrappers,
  • Or conditionally create beans based on environment.

Two methods of defining beans in Spring:​

  • Using stereotype annotations (such as @Component, @Service, @Repository, etc.).
  • Using explicit bean declaration in a configuration class via @Bean methods.

Now let’s explore the Spring Bean Life Cycle.

The bean life cycle refers to the series of stages a Spring bean goes through from creation to destruction. While Spring handles these stages automatically, several steps occur internally that we should understand.

  1. Bean Creation
  • A bean is first created. This can happen in two main ways:
  • Using a static factory method defined in a configuration class.
  • Using annotation-based configuration, such as the stereotype annotations.
  1. Dependency Injection
  • After the bean is created, Spring performs dependency injection.
  • For example, if we have an IntroductionToSpringBootApplication class that depends on an Apple class, both are created as beans.
  • Then Spring injects the Apple bean into the IntroductionToSpringBootApplication bean.
  • In reality, Spring Boot creates many more beans behind the scenes that we don’t immediately see. These can later be viewed using Spring Boot Actuator.
  1. Bean Initialization
  • Once all dependencies are injected, Spring initializes the bean.
  • At this point, the bean is marked as ready to use. The application can now use this bean anywhere it’s needed.
  • As part of initialization, you can define a method with the @PostConstruct annotation.
  • This method will be automatically invoked after the bean is initialized but before it is used.
  1. Bean Usage
  • The bean is now fully ready and is used by the application wherever it is required.
  1. Bean Destruction
  • When the Spring Boot application shuts down (for example, when you click the stop button in your IDE), the bean is no longer needed and is destroyed.
  • Spring invokes a destruction method at this stage. You can define a method annotated with @PreDestroy which will be executed before the bean is destroyed.

Bean Lifecycle and Lifecycle Hooks in Spring

  • We defined two methods in the Apple class. One method is executed before the bean is used, and another is executed before the bean is destroyed.
  • We can name the first method void beforeAppleUse() and inside it, print "Creating the Apple before use". However, if we simply define this method and run the Spring Boot application, it won’t be executed automatically. That’s because we haven’t yet notified Spring to invoke this method during the bean lifecycle.
  • To do this, we use the @PostConstruct annotation. This tells Spring’s component scanner that this method should be invoked immediately after the bean has been constructed and all dependencies injected. Once you annotate the method with @PostConstruct, Spring will automatically call it during the bean initialization phase.
  • After adding the annotation, when you run the application, you'll see the message "Creating the Apple before use" printed before the Apple bean’s actual usage, such as calling an eat() method.
  • Here's what happens:
  • The IntroductionToSpringBootApplication bean is created.
  • The Apple bean is created.
  • The Apple bean is injected into the IntroductionToSpringBootApplication bean.
  • The @PostConstruct method is called.
  • Then the bean is used (e.g., apple.eat() is called).
  • Now, let’s also define a @PreDestroy method. For example:
@PreDestroy
public void beforeDestroy() {
    System.out.println("Destroying the Apple bean");
}

  • When you run the application, the "Creating the Apple before use" and "I am eating the apple" messages are printed. However, "Destroying the Apple bean" is not printed immediately. This is because the bean is not destroyed until the application context is closed.

  • To trigger this, stop the application manually (e.g., by stopping it in the IDE). Once you close the application, the @PreDestroy method will be invoked, and the destruction message will be printed.

  • @PostConstruct: Runs once after the bean is created and dependencies are injected.

  • @PreDestroy: Runs just before the bean is destroyed, typically when the application shuts down.

These hooks are useful:

  • @PostConstruct: For initialization logic (e.g., loading config, setting default values).
  • @PreDestroy: For cleanup logic (e.g., closing database connections, releasing resources).

Bean Scope in Spring

  • Now, let’s talk about Bean Scope.
  • You might wonder: “Is Spring creating a new object every time a bean is used?” The answer is no by default.
  • Spring Boot uses the Singleton scope by default. That means:
  • Only one instance of a bean is created.
  • That same instance is reused wherever the bean is injected. -To demonstrate this, you can define:
@Autowired
Apple apple1;

@Autowired
Apple apple2;
  • Even though there are two variables (apple1, apple2), both refer to the same instance. If you run methods like apple1.eat() and apple2.eat(), both calls will operate on the same bean.
  • This is evident because:
  • The @PostConstruct method is printed only once (bean created once).
  • But eat() is executed twice (bean used twice).
  • So, Spring’s default behavior is:
  • Singleton scope: One instance per Spring container.
  • This is the most commonly used scope in real-world applications (used in over 95% of the cases in most companies).

Bean Scope in Spring Framework – Key Points

  1. Default Behavior
  • By default, Spring beans are of singleton scope.
  • This means only one instance of the bean is created and shared across the application context.
  1. Prototype Scope
  • In some use cases, we need different instances of a bean for each usage.
  • This is achieved by setting the bean scope to prototype using the @Scope("prototype") annotation.
@Scope("prototype")
  1. Behavior of Prototype Beans
  • Each time the bean is requested, a new object is created.
  • You can verify this by printing the hash code of the bean instances, which will be different for each one.
  1. Switching Back to Singleton
  • To revert to singleton behavior, simply use:
@Scope("singleton")  // or remove the annotation, since singleton is the default
  • Now, only one instance of the bean will be used across the application.
  • This can be confirmed by identical hash codes and a single object creation log.
  1. Usage Recommendation
  • In 95% of real-world cases, the singleton scope is sufficient and commonly used.
  1. Other Bean Scopes in Spring
  • Spring provides additional scopes to suit different scenarios:
  • Request Scope – One bean per HTTP request (used in web applications).
  • Session Scope – One bean per HTTP session.
  • WebSocket Scope – One bean per WebSocket session.
  1. When to Use Other Scopes
  • Use request scope when different API calls need their own bean instances.
  • Use session or websocket scopes for stateful interactions over HTTP/WebSocket.

  • The difference between singleton and prototype bean scopes.
