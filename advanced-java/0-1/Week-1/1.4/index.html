Dependency Injection

Spring Framework - Dependency Injection​

Dependency Injection is a design pattern and technique used to achieve loose coupling between components in a software application.

Dependency Injection (DI) — what it is and how it works.​

  • Dependency injection is a core feature of the Spring Framework, not just Spring Boot.
  • It plays a foundational role in building loosely coupled Spring Boot applications — making it one of the most important concepts you need to master.

Analogy for Dependency Injection​

  • Let's understand DI using a simple analogy:
  • Suppose there’s a girl named Alice, who runs a bakery and wants to bake a chocolate cake.
  • However, Alice doesn’t have all the ingredients in her bakery. So she has to go to the market herself to get items like frosting or syrup.
  • This process takes up a lot of her time and distracts her from what she actually enjoys — baking.
  • To solve this, Alice hires Bob to take care of gathering the ingredients. Now, whenever she needs frosting or syrup, Bob brings them.
  • So, Alice has delegated the task of getting dependencies to Bob.

Applying the Analogy to Spring​

  • In a Spring application, you (the developer) are like Alice, and your code is your bakery.
  • Your code often depends on other components, libraries, or services — these are your dependencies.
  • Instead of managing these dependencies manually, you delegate that responsibility to Spring’s Dependency Injection framework — like Bob.
  • Now, Spring is responsible for providing required dependencies when and where needed, based on configuration.

Why Use Dependency Injection?​

  • No manual dependency management: You don't need to write code to create and manage object lifecycles. Spring handles it for you.
  • Loose coupling: Components are not tightly bound to specific implementations. This makes your code more flexible, modular, and easier to test or change.
  • For example, earlier Alice’s bakery only supported chocolate cake. In a tightly coupled system, she would need a whole new bakery to make a strawberry cake.
  • But with loose coupling, she can reuse the same bakery and just change the ingredients.
  • In Spring, this translates to changing configurations or injecting different beans, without changing the entire application structure.
  • For instance, your application can work with different databases (development, staging, production) just by changing the configuration — without rewriting core logic.

DI Scenario​

  • Instead of a component creating its own dependencies,
  • The dependencies are injected into the component from an external source — usually managed by a framework like Spring.
  • You don’t have to manually configure or manage your dependencies.
  • You don’t need to instantiate or inject them yourself.
  • All of this is handled by the IoC (Inversion of Control) container, which includes the Dependency Injection mechanism.

Dependency Injection and Loose Coupling in Spring Boot

  • To understand how dependency injection helps achieve loose coupling, let’s take the help of an example.
  • Suppose we have a service that fetches data from a database.
  • Now, our code might need to connect to different databases (e.g., development and production), but we don’t want to write separate applications for each environment.

Problem: Tight Coupling

  • If we hardcode which database to use inside our service, the service becomes tightly coupled to that specific database.
  • For example:
  • If the DBService is directly using DevDB, then switching to ProdDB would require a code change.
  • This is not ideal for real-world environments where we want to deploy the same codebase to different environments without modification.

How Dependency Injection Helps in Loose Coupling

  • Now let’s understand how dependency injection actually helps us in making our system loosely coupled.
  • Define a Specification (Interface)
  • Create a specification, which is just an interface in Java.
  • This interface will define a contract that all database classes must follow.
  • Both DevDB and ProdDB should implement this DB interface.
  • They must provide their own implementation of the getData() method.

Multiple Bean Ambiguity

  • Spring will now try to inject a DB bean.
  • But there are two beans implementing DB:
  • DevDB
  • ProdDB
  • As a result, Spring throws an error:
  • Expected a single matching bean of type 'DB' but found 2: devDB, prodDB
  • above will give error
  • Field db in com.codecrush.ritik.week1Introduction.Introduction.to.Spring.Boot.DBService required a single bean, but 2 were found:
  • Resolve Ambiguity Using @Primary
  • To fix this, we can use the @Primary annotation on one of the beans.

This tells Spring which bean to use by default when there are multiple implementations.

When you run the application:

  • If ProdDB is marked @Primary, it returns "Prod Data".
  • If DevDB is marked @Primary, it returns "Dev Data".

But Still a Problem: Code is Still Tightly Coupled

  • Even though we’re using interfaces and @Primary, we are still editing the code to switch between dev and prod.
  • Changing the primary bean inside the code every time is not true loose coupling.

To make it truly environment-specific without changing code:

  • Use Spring Profiles (@Profile("dev"), @Profile("prod"))
  • Use external configuration (e.g., YAML or application.properties)
  • Use qualifiers (@Qualifier("devDB")) with configuration-driven injection

  • Conditional Configuration Using Properties
  • We define a property in application.properties: deploy.environment=development
  • we use @ConditionalOnProperty in our beans:
@Component
@ConditionalOnProperty(name = "deploy.environment", havingValue = "development")
public class DevDB implements DB { ... }

@Component
@ConditionalOnProperty(name = "deploy.environment", havingValue = "production")
public class ProdDB implements DB { ... }
  • This ensures only one bean is created depending on the environment.
  • This approach achieves loose coupling.
  • It provides flexibility to switch between environments using external configuration.
  • It avoids the need for code changes during deployment.

Changing the Environment

  • To switch environments, change the value in application.properties:
deploy.environment=production
  • Now ProdDB will be injected automatically. No code change is needed — just configuration.

Verifying with Debug Mode

  • To confirm which beans were created, enable debug mode:
debug=true
  • On application startup, Spring will log:
  • Matched Conditions: Beans and configurations that were created.
  • Unmatched Conditions: Beans that were skipped due to configuration mismatch.

Dynamic Configuration Without Code Changes

  • three dot - edit configurations
  • You might wonder: "We are still changing code to switch between development and production?"
  • The key idea is: you can avoid changing code by defining configurations using environment variables.

Environment Variables at the Server Level

  • Each operating system or server can have its own set of environment variables.
  • You can define environment-specific configurations (e.g., DEVELOPMENT, PRODUCTION) at the server level, without altering the code.

Defining Custom Configurations in IDE (e.g., IntelliJ)

  • Click on the Run/Debug Configurations (gear icon next to the Run button).
  • Create or edit your configuration.
  • Specify:
  • The JDK/SDK to use.
  • Any command-line arguments.
  • Environment variables like DEPLOY_ENVIRONMENT=DEVELOPMENT.

Naming Environment Variables Correctly

  • Always use uppercase letters.
  • Replace dots (.) with underscores (_).
  • Example:
  • deploy.do.environment → DEPLOY_DO_ENVIRONMENT
  • Environment Variable Precedence Over application.properties
  • When you run the application, the value in environment variables will override the one in application.properties.
  • Even if application.properties says production, if your env var is DEPLOY_DO_ENVIRONMENT=DEVELOPMENT, the development config will be used.

Benefits of Dependency Injection (DI) in Spring Framework​

  1. Loose Coupling
  • Components are decoupled from their dependencies.
  • This makes the system easier to maintain and test.
  • You can switch databases (e.g., development, test, production) by just changing configuration, not code.
  1. Flexible Configuration
  • Configurations can be defined from multiple external sources.
  • Examples:
  • application.properties or application.yml
  • Environment variables
  • Future topics (like Auto-configuration) will explore more sources.
  • This allows you to easily switch behaviors without touching the source code.
  1. Improved Testability
  • Components can be mocked or replaced during testing.
  • Enables thorough, isolated unit tests.
  • Since components are not tightly coupled, you can test one independently of others.
  1. System Modularity
  • One component is not tightly bound to another.
  • You can plug in any configuration, any service, or even another database, and it will still work as expected.
  • Encourages modular and maintainable architecture.
  1. Built-in Support in Spring (No Extra Code Needed)
  • In contrast to other stacks (like MERN/Node.js), where you need to write additional boilerplate code to achieve loose coupling and DI...
  • Spring Framework provides this out of the box, making it natural and efficient for Java developers.
  1. Scalable and Maintainable Codebase
  • With DI, your application becomes:
  • Scalable – Easy to expand with new components.
  • Maintainable – Clean separation of concerns.
  • Loosely Coupled – Promotes reusable and flexible design.

Dependency Injection in Spring​

Field Injection​

  • Most basic form of dependency injection.
  • Achieved using the @Autowired annotation directly on class fields.
@Autowired
private DB db;
  • Limitation: Cannot mark the field final (hence mutable).
  • Not recommended for core dependencies due to tight coupling and testing limitations.

Constructor Injection​

  • Dependencies are injected via the constructor.
  • Encourages immutability and clear visibility of required dependencies.
@Service
public class DBService {

    private final DB db;

    public DBService(DB db){
        this.db = db;
    }

    String getData(){
        return db.getData();
    }
}

Why Constructor Injection is Preferred:

  • You can mark fields as final, making them immutable.
  • Makes unit testing easier as dependencies are explicitly provided.
  • Helps in clear dependency management – you know what the class needs to function.
  • Supports early failure if a required dependency is missing.

If the dependency is required for the object to work, inject it via the constructor. If it's optional or secondary, then setter or field injection may be acceptable
