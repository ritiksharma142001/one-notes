Spring Boot vs Spring Framework

Background Context:​

  • We have already covered the core concepts of the Spring Framework, including Beans and Dependency Injection.
  • Now, we will explore the key differences between the Spring Framework and Spring Boot.

Spring Boot is Built on Top of Spring Framework​

  • Spring Framework was introduced in 2003 by Rod Johnson.
  • Spring Boot was introduced by VMware in 2014 to simplify the development process.
  • Spring Boot enhances Spring Framework by adding features like auto-configuration and production-ready defaults.

Starter Dependencies​

  • Spring Boot provides starter dependencies, which are not available in the traditional Spring Framework.
  • These dependencies bundle commonly used libraries and configurations.
  • For example:
    • spring-boot-starter-web for web applications
    • spring-boot-starter-data-jpa for database integration
    • spring-boot-starter-test for testing

Auto-Configuration​

  • Spring Boot offers auto-configuration, which automatically configures beans based on the dependencies present in the classpath.

  • This reduces the need for boilerplate configuration code.

  • This feature will be covered in detail in the next lecture, where we’ll understand:

    • What is auto-configuration?
    • How does a Spring Boot application start?
    • What happens behind the scenes when the application runs?

Spring Initializr Integration​

  • Spring Boot supports Spring Initializr, a web-based tool to generate Spring Boot project structures quickly.
  • It allows developers to add multiple dependencies and generate a ready-to-use Maven/Gradle project.
  • These dependencies come pre-configured with sensible defaults to help start development quickly.

Less Boilerplate Code​

  • Spring Boot significantly reduces the amount of manual configuration and boilerplate code.
  • Developers can focus more on business logic than on setup and configuration.
  • Most configurations are handled internally with opinionated defaults.

Adding Dependencies and Classpath Loading​

  • When we add a dependency (e.g., spring-boot-starter-web) to the project using Maven:
  • Maven adds the corresponding jar file to the classpath.
  • Spring Boot then detects this dependency and tries to configure it automatically.

How Auto-Configuration Initializes Beans​

  • Spring Boot’s auto-configuration mechanism initializes these dependencies using pre-defined configurations.
  • These configurations are provided by Spring Boot inside its auto-configuration module.

Exploring Auto-Configuration Source​

  • You can explore the spring-boot-autoconfigure module to see how it works:
  • Inside this module, navigate to the META-INF directory.
  • Locate the file: spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (or spring.factories in older versions).
  • This file contains a list of all auto-configuration classes used by Spring Boot.

What Are Auto-Configuration Classes?​

  • These are standard Java configuration classes annotated with @Configuration.
  • They often use conditional annotations such as:
    • @ConditionalOnClass
    • @ConditionalOnMissingBean
    • @EnableConfigurationProperties
  • These conditions determine when a particular bean should be configured.

Example: Spring Boot Starter Web​

  • If you include the spring-boot-starter-web dependency:
    • It will load configuration classes like WebMvcAutoConfiguration.
    • You can locate this class using IntelliJ by pressing Double Shift and searching for it.
    • The class includes conditions such as:
      • If certain classes are present on the classpath.
      • If some beans are missing.
      • If specific properties are enabled.

Role of Auto-Configuration in Application Context​

  • Once a dependency is detected and conditions are satisfied:
    • Spring Boot registers the necessary beans into the application context.
    • This reduces boilerplate code and simplifies setup for the developer.

Other Examples of Auto-Configuration​

  • The same mechanism applies to other dependencies like:
    • JPA (spring-boot-starter-data-jpa)
    • Security, Actuator, Mail, etc.
  • Each has its own set of auto-configuration classes defined in the same way.

Externalized Configuration Support​

  • Spring Boot supports externalized configurations using multiple property sources.
  • We can define configurations through different files like application.properties or application.yml.
  • These files allow you to configure your application without modifying the source code.
  • Spring Boot supports 14–17 different property sources (e.g., application.properties, environment variables, command-line arguments, etc.).
  • These configurations are loaded into the Spring ApplicationContext at startup.
  • Example:
    • To quickly navigate to a file like application.properties, press Double Shift in IntelliJ and type the filename.

Embedded Server Support​

  • Spring Boot provides embedded servers like Tomcat or Jetty, eliminating the need to set up an external server.
  • By adding spring-boot-starter-web in the pom.xml, Tomcat gets included automatically.
  • You don’t need to configure the server manually—Spring Boot handles it for you.
  • spring-boot-starter-web internally includes:
    • spring-boot-starter
    • Embedded Tomcat
    • Spring MVC
    • Jackson (for JSON support)

Built-in Metrics & Health Checks​

  • Spring Boot provides production-ready features through the Spring Boot Actuator dependency.
  • With spring-boot-starter-actuator, you get:
    • System metrics (e.g., memory, CPU)
    • Health checks for applications and services
    • Readiness and liveness probes for microservices
  • These come pre-configured—no need to manually implement monitoring endpoints.

Dependency Version Management​

  • Spring Boot simplifies dependency management using a parent project: spring-boot-starter-parent.
  • This parent manages versions of all transitive dependencies in a single place.
  • You only need to specify the Spring Boot version once in your pom.xml.
  • Example:
<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.3.0</version>
</parent>
  • All dependencies added to your project will use compatible versions pre-managed by Spring Boot.
  • This avoids compatibility issues among different libraries like:
    • MongoDB
    • Lombok
    • ModelMapper
    • Logging frameworks

Benefits Over Traditional Spring Framework​

  • With Spring Framework:
    • You had to manually configure the server.
    • You were responsible for defining and managing health checks.
    • You had to manually manage dependency versions and compatibility.
  • Spring Boot automates and simplifies all these tasks, allowing faster and more maintainable development.

No Need to Manually Manage Dependency Versions​

  • Spring Boot manages the versions of commonly used dependencies automatically.
  • Example: If you want to add the Lombok dependency, you don’t need to specify its version explicitly.
  • Spring Boot handles compatibility by using a fixed version defined in the spring-boot-starter-parent.

Centralized Dependency Management​

  • All dependencies are version-managed through the parent project (spring-boot-starter-parent).
  • You only need to maintain the Spring Boot version (e.g., 3.3.0) in your pom.xml.
  • This ensures:
    • No version conflicts
    • Full compatibility between dependencies
    • Reduced overhead for developers

Developer Convenience​

  • Spring Boot removes the burden of managing:
    • Configuration files
    • Server setup
    • Dependency versioning
  • Developers can focus solely on writing business logic instead of infrastructure or boilerplate code.

Why Spring Boot Is the Preferred Choice​

  • Due to its simplicity and automation, Spring Boot has become the default choice for creating new microservices.
  • Even companies that used Spring Framework for years now prefer Spring Boot for new development.
  • New microservices are usually started using a Spring Boot Starter Project.

Easily Reusable Configuration​

  • Organizations can reuse existing standard configurations.
  • Developers can simply:
    • Copy the company-wide configuration files
    • Add or modify the application.properties file based on their own service needs
  • The application will automatically pick up the appropriate settings.
