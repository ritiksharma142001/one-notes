Auto-Configuration and Spring Boot Internal flow

pom.xml​

Understanding pom.xml in Spring Boot​

  • The pom.xml (Project Object Model) file is used by Maven, a popular build automation tool.
  • While pom.xml is mainly for managing dependencies, it also:
    • Defines project configurations.
    • Specifies the Java version.
    • Lists starter dependencies.
    • Loads dependencies into the classpath.

What is Maven and Why It's Used​

  • Maven is widely used in Java projects for build and dependency management.
  • Although tools like Gradle exist, most companies still use Maven.
  • Maven reads pom.xml to:
    • Download required libraries.
    • Set up configurations.
    • Manage project structure.

Dependency Resolution via Maven​

  • Maven resolves dependencies by fetching them from the Maven Central Repository.
  • This repository contains thousands of commonly used libraries uploaded by developers over the years.
  • Developers simply declare dependencies in pom.xml, and Maven handles:
    • Downloading the required JARs.
    • Adding them to the classpath.

How Auto-Configuration Works​

  • Once dependencies are on the classpath, Spring Boot's Auto-Configuration system activates.
  • It:
    • Detects which dependencies are present.
    • Loads corresponding auto-configuration classes.
    • Initializes the application with default settings.

Auto-Configuration in Action​

  • Example: When we added spring-boot-starter-web:
    • We didn’t configure the server or MVC manually.
    • Spring Boot auto-configured them based on defaults and classpath presence.
  • Auto-Configuration checks:
    • Which starter dependencies exist.
    • Loads pre-written configuration classes automatically.

Role of Spring Boot Starter Parent​

  • The spring-boot-starter-parent in pom.xml manages:
    • Dependency versions.
    • Plugin versions.
    • General project settings.
  • You don't need to manually define versions for commonly used libraries like:
    • Spring Core
    • Lombok
    • Jackson
    • Logback
  • This ensures compatibility across all dependencies.

Benefits of Auto-Configuration​

  • Reduces manual configuration.
  • Uses sensible defaults provided by Spring Boot.
  • Greatly improves development speed and reduces boilerplate code.
  • Enables developers to focus on business logic rather than setup.

What is Auto Configuration​

What is Auto-Configuration in Spring Boot?​

  • Definition:
    • Auto-configuration is a mechanism in Spring Boot that automatically configures the application based on the dependencies present in the classpath and other application-specific settings.
  • Purpose:
    • It simplifies the setup and development process.
    • Developers can focus more on writing business logic instead of spending time on framework configuration.

Example of Auto-Configuration​

  • Example Scenario:
    • When the Spring Boot Starter Web dependency is added, it automatically:
      • Starts an embedded Tomcat server.
      • onfigures the dispatcher servlet.
      • Sets up default server ports, request handling, etc.
  • Manual Configuration Not Required:
    • We don’t need to explicitly configure:
      • The server type
      • Port
      • Dispatcher servlet
      • Default behaviors
    • All these are automatically managed by auto-configuration.

How It Works Internally​

  • Auto-configuration Class:
    • Each starter dependency includes an auto-configuration class.
    • This class contains predefined default configurations provided by Spring Boot developers.
  • Accessing Auto-configuration Classes:
    • You can view these in the External Libraries section of your IDE.
    • The spring-boot-autoconfigure module contains a list of all available auto-configuration classes.
  • Current Count:
    • As of now, there are around 152 auto-configuration classes (previously ~130).

Types of Auto-Configurations Available​

  • Examples of Auto-Configuration Modules:
    • Web:
      • WebMvcAutoConfiguration
      • WebClientAutoConfiguration
      • WebFluxAutoConfiguration
    • JPA (Java Persistence API):
      • Automatically configures JPA-related beans.
      • Includes support for Hibernate, EntityManagerFactory, DataSource, etc.

Conditional Configuration Logic​

  • Conditional Loading:
    • Spring Boot uses conditions to decide whether to load a particular bean:
      • Bean must be present or absent.
      • Specific class must be available in the classpath.
      • Certain Spring properties must be enabled, e.g.:
        • spring.data.jpa.repositories.enabled=true
  • If All Conditions Are Met:
    • The specific configuration is applied, and the corresponding beans are loaded into the application context.

How Autoconfiguration Works​

Classpath Scanning Phase​

  • Spring Boot starts by scanning the classpath.
  • It looks for the main annotation: @SpringBootApplication, which serves as the entry point of any Spring Boot application.
  • Every Spring Boot application must have only one @SpringBootApplication annotation.

Annotations Involved in Auto-Configuration​

The @SpringBootApplication annotation is a composite annotation, which includes:

  • @EnableAutoConfiguration

    • This triggers the auto-configuration process.
    • Spring Boot checks which dependencies are present in the classpath and configures beans accordingly.

  • @ComponentScan

  • This starts the component scanning process.

  • It begins scanning from the package where @SpringBootApplication is located.

  • All sub-packages and classes under that root package are scanned for Spring components.

Component Scanning Process​

  • The component scanner searches for Spring annotations such as:
    • @Service
    • @Repository
    • @Controller
    • @Component
  • For example:
    • If it finds a class like DbService annotated with @Service, it registers it as a Spring Bean.
    • It also reads any defined properties or dependencies inside those classes.

Dependency Scanning from Classpath​

  • Apart from annotations, Spring Boot also scans for dependencies added through pom.xml.
  • These dependencies are loaded onto the classpath.

Types of Scanning​

There are two main types of scanning:

  • Annotation-based Scanning
    • Uses annotations like @ComponentScan, @Service, etc.
    • Scans user-defined packages for Spring-managed components.
  • Dependency-based (Classpath) Scanning
    • Scans dependencies listed in pom.xml.
    • Once a dependency is detected, Spring Boot loads the related auto-configuration classes.

Auto-Configuration Loading Process​

  • When a dependency is found in the classpath:
    • Spring Boot checks if there's a matching auto-configuration class.
    • If so, it evaluates conditions (like presence or absence of beans, classes, or properties).
    • If the conditions are satisfied, the configuration is loaded into the application context

Scanning Configuration Classes​

  • After scanning the classpath and components, Spring Boot looks for configuration classes related to each dependency.
  • For example, if a dependency is present (e.g., DataSource or JPA), its corresponding auto-configuration class is also scanned.

Where Are Auto-Configurations Defined?​

  • All Spring Boot projects include the package:
spring-boot-autoconfigure-[version]

Example: spring-boot-autoconfigure-3.3.2

  • Inside this package:
    • Navigate to META-INF/spring/
    • Open the file: spring.factories or spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (depending on Spring version)
    • This file lists all available auto-configuration classes.
  • This file currently includes around 152 auto-configuration entries, and more are added over time.

What Are Auto-Configuration Classes?​

  • These classes define default Spring bean configurations for common dependencies.
  • When a relevant dependency is present in the classpath, Spring Boot:
    • Loads the matching auto-configuration class
    • Uses conditional annotations to determine whether to create a bean or not

Conditional Annotations Used​

Auto-configuration classes use conditional logic such as:

  • @ConditionalOnClass: Loads the configuration only if a specific class is present.
  • @ConditionalOnMissingBean: Loads the bean only if it's not already defined by the user.
  • @EnableConfigurationProperties: Enables loading of configuration properties.

Example:

  • In DataSourceAutoConfiguration:
    • If JDBC classes are present and a DataSource bean is missing, Spring Boot auto-creates one.
  • In HibernateJpaAutoConfiguration:
    • JPA-related beans are conditionally loaded if relevant classes and properties exist.

How Spring Boot Decides What to Configure​

  • Spring Boot checks the classpath for:
    • Libraries added via pom.xml (e.g., Spring Data JPA, Web, Security)
    • Then, it refers to matching auto-configuration classes from the import file
  • Using conditional annotations, it decides which beans to create and register in the application context.

Core Features of AutoConfiguration​

What is Property Sources Auto-Registration?​

  • Spring Boot automatically registers configuration properties from multiple sources.
  • These are referred to as Property Sources.
  • Auto-configuration collects and uses these property sources to configure the application at runtime.

Common Property Sources in Spring Boot​

Spring Boot supports 17 property sources, including (but not limited to):

  • Command-line arguments
  • Java system properties (System.getProperties())
  • OS environment variables
  • application.properties or application.yml files
  • application-{profile}.properties (based on active profile)
  • Properties loaded via @PropertySource
  • Properties from SpringApplication.setDefaultProperties
  • Link

Examples of Property Sources​

These are ordered by precedence (higher sources override lower ones):

  • Command-line arguments
  • Java system properties
  • application.properties / application.yml -- Environment variables
  • Default properties
  • Example:
    • If the same property is defined in both application.properties and OS environment variables, the OS environment variable will override the value from the properties file.

Practical Recommendation​

  • As a developer, you don’t need to remember all 17 sources.
  • In most real-world or industry applications, the following are sufficient:
    • application.properties or application.yml
    • OS environment variables
    • Command-line arguments (used for overriding at runtime)

Why This Matters​

  • Spring Boot loads and applies configuration from multiple property sources.
  • Understanding this helps in:
    • Debugging configuration issues
    • Overriding values properly
    • Customizing application behavior for different environments

Spring Boot Auto-Configuration as a JAR​

  • Spring Boot auto-configuration is packaged as a simple JAR file.
  • This JAR contains most of Spring Boot's magic (i.e., internal logic for automatic configuration).
  • It includes many predefined configuration classes that Spring Boot automatically loads under certain conditions.

Auto-Configuration Uses Conditional Logic​

  • Each auto-configuration class is loaded conditionally, based on the application's current state or environment.
  • These conditions are defined using Spring's @Conditional annotations.

Types of Conditional Annotations in Spring Boot​

Spring Boot provides enhanced conditional support, including:

  • @ConditionalOnBean
    • Loads the bean only if another specific bean is present in the application context.
  • @ConditionalOnMissingBean
    • Loads the bean only if a specific bean is not present in the context.
  • @ConditionalOnClass
    • Loads configuration only if a specific class is present in the classpath.
  • @ConditionalOnMissingClass
    • Loads configuration only if a specific class is not present in the classpath.
  • @ConditionalOnProperty
    • Loads configuration only if a specific property is defined and meets a condition (e.g., equals "true").

Example: @ConditionalOnProperty​

  • In earlier examples, we used @ConditionalOnProperty for a development-specific configuration (e.g., a Dev DB).
  • This configuration was activated only if a property like app.db.dev=true was set in the environment or application properties.
  • If the condition is not satisfied, the corresponding bean will not be added to the context.

Behind the Scenes​

  • All auto-configuration classes include such predefined conditional logic.
  • These conditionals are written by Spring Boot developers to ensure that beans load only when appropriate.
  • Spring Boot checks all these conditions before registering a bean into the ApplicationContext.

Why Conditional Configuration Matters​

  • It provides flexibility and modularity, allowing configurations to adapt automatically based on:
    • Active environment
    • Available classes
    • Set properties
    • Already registered beans
  • It reduces the need for boilerplate configuration and improves application maintainability.

Spring Boot = Spring Framework + Auto-Configuration​

  • Spring Boot is not magic.
  • It is built on top of the Spring Framework and adds auto-configuration capabilities.
  • It simplifies setup by automatically creating beans if certain conditions are met.

What is Auto-Configuration?​

  • Auto-configuration is just a collection of Java configuration classes that:
    • Define beans
    • Are loaded only when specific conditions (defined using annotations like @ConditionalOn...) are satisfied
  • These classes are packaged into a JAR file included in your application.

Spring Boot Does Not Do Anything “Fancy”​

  • There is no hidden magic.
  • It simply checks for:
    • Classes in the classpath
    • Property values
    • Beans already defined in the context
  • Based on these checks, it conditionally loads the appropriate configurations and registers beans

How Spring Boot Determines What to Load​

  • It scans the classpath and property sources.
  • Based on predefined conditions, it registers beans into the application context.
  • This allows Spring Boot to adapt the configuration dynamically, depending on the environment and dependencies.

Spring Boot Offers Ready-to-Use Features​

Because of this mechanism, developers get many features "out of the box", such as:

  • Database configuration
  • Embedded server setup (like Tomcat)
  • Spring Actuator
  • Spring Security
  • And many other features with minimal or no setup

Spring Boot Internal Flow​

Let's look at how Spring Boot works internally, starting from the moment you click the “Run” button.

Step 1: Initialization Begins​

  • When a Spring Boot application is launched, the initialization phase begins.

Main Class Execution

  • The application starts from the main class, which is typically annotated with @SpringBootApplication.
  • This annotation is a composite annotation that bundles three important annotations:
    • @SpringBootConfiguration
    • @EnableAutoConfiguration
    • @ComponentScan

Role of @SpringBootApplication

  • This single annotation initiates the entire Spring Boot setup.
  • It acts as the entry point of the Spring Boot application.

@SpringBootConfiguration

  • Marks the class as a configuration class.
  • Equivalent to using @Configuration, meaning this class can define beans using @Bean methods.

@EnableAutoConfiguration

  • Triggers auto-configuration by scanning the classpath and activating configurations based on available libraries and defined properties.
  • It loads beans conditionally, based on the environment.

@ComponentScan

  • Enables component scanning, which means:
    • Spring will scan the package (where your main class resides) and its sub-packages.
    • It automatically detects and registers components like @Service, @Repository, @Component, and @Controller.

Step 2: Spring Application Context Creation​

1. What is the Application Context?

  • The Spring Application Context is the core container of a Spring Boot application.
  • It manages all the beans, their lifecycle, and how they are interconnected.
  • Since a Spring application is entirely built using beans, the context holds all the bean-related metadata.

Purpose of the Application Context

  • Acts as a central registry that:
    • Stores information about all beans.
    • Maintains relationships and dependencies between beans.
    • Provides bean creation, injection, and management services.

How Spring Boot Creates the Context

  • Spring Boot initializes the Application Context during startup.
  • It:
    • Scans the classpath for components, configurations, and auto-configuration classes.
    • Processes annotations like @Component, @Configuration, and @EnableAutoConfiguration.
    • Registers and manages all detected beans.

Conditional Bean Initialization

  • Spring Boot uses conditional annotations to decide which beans to load:
    • @ConditionalOnClass – Loads the bean if a specific class is present in the classpath.
    • @ConditionalOnBean – Loads the bean only if another specific bean is already present.
    • @ConditionalOnProperty – Loads the bean based on specific property values.
  • These conditions are evaluated before bean initialization.

Step 3: Auto-Configuration​

Role of Auto-Configuration

  • After creating the Application Context:
    • Auto-configuration begins.
    • Spring Boot automatically configures beans based on the environment, dependencies, and conditions.
  • If a condition does not match, the corresponding bean is not loaded into the context.

Step 4: Loading Externalized Configurations​

Purpose of Externalized Configuration

  • Spring Boot supports externalized configuration, allowing developers to configure applications without changing the code.
  • These configurations are used to override default values and customize behavior at runtime.

Sources of Configuration Spring Boot loads configuration properties from multiple sources, including:

  • application.properties or application.yml files.
  • Environment variables.
  • Command-line arguments.
  • System properties.
  • JNDI attributes.
  • Servlet context and initialization parameters.
  • Random values.
  • application-{profile}.properties for profile-specific configuration.
  • And up to 15 predefined property sources (as documented on Spring’s official site).

File Format Options

  • You can use either .properties or .yml/.yaml files.
    • Functionally similar, only the syntax differs.
    • YAML files offer better hierarchical structure.

How Spring Boot Uses These Configurations

  • Spring Boot loads all property sources and merges them into the environment.
  • It provides sensible default values for most properties.
  • These default values can be overridden via external configuration.

Example of Overriding Defaults

  • Example: By default, Spring Boot uses port 8080, but you can override it:
server.port=9000
  • Example: If you add the spring-boot-starter-data-jpa and H2 dependencies, it will:
    • Automatically configure an in-memory H2 database.
    • Create a default JDBC connection.

Customizing Default Behavior

  • You can customize default settings by defining properties:
    • Change server port, database URLs, logging levels, etc.
  • Spring Boot will respect your custom configurations over default ones.
    • For example:
server.port=9000
spring.datasource.url=jdbc:h2:mem:mydb

Step 5: Embedded Web Server Initialization​

Embedded Web Server Initialization

  • After loading configurations, Spring Boot initializes an embedded web server.
  • Common servers supported:
    • Tomcat (default with spring-boot-starter-web)
    • Jetty
    • Undertow
  • Since we are using the Spring Boot Web dependency, the Tomcat server is initialized by default.

Step 6: Appication Startup​

Application Context Fully Starts

  • Once the server is initialized, Spring Boot completes the application context initialization.

Lifecycle Callback Execution

  • Spring Boot then executes lifecycle callback methods such as:
    • @PostConstruct
    • Initialization methods defined in the beans.
  • These callbacks run after the beans are created and dependencies injected, but before the application starts handling requests.

Order of Bean Initialization

  • Spring Boot analyzes the dependencies between beans and ensures proper initialization order.
  • Internally, it likely uses a form of topological sorting to:
    • Detect dependency chains.
    • Call @PostConstruct methods in the correct order.

Step 7: Appication Ready​

Application Context is Fully Initialized

  • Once the initialization process completes, the Spring ApplicationContext is:
    • Fully loaded.
    • All beans are created and wired.
    • All lifecycle methods (@PostConstruct, init methods) are executed.

Embedded Web Server is Running

  • The embedded web server (e.g., Tomcat) is now fully up and running.
  • The application is ready to receive and handle incoming HTTP requests.

Application is Ready

  • At this stage, your Spring Boot application is completely ready to:
    • Serve requests.
    • Perform any tasks or operations defined in the code.

Summary of What Happened Behind the Scenes

  • When you click the "Run" button:
    • Multiple steps are triggered automatically.
    • Many dependencies are injected.
    • Beans are created and wired.
    • PostConstruct methods are called.
    • Auto-configurations are applied.
    • The application is then ready to start functioning.

Spring Boot vs Traditional Spring

  • In Spring Boot, much of the heavy lifting is automated:
    • Auto-configuration for web servers, databases, etc.
    • No need for complex manual setup.
  • In traditional Spring Framework:
    • You must manually configure the server, databases, and other components.
