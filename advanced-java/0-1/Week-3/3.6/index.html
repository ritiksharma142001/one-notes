Mapping-2

Mapping Types between Two Entities‚Äã

  • We have already seen a One-to-One mapping between Department and Employee.
  • Now, we‚Äôll demonstrate Many-to-One mapping between the same entities.
  • No need to create new entities ‚Äî we'll reuse Department and Employee.

Real-World Scenario‚Äã

  • A Department:
    • Has one manager (One-to-One with Employee)
    • Has many workers (One-to-Many with Employee)
  • An Employee:
    • Works in only one department (Many-to-One)
  • So:
    • From Department ‚Üí Employee: One-to-Many
    • From Employee ‚Üí Department: Many-to-One

Database Consideration‚Äã

  • In SQL:
    • A Department table cannot store a list of employees.
    • But an Employee table can store a foreign key (department_id).
  • Therefore:
    • The foreign key should be placed in the Employee entity/table.
    • The Department table won‚Äôt know who its workers are directly.

Using Spring Data JPA‚Äã

  • With Spring Data JPA and Hibernate:
    • We can still get a list of employees in a department easily.
    • Even if the Department entity does not store them directly.
    • Hibernate handles the retrieval of related records efficiently.

Implementation Steps‚Äã

  • In Employee entity:
    • Add a field to store the department it works in, e.g., workerDepartment.
    • Annotate it as a @ManyToOne relationship.
  • Explanation:
    • @JoinColumn defines the foreign key column:
      • name: column name in the Employee table (e.g., worker_department_id)
      • referencedColumnName: which column in Department it maps to (typically id)
    • You can omit referencedColumnName if it's the default (id), but specifying it is safer when names differ.

Database Output After Mapping‚Äã

  • After applying the mapping and running the app:
    • Employee table now contains:
      • id (Primary key)
      • name
      • worker_department_id (Foreign key to Department)
    • A foreign key constraint is added from worker_department_id to Department.id.

Problem Summary‚Äã

  • You have two entities: Department and Employee.
  • An employee belongs to a department, and a department can have many employees ‚Üí this is called a bidirectional one-to-many and many-to-one mapping.
  • But initially, data was not syncing correctly between both sides of the relationship. One side worked (e.g., Employee ‚Üí Department), but the other (Department ‚Üí List of Employees) did not work properly.

What was wrong?‚Äã

1. Lombok's @Data annotation‚Äã

  • @Data automatically generates hashCode() and equals() using all fields, including the entity references (like Department having List<Employee> and Employee having Department).
  • This caused:
    • Infinite recursion
    • StackOverflowError
    • Unexpected behavior in bidirectional mapping

What was done to fix it?‚Äã

1. Removed Lombok‚Äôs @Data‚Äã

  • Instead, only used:
    • @Getter
    • @Setter
  • Manually implemented equals() and hashCode().

2. Wrote custom equals() and hashCode()‚Äã

  • Only used non-entity fields, like:
    • id
    • name
  • Excluded fields like List<Employee> or Department, to avoid recursion or performance issues.

3. Added @JsonIgnore‚Äã

  • In REST APIs, when Spring Boot serializes objects to JSON, bidirectional references can cause infinite loops.
  • So, he added @JsonIgnore on one side:
    • Either on Employee.department or Department.employees
  • He chose to ignore it on Employee.department, so:
    • Department will show the list of employees.
    • Employee won‚Äôt directly show the department in JSON (can be fetched using a separate API call).

Testing the Fix‚Äã

  • Created a new Department and Employees.
  • Assigned Employees to the Department.
  • Fetched:
    • Department ‚Üí Got list of employees ‚úÖ
    • Employee ‚Üí Department is hidden (because of @JsonIgnore) ‚úÖ
  • No recursion, no crash, working as expected.

CascadeType.ALL‚Äã

  • He mentioned using cascade = CascadeType.ALL.
  • This means when you save or delete a Department, it automatically saves or deletes all its employees.

First, what is @Data in Lombok?‚Äã

When you write:

@Data
public class Employee {
    private Long id;
    private String name;
    private Department department;
}

Lombok automatically adds:

  • Getters & Setters
  • toString()
  • equals() and hashCode() ‚ö†Ô∏è Important: It includes all fields, including department.

What is bidirectional mapping?‚Äã

Example:

class Department {
    List<Employee> employees; // one department has many employees
}

class Employee {
    Department department; // each employee belongs to one department
}

So:

  • Department ‚Üí List<Employee> (One-to-Many)
  • Employee ‚Üí Department (Many-to-One) This is called bidirectional mapping ‚Äî both sides "point" to each other.

What problem does @Data cause?‚Äã

Let‚Äôs say:

  • Employee e has a Department d
  • Department d has a list with Employee e in it Now, if Lombok includes department in equals() and hashCode() of Employee, and employees in the same methods of Department, we get a loop:
  • To compare two Employee, it compares their Department
  • To compare two Department, it compares the list of Employee
  • Again, each Employee has Department... and so on

üîÅ This never ends, causing:

  • Infinite recursion
  • StackOverflowError

Understanding CascadeType‚Äã

  • The cascade attribute in JPA helps propagate operations (save, delete, etc.) from one entity to its related entities.
  • For example:
    • If you save an Employee, the associated Department is also saved if cascade is enabled.
    • If you delete an Employee, the reference can also be removed from the Department, based on the cascade strategy.
  • Common Cascade Types:
    • PERSIST: Propagates only save (insert) operations.
    • MERGE: Propagates update operations.
    • REMOVE: Propagates delete operations.
    • REFRESH: Refreshes data from DB.
    • DETACH: Detaches the entity from persistence context.
    • ALL: Combines all the above.

Understanding FetchType‚Äã

  • Defines when related entities are loaded:
    • EAGER (default for many-to-one): Loads related entities immediately.
    • LAZY (default for one-to-many): Loads related entities only when accessed.
  • Example:
    • If FetchType.LAZY is set on the Department‚Äôs workers list:
      • Workers are not fetched immediately.
      • When .getWorkers() is called, Hibernate fetches them from the DB.

Note About Jackson and Lazy Fetching‚Äã

  • Even with LAZY fetch type, if you're returning the entity via a REST API:
    • Jackson may call the getter methods (like .getWorkers()).
    • This triggers Hibernate to fetch data lazily ‚Äî so it may seem like it is EAGER.
  • Best Practice:
    • Use LAZY for better performance.
    • Use DTOs (Data Transfer Objects) to control what is returned in APIs.

Many-to-Many Mapping Preview (Freelancers)‚Äã

  • Scenario:
    • Some Employee instances are Freelancers.
    • Freelancers can work in multiple Departments.
    • A Department can have multiple Freelancers.
  • This is a perfect example of a Many-to-Many relationship.

Strategy for Many-to-Many Mapping‚Äã

  • For such cases, a join table is used (not @JoinColumn, but @JoinTable).
  • Ownership doesn't matter much here because:
    • Both sides (Employee and Department) are many-to-many.
  • Still, you need to choose one side to define the @JoinTable.

Many-to-Many Mapping: Freelancers and Departments‚Äã

üß© Step 1: Define Many-to-Many Mapping in Employee Entity‚Äã

  • We added a Set<Department> to the Employee entity to represent the departments a freelancer is associated with.
  • This field is named freelanceDepartments.
  • Annotated it with @ManyToMany and @JoinTable.

üõ†Ô∏è Step 2: Configure @JoinTable‚Äã

  • Created a join table named freelance_department_mapping.
  • Used @JoinColumn(name = "employee_id") for the Employee side.
  • Used @InverseJoinColumn(name = "department_id") for the Department side.
  • This join table maintains the relationship between freelancers and departments.

üß© Step 3: Make Mapping Bi-Directional (Optional)‚Äã

  • Added a Set<Employee> named freelancers in the Department entity.
  • Annotated with @ManyToMany(mappedBy = "freelanceDepartments").
  • This makes the mapping two-way.

üöÄ Step 4: Creating and Linking Data‚Äã

  • Created a new department and some employee records.
  • Used a controller method assignFreelancerToDepartment(deptId, empId) to add the relationship.
  • Initially, used departmentRepository.save() to persist, but ran into issues.

üõ†Ô∏è Step 5: Fixing the Save Strategy‚Äã

  • Since the owning side of the join table is the Employee entity:
    • Switched the save() call from Department to Employee.
    • This ensures the join table is updated correctly.

üîÅ Step 6: Preventing Infinite Recursion‚Äã

  • Used @JsonIgnore on the mapped fields to avoid recursive JSON serialization.

Many-to-Many Mapping and Advanced JPA Use Cases‚Äã

üß© Adding Freelancers to Departments‚Äã

  • Created additional employees (e.g., "Xiaomi", "Shyam").
  • Added multiple employees to a department as freelancers.
  • Created another department, Engineering, and assigned freelancers to it.
  • Demonstrated that:
    • An employee can belong to multiple departments.
    • A department can have multiple freelancers.

Verifying Data‚Äã

  • Used getDepartmentById to confirm assigned freelancers.
  • Verified that employee Anu is part of both "HR" and "Engineering" departments.
  • Checked database entries:
    • Department ID 1 ‚Üí Employees 1 & 2
    • Department ID 2 ‚Üí Employees 1 & 3

Optional: Join Table in One-to-Many Mapping‚Äã

  • In the existing setup, Employee stores workerDepartmentId as a foreign key.
  • Alternative approach: Use a join table instead of a join column for one-to-many.
  • Benefits:
    • Keeps the Employee entity clean.
    • Suitable when you want to separate relational data in a dedicated table.

üîß Implementation:‚Äã

  • Replaced @JoinColumn with @JoinTable.
  • Defined a new table: worker_department_mapping.
  • Result:
    • The Employee table no longer contains workerDepartmentId.
    • A separate table stores the mapping between Employee ID and Department ID.

package com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.entities;

import com.fasterxml.jackson.annotation.JsonIgnore;
import jakarta.persistence.*;
import lombok.*;

import java.util.Objects;
import java.util.Set;

@Entity
@Getter
@Setter
@AllArgsConstructor
@NoArgsConstructor
@Builder
@Table(name = "employees")
public class EmployeeEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String name;

    @OneToOne(mappedBy = "manager")
    @JsonIgnore
    private DepartmentEntity managedDepartment;

    @ManyToOne(cascade = CascadeType.ALL)
//    @JoinColumn(name = "worker_department_id", referencedColumnName = "id")
    @JoinTable(name = "worker_department_id")
    @JsonIgnore
    private DepartmentEntity workerDepartment;

    @ManyToMany
    @JoinTable(name = "freelance_department_mapping",
            joinColumns = @JoinColumn(name = "employee_id"),
            inverseJoinColumns = @JoinColumn(name = "department_id")
    )
    @JsonIgnore
    private Set<DepartmentEntity> freelanceDepartments;

    @Override
    public boolean equals(Object o) {
        if (o == null || getClass() != o.getClass()) return false;
        EmployeeEntity that = (EmployeeEntity) o;
        return Objects.equals(getId(), that.getId()) && Objects.equals(getName(), that.getName());
    }

    @Override
    public int hashCode() {
        return Objects.hash(getId(), getName());
    }
}


package com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.entities;

import com.fasterxml.jackson.annotation.JsonIgnore;
import jakarta.persistence.*;
import lombok.*;

import java.util.List;
import java.util.Objects;
import java.util.Set;

@Entity
@Setter
@Getter
@AllArgsConstructor
@NoArgsConstructor
@Table(name = "departments")
public class DepartmentEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String title;

    @OneToOne
    @JoinColumn(name = "department_manager")
    private EmployeeEntity manager;

    @OneToMany(mappedBy = "workerDepartment")
    private Set<EmployeeEntity> workers;

    @ManyToMany(mappedBy = "freelanceDepartments")
    private Set<EmployeeEntity> freelancers;

    @Override
    public boolean equals(Object o) {
        if (o == null || getClass() != o.getClass()) return false;
        DepartmentEntity that = (DepartmentEntity) o;
        return Objects.equals(getId(), that.getId()) && Objects.equals(getTitle(), that.getTitle());
    }

    @Override
    public int hashCode() {
        return Objects.hash(getId(), getTitle());
    }
}

package com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.controllers;

import com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.entities.DepartmentEntity;
import com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.entities.EmployeeEntity;
import com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.services.DepartmentService;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping(path = "/departments")
public class DepartmentController {

    private final DepartmentService departmentService;

    public DepartmentController(DepartmentService departmentService) {
        this.departmentService = departmentService;
    }

    @GetMapping("/{departmentId}")
    public DepartmentEntity getDepartmentById(@PathVariable Long departmentId) {
        return departmentService.getDepartmentById(departmentId);
    }

    @PostMapping
    public DepartmentEntity createNewDepartment(@RequestBody DepartmentEntity departmentEntity) {
        return departmentService.createNewDepartment(departmentEntity);
    }

    @PutMapping(path = "/{departmentId}/manager/{employeeId}")
    public DepartmentEntity assignManagerToDepartment(@PathVariable Long departmentId,
                                                      @PathVariable Long employeeId) {
        return departmentService.assignManagerToDepartment(departmentId, employeeId);
    }

    @GetMapping(path = "/assignedDepartmentOfManager/{employeeId}")
    public DepartmentEntity getAssignedDepartmentOfManager( @PathVariable Long employeeId) {
        return departmentService.getAssignedDepartmentOfManager(employeeId);
    }

    @PutMapping(path = "/{departmentId}/worker/{employeeId}")
    public DepartmentEntity assignWorkerToDepartment(@PathVariable Long departmentId,
                                                     @PathVariable Long employeeId) {
        return departmentService.assignWorkerToDepartment(departmentId, employeeId);
    }

    @PutMapping(path = "/{departmentId}/freelancers/{employeeId}")
    public DepartmentEntity assignFreelancerToDepartment(@PathVariable Long departmentId,
                                                         @PathVariable Long employeeId) {
        return departmentService.assignFreelancerToDepartment(departmentId, employeeId);
    }
}
