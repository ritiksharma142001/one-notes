Mapping-1

Relationships in Spring Data JPA​

Introduction to Relationships in Databases​

  • In real-world systems like a college database:
    • A college can have multiple classrooms.
    • A professor can teach multiple students.
    • A student can enroll in multiple courses, and a course can have multiple students.
  • These examples demonstrate relationships between entities.

Entity Relationship Example: Department, Employee, and Project​

  • Entities: Department, Employee, Project
  • Relationships:
    • One-to-One:
      • A Department can have one manager, who is an Employee.
    • One-to-Many / Many-to-One:
      • A Department can have many workers (Employees).
      • Each Employee works in one department.
    • Many-to-Many:
      • An Employee can work on multiple projects.
      • A Project can have multiple employees working on it.

Relationship Types in JPA​

  • @OneToOne – One entity has exactly one related entity.

  • @OneToMany – One entity is related to many others.

  • @ManyToOne – Many entities are related to one.

  • @ManyToMany – Many entities are related to many others.

  • Example:

    • Department ↔ Manager → @OneToOne
    • Department ↔ Employees → @OneToMany
    • Employee ↔ Department → @ManyToOne
    • Employee ↔ Project → @ManyToMany

Bidirectional vs. Unidirectional Mapping​

  • JPA allows relationships to be mapped in both directions:
    • Example: From Employee to Department and vice versa.
  • Relationships can be:
    • Unidirectional – Only one side knows about the other.
    • Bidirectional – Both sides reference each other using mapped entities.

SQL vs JPA Mapping​

  • In SQL:

    • Relationships are handled using foreign keys and constraints.

  • In Spring Data JPA:

    • Relationships are mapped using annotations (@OneToOne, @ManyToOne, etc.).
    • SQL schema is generated automatically by Hibernate.

  • Hibernate handles table creation, foreign keys, and join tables internally.

Focus of This Course​

  • The course emphasizes Spring Data JPA, not raw SQL.
  • Though SQL knowledge is helpful, it's not mandatory.
  • Entity mapping and relationship creation will be demonstrated using JPA annotations, not manual SQL.

Clarification on Entities vs Tables​

  • The focus is on Java Entities, not direct SQL tables.
  • While SQL has tables, in JPA we model entities, and Hibernate manages table creation automatically.
  • Example: An Employee entity in Java corresponds to an employee table in SQL, created by Hibernate.

Overview of the Project Structure​

  • We are creating a project that demonstrates JPA mappings.
  • The project will include three entities:
    • Department
    • Employee
    • Project
  • Currently, only the Department and Employee entities are created. The Project entity will be added in a future lecture.

Goal of This Session: One-to-One Mapping​

  • We aim to demonstrate One-to-One Mapping between Department and Employee:
    • Each Department has one Manager.
    • One Employee can be a Manager for at most one Department.
  • Additional details:
    • Every Department must have a Manager.
    • Not all Employees are Managers.

Key JPA Mapping Concepts​

1. @Entity

  • A class annotated with @Entity represents a table in the database.

2. @Id

  • The @Id annotation marks the primary key of the table.
  • It guarantees uniqueness — two records cannot have the same ID.

3. Foreign Key

  • A foreign key is a field in one table that refers to the primary key of another table.
  • Example:
    • department table may have a manager_id column.
    • This manager_id references the id field of the employee table.

Database Auto-Creation and Setup​

  • We are using spring.jpa.hibernate.ddl-auto=create in application.properties.
    • This setting drops the existing schema and recreates it on application startup.
  • We created a new database named mappings in MySQL.
  • How to create a new database in tools like IntelliJ/Datagrip:
    • Right-click on Databases
    • Choose New > Database
    • Enter database name (e.g., mappings) and default options

Tables Created Automatically​

  • After application startup, the following tables are created:

    • departments
    • employees
    • department_sequence (auto-generated)
    • employee_sequence (auto-generated)

  • The sequence tables are used to generate ID values using @GeneratedValue(strategy = GenerationType.SEQUENCE).

  • These sequence tables:

    • Only have a column like next_val which stores the next ID value.
    • Are not always needed, especially in MySQL.

Avoiding Extra Sequence Tables in MySQL​

  • The creation of sequence tables occurs when using @GeneratedValue(strategy = GenerationType.SEQUENCE).
  • To avoid this in MySQL (which does not support sequences natively):
    • Use @GeneratedValue(strategy = GenerationType.IDENTITY)
  • IDENTITY strategy allows MySQL to auto-increment the ID without extra tables.
  • How to Fix:
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
  • After changing to IDENTITY, the sequence tables are no longer needed.
  • You can manually delete the old sequence tables if required.
  • Never delete tables in production manually. Doing so can lead to data loss. Always use migrations or schema management tools.

Current State of Tables​

  • After cleanup and re-run:
    • Only two tables exist: departments, employees
    • Each table currently has only:
      • id
      • title or name
    • No foreign key or relationship defined yet.

Key Concepts Before Defining Mappings​

1. Foreign Key

  • A foreign key in one table points to a primary key in another table.
  • Example:
    • Department has a manager_id (foreign key)
    • Employee has id (primary key)

2. Cascade Types

  • Controls what happens to related entities when an operation is performed on the owning entity.
  • For example:
    • If a Department is deleted, should its Manager (Employee) also be deleted?
  • Common cascade types:
    • CascadeType.ALL, PERSIST, MERGE, REMOVE, REFRESH, DETACH

3. Fetch Types

  • Controls when related data is fetched from the database.
    • FetchType.EAGER: Loads related entity immediately.
    • FetchType.LAZY: Loads related entity only when accessed.
  • Example:
    • When fetching a Department:
    • EAGER fetch: Manager is fetched too.
    • LAZY fetch: Manager is fetched only when getManager() is called.

Defining One-to-One Mapping Between Department and Manager (Employee)​

Deciding Ownership of the Relationship​

  • Each Department has one Manager (an Employee).
  • Decision: Where should we define the relationship?
    • Option 1: Define department in the Employee entity.
    • Option 2: Define employee (manager) in the Department entity.
  • ✅ Chosen: Define the manager in the Department entity.
    • Reason: Not all employees are managers, so keeping a nullable department field in Employee is not meaningful.
    • But every department must have a manager, hence it makes logical sense to give ownership of the mapping to the Department.

Creating the One-to-One Relationship​

  • Inside the Department entity:
@OneToOne
private Employee manager;
  • manager is directly linked to the Employee entity.
  • Without any annotations, the system throws an error:
    • "Entity attribute is not marked with an association annotation"
    • So, we annotate with @OneToOne to indicate the relationship.

Customizing the One-to-One Mapping​

  • @OneToOne allows configuration:
  • CascadeType: To propagate operations (e.g., persist, delete).
  • FetchType:
    • EAGER (default): Loads manager immediately when department is fetched.
    • LAZY: Loads manager only when explicitly accessed.
  • mappedBy: Used for bidirectional relationships (not applied yet here).

Running and Inspecting Database​

  • After running the application:
    • A foreign key column manager_id is added to the departments table.
    • By default, the column name is inferred as manager_id.
    • A unique constraint is automatically added:
      • Ensures one-to-one relationship (i.e., one manager cannot manage multiple departments).

Understanding the Unique Constraint​

  • The unique constraint on manager_id ensures:
    • One manager → One department
    • Prevents many-to-one behavior (i.e., multiple departments with same manager)

Customizing Column Name Using @JoinColumn​

  • To rename the foreign key column:
@OneToOne
@JoinColumn(name = "department_manager")
private Employee manager;
  • Now, the column in departments table will be named department_manager.

Final Table Structure​

  • After refresh, departments table contains:
    • id
    • title
    • department_manager (foreign key to employees table)

Current Mapping Direction​

  • The mapping is currently unidirectional:

    • Department knows who the manager is.
    • But the Employee entity does not know which department they are managing.

  • 🔍 Example:

    • Multiple employees exist, but none of them are aware that they are managing a department.
    • Meanwhile, each department clearly knows its manager.

Implementing Bidirectional One-to-One Mapping​

Enabling Two-Way (Bidirectional) Mapping​

  • If you want both Department and Employee to know about their association, you can make the mapping bidirectional.
  • This is useful when:
    • You want to access the department managed by an employee.
    • And the manager of a given department.

Step 1: Define Mapping in Employee Entity​

  • In the Employee entity, define a field that points back to the Department:
@OneToOne(mappedBy = "manager")
private Department managedDepartment;
  • Here
    • mappedBy = "manager" refers to the field in the Department entity.
    • This tells Hibernate that the Department entity owns the relationship.
    • Hence, no additional foreign key column will be created in the Employee table.

Why Use mappedBy?​

  • Without mappedBy, Hibernate will create a separate foreign key in the Employee table.
  • But since the relationship is already defined in the Department entity, using mappedBy avoids duplication.
  • This ensures only one side (Department) owns and maintains the mapping in the database schema.

Result in Database Schema​

  • After running the application:
    • The Employee table remains unchanged.
    • It does not get a new column or constraint for the Department.
    • The foreign key still exists only in the Department table (manager_id or renamed as department_manager if @JoinColumn is used).
  • This proves that bidirectional mapping is purely object-level, not schema-level on both sides.

Verifying Employee Table​

  • Upon refreshing the database:
    • The Employee table has no new foreign key related to departments.
    • Only contains its primary key constraint on id.

Practical Behavior​

  • Now you can do the following:
    • From Department, access the manager (Employee).
    • From Employee, access the managedDepartment (Department).
  • Hibernate handles the underlying SQL joins automatically.

Hibernate Responsibility​

  • When accessing managedDepartment from an Employee object:
    • Hibernate will automatically generate the SQL query to fetch the related Department.
    • Developers do not need to write manual queries for this association.

Assign Manager to Department​

Controller Setup​

  • We already have endpoints to create a department and fetch a department by ID.
  • Now, we’ll add a PUT mapping to assign a manager (an employee) to a department.

Define PUT Mapping in Controller​

@PutMapping("/departments/{departmentId}/manager/{employeeId}")
public Department assignManagerToDepartment(
    @PathVariable Long departmentId,
    @PathVariable Long employeeId
) {
    return departmentService.assignManagerToDepartment(departmentId, employeeId);
}

Implement Service Method​

  • Create a new method in the DepartmentService to handle assignment:
public Department assignManagerToDepartment(Long deptId, Long empId) {
    Optional<Department> optionalDept = departmentRepository.findById(deptId);
    Optional<Employee> optionalEmp = employeeRepository.findById(empId);

    return optionalDept.flatMap(dept ->
        optionalEmp.map(emp -> {
            dept.setManager(emp);
            return departmentRepository.save(dept);
        })
    ).orElse(null); // Can be replaced with proper exception handling
}

Dependency Injection​

  • Inject both DepartmentRepository and EmployeeRepository in the service class.
@Autowired
private DepartmentRepository departmentRepository;

@Autowired
private EmployeeRepository employeeRepository;

Why Optional and flatMap?​

  • Avoids multiple if-else checks.
  • Ensures both Department and Employee exist before assignment.
  • Clean functional style using Optional.map() and flatMap().

Testing with Postman​

Recursive Issue in Bidirectional One-to-One Mapping & Solutions Using Jackson​

Problem Identification​

  • When fetching a Department by ID, the response includes:
    • Department → Manager (Employee)
    • Manager → Managed Department (same Department)
    • Which again includes Manager → Department → and so on...
  • This creates an infinite recursive loop during JSON serialization.

Cause​

  • The bidirectional relationship between Department and Employee leads to recursive serialization.
  • This occurs due to Jackson (the JSON serialization library) trying to serialize both sides of the relationship.

Solution​

  • Use @JsonIgnore on the back-reference field to prevent recursion.
  • Add @JsonIgnore on Employee.getManagedDepartment() or the relevant field.
@JsonIgnore
@OneToOne(mappedBy = "manager")
private Department managedDepartment;

Post-Solution Behavior​

  • Fetching Department by ID now:

    • Returns the manager field correctly.
    • Does not return managedDepartment under the manager object.

  • Fetching Employee by ID:

    • Returns basic employee info.
    • managedDepartment is now excluded from JSON.

Creating a New API to Fetch Department Managed by a Manager​

New Endpoint Design​

  • Goal: Fetch the department managed by a specific employee.
  • Endpoint:
GET /departments/assigned-department-of-manager/{employeeId}

Controller Method​

@GetMapping("/departments/assigned-department-of-manager/{employeeId}")
public Department getAssignedDepartment(@PathVariable Long employeeId) {
    return departmentService.getAssignedDepartmentByManagerId(employeeId);
}

Service Method (Approach 1 – Via Employee)​

public Department getAssignedDepartmentByManagerId(Long empId) {
    return employeeRepository.findById(empId)
        .map(Employee::getManagedDepartment)
        .orElse(null);
}
  • ✅ Pros: Clean and readable.
  • ⚠️ Cons: Relies on employee-to-department mapping being populated.

package com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.entities;

import jakarta.persistence.*;
import lombok.*;

@Entity
@Data
@AllArgsConstructor
@NoArgsConstructor
@Table(name = "departments")
public class DepartmentEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String title;

    @OneToOne
    @JoinColumn(name = "department_manager")
    private EmployeeEntity manager;
}

package com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.services;

import com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.entities.DepartmentEntity;
import com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.entities.EmployeeEntity;
import com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.repositories.DepartmentRepository;
import com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.repositories.EmployeeRepository;
import org.springframework.stereotype.Service;

import java.util.Optional;

@Service
public class DepartmentService {

    private final EmployeeRepository employeeRepository;

    private final DepartmentRepository departmentRepository;
    public DepartmentService(EmployeeRepository employeeRepository, DepartmentRepository departmentRepository) {
        this.employeeRepository = employeeRepository;
        this.departmentRepository = departmentRepository;
    }

    public DepartmentEntity createNewDepartment(DepartmentEntity departmentEntity) {
        return departmentRepository.save(departmentEntity);
    }

    public DepartmentEntity getDepartmentById(Long id) {
        return departmentRepository.findById(id).orElse(null);
    }

    public DepartmentEntity assignManagerToDepartment(Long departmentId, Long employeeId) {
        Optional<DepartmentEntity> departmentEntity = departmentRepository.findById(departmentId);
        Optional<EmployeeEntity> employeeEntity = employeeRepository.findById(employeeId);

        return departmentEntity.flatMap(department ->
                employeeEntity.map(employee -> {
                    department.setManager(employee);
                    return departmentRepository.save(department);
                })).orElse(null);
    }

    public DepartmentEntity getAssignedDepartmentOfManager(Long employeeId) {
//        Optional<EmployeeEntity> employeeEntity = employeeRepository.findById(employeeId);
//        return employeeEntity.map(employee-> employee.getManagedDepartment()).orElse(null);

        EmployeeEntity employeeEntity = EmployeeEntity.builder().id(employeeId).build();

        return departmentRepository.findByManager(employeeEntity);
    }
}

package com.codingshuttle.tutorial.dataMapping.DataMappingTutorial.entities;

import com.fasterxml.jackson.annotation.JsonIgnore;
import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Builder;

@Data
