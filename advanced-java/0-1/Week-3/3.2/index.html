Hibernate and JPA

Hibernate ORM Mapping​

How tables in the database are mapped to Java classes using Hibernate.​

  • The Hibernate framework:
    • Works behind the scenes to map columns in database tables to fields in Java classes.
    • For example, a column like id in the database is automatically mapped to an id field in the Student Java class.
    • We, as a developer, do not need to manually map every field or column.
    • Hibernate handles this mapping automatically.
  • As seen in earlier sessions:
    • We didn’t manually map fields between the database and Java classes.
    • This was handled by Hibernate working in the background.
  • Why is Hibernate useful?
    • It simplifies database operations.
    • Developers don’t need to worry about:
      • Writing SQL queries.
      • Managing columns or table structure directly.
      • Instead, you just work with Java objects and fields.
      • Hibernate handles the conversion between Java fields and database columns.

Deeper Understanding of Hibernate, JPA & Spring Data JPA​

Hibernate Overview​

  • It’s important to understand the Hibernate framework in more detail.
  • Hibernate simplifies the interaction between Java objects and relational databases.
  • Hibernate is an ORM (Object-Relational Mapping) framework that:
    • Maps Java objects to database tables.
    • Converts data between Java classes and relational database tables.

Traditional Approach (Without Hibernate)​

  • You typically connect to a database (e.g., MySQL, SQL Server, MongoDB) using a JDBC driver.
  • With JDBC, you:
    • Manually write SQL queries in your Java code.
    • Pass those queries to the database.
    • Handle data manipulation yourself.
  • This approach requires managing:
    • SQL syntax.
    • Table structures.
    • Result sets manually.

Why Use Hibernate?​

  • Hibernate abstracts the low-level SQL and JDBC operations.
  • With Hibernate:
    • You don’t need to worry about writing SQL queries.
    • You work directly with Java objects (entities).
    • Data is automatically mapped between classes and tables.
  • Hibernate handles the conversion:
    • From database rows to Java objects.
    • From Java objects to database rows.

Hibernate = Implementation of JPA​

  • Hibernate is a concrete implementation of the JPA specification.
  • JPA (Java Persistence API):
    • Is a set of standards (interfaces) for object-relational mapping.
    • Defines how Java objects should be persisted in a relational database.
  • Hibernate implements JPA and provides:
    • The actual functionality for the abstract definitions of JPA.

JPA and Hibernate Annotations​

  • @Entity: Marks a Java class as a JPA entity (a database table).
  • @Column: Defines mapping between a field and a column in the table.
  • JPA defines what should be done, and Hibernate defines how it should be done.

Alternatives to Hibernate​

  • Hibernate is the most widely used ORM tool.
  • Alternatives include:
    • OpenJPA
    • EclipseLink
  • These alternatives also implement the JPA specification.

Advantages of Hibernate​

  • Provides:
    • ORM capabilities.
    • Optimized, scalable, and vendor-independent code.
  • Supports multiple databases (e.g., MySQL, PostgreSQL, MongoDB, etc.).
  • Reduces boilerplate code and simplifies data persistence.

Spring Data JPA​

  • Spring Data JPA is built on top of JPA and integrated into Spring Boot.
  • It further simplifies data access by providing repository-based architecture.
  • Uses interfaces like:
    • CrudRepository
    • JpaRepository

Benefits of Spring Data JPA​

  • Eliminates the need to write boilerplate queries.
  • You can:
    • Save, update, delete, and retrieve records using method names only.
    • Avoid writing actual SQL or JPQL queries for basic operations.
  • This is achieved using DAO (Data Access Object) methods provided by the repository interfaces.

Query Languages – JPQL vs SQL​

  • With JPA, you can write queries in JPQL (Java Persistence Query Language).
  • JPQL is database-independent and works with entity objects.
  • Hibernate:
    • Converts JPQL into native SQL or MongoDB queries depending on the database.
    • Also maps the results back into Java objects.

How It All Connects​

  • You can work at different layers depending on your project:
    • Spring Data JPA: Use predefined methods in repository interfaces.
    • JPA: Write custom JPQL queries.
    • Hibernate: Handle deeper customization and mapping.
  • Hibernate sits at the core, converting:
    • JPQL → SQL (or NoSQL)
    • Java objects → Database tables (or documents)
    • And vice versa.

What Does Persistence Mean in Programming (Java/JPA)?-​

  • Persistence – Simple Meaning:
    • Persistence means storing data in a way that it remains available even after the application stops.
  • In Other Words:
    • Data is persistent if it is not lost when:
    • he app is closed,
    • The server is restarted,
    • Or the computer is shut down.
  • Example:
    • If you store user details in RAM (memory) – they will be lost when the app stops.
    • If you store them in a database (like MySQL/PostgreSQL) – they are persisted.

Let's take a simple Java class (entity) and walk through how Spring Data JPA works from your Java code all the way to the database, step-by-step.

  • Example Class: Student
import jakarta.persistence.Entity;
import jakarta.persistence.Id;

@Entity
public class Student {
    @Id
    private Long id;
    private String name;
    private int age;

    // Getters and Setters
}

Step-by-Step Flow: Spring Data JPA to Database

  1. Entity Class
  • The @Entity annotation tells JPA/Hibernate that this class should be mapped to a table in the database.
  • Fields like id, name, and age will become columns.
  • Student.java → student table in the database.
  1. Spring Data JPA Repository Interface
import org.springframework.data.repository.CrudRepository;

public interface StudentRepository extends CrudRepository<Student, Long> {
}
  • CrudRepository provides built-in methods like:
    • save(), findById(), findAll(), deleteById() etc.
  • You don’t need to write SQL queries for these operations!
  1. Saving Data in Controller or Service
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/students")
public class StudentController {

    @Autowired
    private StudentRepository studentRepository;

    @PostMapping
    public Student createStudent(@RequestBody Student student) {
        return studentRepository.save(student);  // Step: Save student
    }
}
  • When a POST request comes in with student data, Spring calls:
studentRepository.save(student);
  1. What Happens Internally?

  • Spring Data JPA calls the save() method (from CrudRepository).

    • This is a proxy method, implemented by Hibernate.

  • Hibernate (ORM framework) checks:

    • Is this object new or existing?
    • It then translates this Java object into a SQL INSERT query, e.g.:
INSERT INTO student (id, name, age) VALUES (1, 'Ravi', 22);
  • Hibernate uses JDBC to send this SQL to the actual database (e.g., MySQL).
  • Database executes the query and persists the data in the table.
  1. How Hibernate Knows About the Database?
  • From your application.properties or application.yml:
spring.datasource.url=jdbc:mysql://localhost:3306/testdb
spring.datasource.username=root
spring.datasource.password=root
spring.jpa.hibernate.ddl-auto=update
  • This config tells Spring Boot:
    • How to connect to MySQL.
    • Hibernate should create/update the table if not present.
  1. Table Auto-Creation
  • Because of:
spring.jpa.hibernate.ddl-auto=update
  • Hibernate will automatically:
    • Check the Student class.
    • If the student table is missing, it will create:
CREATE TABLE student (
    id BIGINT PRIMARY KEY,
    name VARCHAR(255),
    age INT
);
  1. Getting Data Back
@GetMapping("/{id}")
public Student getStudent(@PathVariable Long id) {
    return studentRepository.findById(id).orElse(null);
}
  • This will result in Hibernate generating and running:
SELECT * FROM student WHERE id = 1;
  • Converts result into a Student object and returns it.

What is JPQL?

  • JPQL is like SQL, but instead of querying database tables, you query Java objects (entities).
  • It works on entity class names and field names, not table and column names.
  • SQL:
SELECT * FROM student WHERE name = 'Ravi';
  • JPQL:
SELECT s FROM Student s WHERE s.name = 'Ravi';

  • When Do You Use JPQL?

    • You use JPQL when:
    • You need custom queries beyond basic CRUD.
    • Spring Data JPA's built-in repository methods (like findById, save, etc.) are not enough.

  • Custom Query in Repository

import org.springframework.data.jpa.repository.Query;
import org.springframework.data.repository.CrudRepository;

public interface StudentRepository extends CrudRepository<Student, Long> {

    @Query("SELECT s FROM Student s WHERE s.name = :name")
    List<Student> findByName(@Param("name") String name);
}
  • Here, "SELECT s FROM Student s WHERE s.name = :name" is a JPQL query.
  • Student is the Java class, and s.name refers to the Java field.
  • You're not using the actual table name or SQL column.

What Happens Internally? Let’s break down what happens when findByName("Ravi") is called:

  • Spring Data JPA detects the @Query annotation.
  • It passes the JPQL query to Hibernate.
  • Hibernate:
    • Parses the JPQL.
    • Translates it into SQL (e.g., SELECT * FROM student WHERE name = 'Ravi').
  • Sends the SQL to the database using JDBC.
  • Converts the result back into Student Java objects.

Where JPQL Is NOT Needed

  • You don’t need JPQL when:
    • You use method name-based queries like:
List<Student> findByAge(int age);
  • Spring Data JPA parses the method name and builds the query automatically.

Hibernate​

What is Hibernate?​

  • Hibernate is a powerful, high-performance Object-Relational Mapping (ORM) framework widely used with Java.
  • It provides a framework for mapping object-oriented domain models to relational databases.
  • With Hibernate, you can:
    • Convert Java objects (entities) to database tables.
    • Automatically map relationships between tables and classes.

Object-Oriented to Relational Mapping​

  • In a relational database, data is stored in tables that are often related (e.g., Employee → Department).
  • In Java, we use objects and classes.
  • Hibernate handles the mapping between:
    • Java classes and objects (object-oriented model)
    • And tables and rows in the relational database.

Hibernate and JPA​

  • Hibernate is one of the implementations of JPA (Java Persistence API).
  • JPA is the standard specification for ORM in Java.
  • There are multiple implementations of JPA, including:
    • Hibernate
    • EclipseLink
    • OpenJPA

What is JPA (Java Persistence API)?​

  • JPA is a specification, not an implementation.
  • It provides:
    • A set of interfaces and annotations for ORM.
    • Rules and standards for how to persist Java objects to a database.
  • JPA does not perform any database operations itself — it relies on an implementation like Hibernate to do so.

Why Use JPA?​

  • JPA provides a common programming model for ORM in Java.
  • Benefits:
    • Developers can switch between different JPA providers (like Hibernate or EclipseLink) without changing their code.
    • It creates an abstraction layer over ORM implementations.

Why Hibernate is Preferred​

  • Hibernate is the most widely used JPA provider because it:
    • Implements all features of JPA.
    • Adds additional features and optimizations beyond JPA.
    • Helps write optimized queries and manage efficient table structures.
  • Hibernate provides:
    • Powerful caching
    • Lazy and eager loading
    • Advanced query capabilities
    • Integration with various databases

Hibernate DDL Auto Configuration​

The property spring.jpa.hibernate.ddl-auto defines how Hibernate handles schema updates:

  • Use update, create, or create-drop in development.
  • Use none or validate in production. | Value | Description | | ------------- | --------------------------------------------------------------------------- | | none | No schema change. Hibernate will not interact with the schema. | | validate | Validates the schema with the entity definitions. Throws error if mismatch. | | update | Updates the schema to match the entity without dropping existing data. | | create | Creates a new schema each time the app starts. Drops existing data. | | create-drop | Creates schema on startup and drops it on shutdown (useful in dev only). |

These properties help during development and debugging:

spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true
PropertyPurpose

Hibernate Dialect​

spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MySQL8Dialect
  • Dialect tells Hibernate how to convert JPQL into the correct SQL for the target database.
  • Common dialects:
    • MySQL5Dialect
    • MySQL8Dialect
    • PostgreSQLDialect
    • OracleDialect, etc.
  • Modern Spring Boot + Hibernate can auto-detect dialect based on the JDBC driver, so it's optional.

Hibernate Entity Configuration and application.properties Setup​

Where to Add Hibernate Configurations?​

  • All Hibernate and JPA configurations are added in the application.properties file of the Spring Boot project.
  • Example configurations:
application.properties

# Auto-create or update schema based on entity classes
spring.jpa.hibernate.ddl-auto=update

# Show SQL queries generated by Hibernate in the console
spring.jpa.show-sql=true

# Format SQL queries for better readability
spring.jpa.properties.hibernate.format_sql=true
  • Note: Dialect (spring.jpa.properties.hibernate.dialect) is optional. Spring Boot auto-detects it based on the JDBC driver.

@Entity Annotation​

  • Marks a Java class as a JPA entity that Hibernate should persist to the database.
  • Hibernate will automatically create a corresponding table for the entity in the database.
@Entity
public class Product { ... }

@Table Annotation (Optional)​

  • Allows specifying the table name and configuration explicitly.
@Table(name = "product_table")
  • You can define additional configurations within @Table, like unique constraints.

@Id and @GeneratedValue​

  • @Id: Marks a field as the primary key.
  • @GeneratedValue: Tells Hibernate to auto-generate the primary key value.
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
  • Strategy IDENTITY lets the database handle ID generation (like MySQL's AUTO_INCREMENT).

@Column Annotation​

  • Used to customize the database column for a field:
@Column(name = "title_x", nullable = false, length = 100)
private String title;
AttributeDescription
nameChanges the column name in the table.
nullable=falseMakes the column NOT NULL in the database.
length=20Restricts maximum length of string fields.
unique=trueMakes the column values unique.
insertableControls whether the column is included in SQL INSERTs.
updatableControls whether the column is included in SQL UPDATEs.
  • Naming Convention:
    • Java fields like createdAt → automatically converted to created_at in the table.

@CreationTimestamp and @UpdateTimestamp​

  • Provided by Hibernate (org.hibernate.annotations package).
  • Automatically populate the timestamps when an entity is created or updated.
@CreationTimestamp
private LocalDateTime createdAt;

@UpdateTimestamp
private LocalDateTime updatedAt;
AnnotationDescription
@CreationTimestampAuto-sets the value when the record is inserted.
@UpdateTimestampAuto-updates the value every time the entity is updated.

Understanding @Table Annotation in JPA (Hibernate)​

  • The @Table annotation provides metadata about how the entity maps to a specific table in the database. You can customize various aspects of the table using this annotation.

Basic Syntax of @Table​

@Entity
@Table(
    name = "product_table",
    catalog = "inventory_catalog",
    schema = "hr",
    uniqueConstraints = { ... },
    indexes = { ... }
)
public class Product {
    ...
}

Custom Table Name​

  • Use the name parameter to change the default table name (which is otherwise the same as the class name).
@Table(name = "product_table")

Catalog and Schema​

  • Catalog: Groups tables under a specific catalog.
  • Schema: Logical grouping of tables, commonly used in enterprise databases.
@Table(catalog = "inventory_catalog", schema = "hr")

Unique Constraints​

  • Ensures one or more columns contain unique values across all rows. Single Column Unique Constraint
@Table(
    uniqueConstraints = @UniqueConstraint(
        name = "sku_unique",
        columnNames = "sku"
    )
)
  • This ensures that no two products can have the same SKU.

Multiple Column Unique Constraint

@Table(
    uniqueConstraints = {
        @UniqueConstraint(
            name = "title_price_unique",
            columnNames = { "title", "price" }
        )
    }
)

  • This ensures that no two products can have the same combination of title and price.

  • Example:

    • ✅ Allowed: Parle + ₹50 and Parle + ₹20
    • ❌ Not Allowed: Two entries with Parle + ₹20

Indexes for Performance Optimization​

  • Improves the speed of read queries by creating an index on specific columns.

  • Indexes use internal data structures like B-trees or Hash Tables for faster lookup.

  • Example Index on sku

@Table(
    indexes = {
        @Index(name = "sku_index", columnList = "sku")
    }
)
  • If you have renamed a column using @Column(name = "title_x"), use title_x instead of title in your index/constraints:
@Index(name = "title_x_index", columnList = "title_x")

Multiple Indexes​

  • You can define more than one index:
@Table(
    indexes = {
        @Index(name = "sku_index", columnList = "sku"),
        @Index(name = "title_price_index", columnList = "title_x, price")
    }
)

Why Application Failed Initially​

Required MySQL Configuration in application.properties​

spring.datasource.url=jdbc:mysql://localhost:3306/test?useSSL=false
spring.datasource.username=root
spring.datasource.password=root
  • url: Points to the local MySQL server, port 3306, and database test.
  • useSSL=false: Since this is a development environment, SSL is disabled.

Table Schema Generated by Hibernate​

  • Hibernate log showed the following column definitions:
    • id → BIGINT, NOT NULL, AUTO_INCREMENT (Primary Key)
    • created_at → DATETIME
    • price → DECIMAL
    • quantity → INT
    • sku → VARCHAR(20) (Defined with length = 20)
    • title → Renamed to title_x using @Column(name = "title_x")

Indexes Created​

  • Hibernate created the following indexes:
    • sku_index: Explicitly defined via @Index.
    • sku_unique: Created automatically due to a unique constraint on the sku column.
    • title_price_unique: Created due to the composite unique constraint on title_x and price.

Constraint Definitions in Logs​

  • Hibernate internally executed the following:
    • ALTER TABLE to drop duplicate or unnecessary indexes.
    • ADD CONSTRAINT for:
      • sku_unique → Ensures all SKUs are unique.
      • title_price_unique → Ensures no duplicate title and price pair exists.

Hibernate Data Initialization with data.sql & Key Features of JPA​

Using data.sql for Initial Data Insertion​

  • Created a file named data.sql in the classpath.
  • Inside this file, SQL INSERT statements can be added to pre-populate the database.
  • Required Spring Boot configurations in application.properties:
spring.jpa.defer-datasource-initialization=true
spring.sql.init.mode=always
spring.sql.init.data-locations=classpath:data.sql
  • These settings:
    • Allow Spring Boot to initialize the database with data.sql.
    • Ensure the script is always run during app startup.

Handling Unique Constraint Errors​

  • Initially encountered an error: "Duplicate entry for key 'sku_unique'"
  • Cause: Data already existed in DB and sku was defined as UNIQUE.
  • Solution:
    • Changed ddl-auto property from update to create to drop the old table and recreate it.
spring.jpa.hibernate.ddl-auto=create
  • Not recommended for production; only use create in development mode.

Observing Hibernate Behavior on Application Restart​

  • On restart:
    • Hibernate executed: DROP TABLE IF EXISTS product_table
    • Then: CREATE TABLE product_table
    • Finally, ran the SQL commands from data.sql to insert the data.

Validating Inserted Data​

  • After running the application:
    • Verified that the inserted products were present in the table.
    • Added one more product (Coca-Cola) to verify fresh insertion.
INSERT INTO product_table (sku, title_x, price, quantity, created_at, updated_at)
VALUES ('coke1124', 'Coca-Cola', 10, 100, NOW(), NOW());
  • Confirmed that this entry was also added successfully.
  • Demonstrated how data.sql is useful for initializing test data during development.

Key Features of JPA​

Entity Management​

  • Provides a standardized way to define and manage entities.
  • Uses annotations like @Entity, @Column, @Id, @GeneratedValue.

JPQL (Java Persistence Query Language)​

  • JPA introduces JPQL, an object-oriented query language.
  • Repositories use JPQL internally to query entities.
  • JPQL is then converted to native SQL by Hibernate.

Transaction Support​

  • Built-in transaction management.
  • Makes it easy to wrap multiple DB operations in a single transactional context.

Entity Relationships​

  • JPA supports defining relationships like:
    • @OneToOne
    • @OneToMany
    • @ManyToOne
    • @ManyToMany
  • Useful for mapping complex business models like:
    • Product → Category
    • Order → Products
  • These relationships will be covered in upcoming lectures.

import lombok.Builder;

import lombok.NoArgsConstructor;

import org.hibernate.annotations.UpdateTimestamp;

import java.math.BigDecimal;
import java.time.LocalDateTime;

@Data
@AllArgsConstructor
@NoArgsConstructor
@Builder
@Entity
@Table(
        name = "product_table",
        uniqueConstraints = {
                @UniqueConstraint(name = "title_price_unique", columnNames = {"title_x", "price"})

    private String title;

    private BigDecimal price;

    private Integer quantity;

    @CreationTimestamp
    private LocalDateTime createdAt;

    @UpdateTimestamp
    private LocalDateTime updatedAt;
}
