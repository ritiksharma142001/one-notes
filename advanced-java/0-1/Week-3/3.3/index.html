Spring Data JPA & Dynamic Query Methods

Summary: Introduction to Spring Data JPA​

  • Recap of Previous Class:
    • Learned how Hibernate works and how it adheres to JPA specifications.
  • What is Spring Data JPA?
    • Spring Data JPA is built on top of JPA.
    • It helps simplify data access layers using predefined repositories.
    • Eliminates the need to write boilerplate DAO (Data Access Object) methods.
  • Why Use Spring Data JPA?
    • Abstracts the creation of data access methods.
    • Provides various repository interfaces like:
      • CrudRepository
      • JpaRepository
      • PagingAndSortingRepository, etc.
    • These interfaces simplify common CRUD operations.
  • Benefits of Repositories:
    • Reduces the need to write SQL or JPQL queries.
    • Database operations (create, update, delete) can be performed without writing a single query.
    • Enhances productivity by hiding the complexity of SQL under abstraction layers.
  • From JPA to Spring Data JPA:
    • JPA provides a standard specification and JPQL (Java Persistence Query Language).
    • Spring Data JPA goes further:
      • Allows method-name-based query generation (e.g., findByTitleAndPrice()).
      • Even JPQL is autogenerated based on method names.
      • We still have the option to write custom JPQL or native SQL queries if needed.
  • Magic Behind Spring Data JPA:
    • Just define method names correctly in your repository interface.
    • Spring Data JPA will generate the underlying queries automatically.
  • Spring Data JPA is Part of a Larger Family:
    • It belongs to the broader Spring Data ecosystem.
    • Provides a higher-level abstraction and convenient data access utilities.
    • Reduces the amount of code needed to interact with the database.
  • Flexibility:
    • You can still write:
      • Custom JPQL queries using @Query annotation.
      • Native SQL queries when needed.

Spring Data JPA Repositories​

Repository Hierarchy in Spring Data JPA:​

  • We've previously seen the CrudRepository, which provides basic CRUD operations.
  • Spring Data JPA offers additional repository interfaces to extend functionality without custom logic.

Pagination and Sorting Support:​

  • PagingAndSortingRepository is used for implementing pagination and sorting features.
  • No need to write custom pagination logic—this repository handles it.
  • Detailed discussion on this will be covered in the next lecture.

Focus of This Class – JpaRepository:​

  • JpaRepository is the primary interface in focus for today.
  • It extends both CrudRepository and PagingAndSortingRepository.

Advantages of JpaRepository:​

  • Provides advanced features such as:
    • findAll() returning a List instead of an Iterable (unlike CrudRepository).
    • saveAll(), flush(), getOne(), and many more utility methods.
  • Improves ease of use by returning more developer-friendly data structures like List.

Repository Interface Inheritance Flow:​

  • JpaRepository<T, ID>:
    • extends PagingAndSortingRepository<T, ID>
    • which itself extends CrudRepository<T, ID>
    • Additionally, JpaRepository also implements QueryByExampleExecutor<T>

Implementation Responsibility:​

  • You don't need to write the implementation of repository interfaces.
  • Spring Data JPA automatically generates concrete implementations of these interfaces at runtime.
  • This is handled by Spring under the hood using dynamic proxies and reflection.

Implementing a ProductRepository using Spring Data JPA​

Step-by-Step Repository Setup​

  • Package Creation:
    • A new package named repositories was created to contain all repository interfaces.
  • Repository Interface Creation:
    • Inside this package, a new interface ProductRepository was created.
  • Entity Naming Clarification:
    • Renamed the entity to ProductEntity for better clarity between Entity and DTO.

Extending Spring Data Repository​

  • Interface Declaration:
    • ProductRepository is declared as an interface — no need to write its implementation manually.
  • Extending JpaRepository:
    • The interface extends JpaRepository<ProductEntity, Long>.
      • ProductEntity: The entity class representing the table.
      • Long: The type of the primary key (ID).
  • Why JpaRepository:
    • By extending JpaRepository, we automatically inherit:
      • CRUD operations from CrudRepository
      • Pagination and sorting from PagingAndSortingRepository
      • Additional advanced features from JpaRepository

Behind the Scenes: Who Implements the Methods?​

  • No Manual Implementation Required:
    • You only define the interface (ProductRepository). You don’t implement any method.
  • Spring's Internal Implementation:
    • Spring Boot provides runtime implementation through internal classes.
  • Key Implementing Class:
    • The actual implementation is done by the SimpleJpaRepository class.

How SimpleJpaRepository Works​

  • SimpleJpaRepository is a concrete class that implements all methods defined in JpaRepository.

  • Examples:

    • deleteById(), save(), findAll() — all implemented internally.
    • Spring automatically injects this implementation using dynamic proxies and reflection.

  • You can view this in your IDE:

    • Navigate to JpaRepository → SimpleJpaRepository to explore the method definitions.

Key Features of Spring Data JPA​

Repository Abstraction​

  • Spring Data JPA provides a powerful abstraction for repositories.
  • Interfaces such as JpaRepository, CrudRepository, etc., expose predefined CRUD methods (e.g., findAll(), save(), deleteById()).
  • No need to write boilerplate DAO code manually

Custom Query Methods​

  • You can define custom finder methods based on naming conventions.
  • Spring Data JPA will automatically generate the corresponding SQL/HQL queries at runtime.
  • Example: Find Products by Price
List<ProductEntity> findByPriceGreaterThan(BigDecimal price);
  • This method will automatically generate SQL like:
SELECT * FROM product WHERE price > ?

Pagination and Sorting Support​

  • Built-in support for pagination and sorting via:
    • PagingAndSortingRepository
    • Pageable, Sort objects
  • Methods like:
Page<ProductEntity> findAll(Pageable pageable);

Query Derivation from Method Names​

  • Spring Data JPA can derive queries from method names following a specific pattern.
  • Example:
List<ProductEntity> findByTitle(String title);
  • Automatically generates:
SELECT * FROM product WHERE title = ?

Creating and Using Custom Finder Methods​

Step-by-step:

  1. Trying to Use an Undefined Method:
productRepository.findByTitle("Pepsi");
  • IDE shows an error because this method is not yet defined.
  1. Auto-Create Method (via IDE shortcut or manually):
List<ProductEntity> findByTitle(String title);
  1. Run Test Case:
  • Observe that Hibernate automatically generates the following SQL:
SELECT * FROM product WHERE title = ?
  1. Important Note:
  • The method name must exactly match the field name in the entity (title).
  • Using incorrect names like findByTitles() (when field is title) will result in:
Error: No property 'titles' found for type 'ProductEntity'

Best Practices for Method Naming​

  • ✅ Use exact entity property names (e.g., findByTitle, findByPriceGreaterThan).
  • ❌ Avoid typos or incorrect pluralizations (e.g., findByTitles when field is title).
  • Spring parses method names and maps them to properties and operators:
    • And, Or, Between, LessThan, GreaterThan, etc.

Spring Data JPA — Method Naming Rules & Query Keywords​

1. Rules for Creating Derived Query Methods​

  • Spring Data JPA allows query creation based on method names. These method names are parsed to automatically generate SQL queries.
  • Basic Structure:
ReturnType findBy<FieldName><Operator>(Parameters)

2. Query Method Components​

  • Each derived query method typically consists of:

  • Return Type:

    • ProductEntity
    • Optional<ProductEntity>
    • List<ProductEntity>
    • Other supported types

  • Query Subject:

    • Action being performed: findBy, readBy, getBy, queryBy
    • You can also use: countBy, existsBy, deleteBy, removeBy

Predicate/Filter Condition:​

  • Field and operator part of the method, e.g.:
findByCreatedAtBetween(...)
  • Input Parameters:
    • Must match the field types in the entity
    • Example: if field is LocalDateTime, parameters should be LocalDateTime too

3. Common Operators in Method Names​

OperatorExampleGenerated SQL Equivalent
BetweenfindByCreatedAtBetween(start, end)WHERE created_at BETWEEN ? AND ?
AfterfindByCreatedAtAfter(date)WHERE created_at > ?
BeforefindByCreatedAtBefore(date)WHERE created_at < ?

IsNotNullfindByTitleIsNotNull()WHERE title IS NOT NULL

4. Query Modifiers​

  • Limit Results Use First, Top, followed by number:
findFirst2ByOrderByPriceDesc()
findTop5ByQuantityGreaterThan(10)
  • Distinct Results Use Distinct to avoid duplicates:
findDistinctByTitle(String title)

5. Examples of Custom Queries Using Method Names​

  • Find all products created between two dates:
List<ProductEntity> findByCreatedAtBetween(LocalDateTime start, LocalDateTime end);
  • Find products created after a certain date:
List<ProductEntity> findByCreatedAtAfter(LocalDateTime date);
  • Note: Use camelCase for field names, matching exactly with entity field names:
  • If the field is createdAt, use CreatedAt in method names.

6. Important Naming Conventions​

-✅ findBy, readBy, queryBy, getBy → All generate SELECT queries -✅ countBy → Returns number of rows -✅ existsBy → Returns true/false if data exists -✅ deleteBy, removeBy → For deletion operations -❌ Invalid naming (e.g., findByTitles when field is title) leads to errors:

Error: No property 'titles' found for type 'ProductEntity'

7. Official Reference: Spring Data JPA Keywords​

  • You can find the full list of keywords in the Spring Data JPA documentation
  • Keywords include:
    • After, Before, Between, IsNull, IsNotNull
    • Like, NotLike, Containing, In, NotIn
    • True, False, IgnoreCase, Distinct, OrderBy
    • Logical operators: And, Or

8. Test Example: Using findByCreatedAtAfter(...)​

List<ProductEntity> findByCreatedAtAfter(LocalDateTime date);
  • Test Case:
List<ProductEntity> result = productRepository.findByCreatedAtAfter(
    LocalDateTime.of(2024, 1, 1, 0, 0)
);
  • Returns all products created after 01-Jan-2024 00:00

Advanced Spring Data JPA Query Methods​

Using after in Queries​

  • To fetch entities created after a certain date and time, use the after keyword
List<ProductEntity> findByCreatedAtAfter(LocalDateTime date);
  • Example usage:
LocalDateTime date = LocalDateTime.of(2024, 1, 1, 0, 0);
List<ProductEntity> result = productRepository.findByCreatedAtAfter(date);
  • If no data is returned, ensure that:
    • The database table contains data.
    • The createdAt field is populated.

Combining Filters Using And​

  • You can filter using multiple fields with And:
List<ProductEntity> findByQuantityAndPrice(int quantity, BigDecimal price);
  • Example usage:
List<ProductEntity> result = productRepository.findByQuantityAndPrice(4, new BigDecimal("2345"));
  • This query will return products where quantity = 4 and price = 2345.

Using Comparison Operators (GreaterThan, LessThan)​

  • You can apply numeric comparisons:
List<ProductEntity> findByQuantityGreaterThanAndPriceLessThan(int quantity, BigDecimal price);
  • This will return all products where:
    • quantity > ?
    • price < ?

Using Logical Operator Or​

  • Replace And with Or to apply alternative filters:
List<ProductEntity> findByQuantityGreaterThanOrPriceLessThan(int quantity, BigDecimal price);
  • This retrieves products that match either condition.

Using In Operator​

Checks whether a field value exists in a given list:

List<ProductEntity> findByIdIn(List<Long> ids);

Using Like for Pattern Matching​

  • Searches based on SQL LIKE pattern:
List<ProductEntity> findByTitleLike(String title);
  • Usage:
findByTitleLike("%choco%")
  • % is the wildcard representing any number of characters.

Using Containing for Simpler Like​

  • Equivalent to LIKE %...% but more readable:
List<ProductEntity> findByTitleContaining(String title);
  • You do not need to pass %; Spring/Hibernate adds it automatically.

Case-Insensitive Queries with IgnoreCase​

  • Add IgnoreCase to perform case-insensitive matching:
List<ProductEntity> findByTitleContainingIgnoreCase(String title);
  • Converts both DB value and input to uppercase for comparison.

Getting a Single Entity with Unique Fields​

  • If you're querying on a unique combination (e.g., title + price), use:
ProductEntity findByTitleAndPrice(String title, BigDecimal price);
  • Or use Optional to avoid null checks:
Optional<ProductEntity> findByTitleAndPrice(String title, BigDecimal price);

Writing a Test for a Single Entity Query​

  • Example unit test:
@Test
void getSingleFromRepository() {
    Optional<ProductEntity> product = productRepository.findByTitleAndPrice("Pepsi", new BigDecimal("50"));
    product.ifPresent(System.out::println);
}
  • Ensures you fetch a single product using unique fields.

| Keyword | Description |​

| ---------------------------- | -------------------------------- | | IsNull, IsNotNull | Checks for null values | | True, False | Checks for boolean fields | | IsEmpty, IsNotEmpty | For collections or string fields | | ExistsBy... | Returns boolean if entity exists | | IgnoreCase | Case-insensitive match | | StartingWith, EndingWith | Prefix/suffix search | | OrderBy...Asc/Desc | To sort the result set |

Spring Data JPA – Custom Queries & Optional Handling​

Handling Optional and Fetching a Single Entity​

  • A BigDecimal value (e.g., 23.45) was used for price matching.
  • To handle Optional<ProductEntity>, the ifPresent() method was used to print the value:
productEntity.ifPresent(System.out::println);
  • Java's method inference feature was used for cleaner syntax.
  • The query failed initially because the test that inserts the data didn’t run before the fetch test.
  • Test cases are independent, and data from one is not shared with another unless explicitly handled.
  • After inserting "Pepsi" with the price 14.4, the query returned the correct entity.
  • Changing the title to "Pepsi Coo" (which didn’t exist) returned no result, validating the single-fetch mechanism.

Caution with Optional​

  • When using Optional, ensure the query returns exactly one result.
  • If multiple results are returned, it will throw an error because an Optional cannot hold a list.
  • If there’s a possibility of multiple results, prefer using a List<ProductEntity> as the return type.

Limitations of Derived Query Methods​

  • Method name-based queries in Spring Data JPA are powerful but limited for complex logic:
    • Cannot easily handle aggregate queries (e.g., SUM, AVG).
    • Not suitable for nested joins or subqueries.
  • For such use cases, custom queries using @Query and JPQL are preferred.

Writing Custom Queries Using JPQL​

  • JPQL (Java Persistence Query Language) syntax is similar to SQL but operates on Java entity names and fields, not table names.
  • Example custom query using positional parameters:
@Query("SELECT e FROM ProductEntity e WHERE e.title = ?1 AND e.price = ?2")
Optional<ProductEntity> findByTitleAndPrice(String title, BigDecimal price);
  • Example using named parameters:
@Query("SELECT e FROM ProductEntity e WHERE e.title = :title AND e.price = :price")
Optional<ProductEntity> findByTitleAndPrice(@Param("title") String title, @Param("price") BigDecimal price);
  • Make sure:
    • Entity name (ProductEntity) is used, not table name (product).
    • Field names in the entity are correctly used (title, price).

Query Result Types​

  • You can modify the query to fetch specific fields instead of the whole entity:
@Query("SELECT e.title FROM ProductEntity e WHERE e.title = :title AND e.price = :price")
String findTitleOnly(...);

package com.jpaTut.jpatuts.entities;

import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Builder;
import lombok.Data;
import lombok.NoArgsConstructor;
import org.hibernate.annotations.CreationTimestamp;
import org.hibernate.annotations.UpdateTimestamp;

import java.math.BigDecimal;
import java.time.LocalDateTime;

@Data
@AllArgsConstructor
@NoArgsConstructor
@Builder
@Entity
@Table(
        name = "product_table",
        uniqueConstraints = {
                @UniqueConstraint(name = "title_price_unique", columnNames = {"title_x", "price"})
        },
        indexes = {
                @Index(name = "sku_index", columnList = "sku")
        }
)
public class ProductEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 20)
    private String sku;

    @Column(name = "title_x")
    private String title;

    private BigDecimal price;

    private Integer quantity;

    @CreationTimestamp
    private LocalDateTime createdAt;

    @UpdateTimestamp
    private LocalDateTime updatedAt;
}
