Javscript Web APIs and Working of Javascript Event Loop

  • Javascript Visualization
  • Notes

What are Web APIs ?​

  • A set of functions which are not directly provided by javascript.
  • These are provided by runtime like directly by browser or backend environment like NodeJS.
  • fetch()

setTimeout()​

  • function executed after 2000 sec.
  • setTimeout() will give id.

code-1​

console.log("Code Starting")
setTimeout(()=>{
    console.log("Inside the timeout")
}, 3000)

console.log("Code Ended")

code-2​

console.log("Code Starting");
const id = setTimeout(() => {
  console.log("Inside the timeout");
}, 3000);

console.log(id);

console.log("Code Ended");

code-3​

console.log("Code Starting");
const id = setTimeout(() => {
  console.log("Inside the timeout");
}, 3000);

console.log(id);

clearInterval(id) // setTimeout won't be executed

console.log("Code Ended");

setInterval()​

  • execute at regular interval multiple times.
  • setInterval() will give id.

code-1​

function showTime(){
    const time = new Date()
    console.log(time.toISOString())
}
showTime()

setInterval(showTime, 2000)

code-2​

  • using clearInterval(), we can stop the or clear interval anytime.
let i=0
function showTime(){
    const time = new Date()
    console.log(time.toISOString())
    i++
    if(i == 5) clearInterval(id)
}
showTime()

const id = setInterval(showTime, 2000)

JavaScript Execution Model (Call Stack, Event Queue, and Event Loop)​

Core Concepts:

  • Call Stack:
    • Executes synchronous code (line by line).
    • Follows LIFO (Last-In-First-Out) principle.
  • Web APIs (Browser APIs):
    • Handle asynchronous tasks like setTimeout, fetch, DOM events, etc.
  • Event Queue (Task Queue):
    • Stores callback functions waiting to be executed after async tasks finish.
  • Event Loop:
    • Constantly monitors the Call Stack and Event Queue.
    • Responsible for pushing callbacks from the Event Queue into the Call Stack.

Step-by-Step Execution Flow:

  • JavaScript starts executing a function → it gets pushed onto the Call Stack.
  • If the function contains an async task (e.g., setTimeout, fetch):
  • It is sent to the Web APIs for processing.
  • Once the async task completes:
  • The callback is placed into the Event Queue.
  • The Event Loop checks continuously:
    • Is the Call Stack empty?
    • ✅ Yes → Move the next callback from the Event Queue into the Call Stack for execution.
    • ❌ No → Wait until it becomes empty.

Concepts​

Event Loop

  • JavaScript is Single Threaded.
  • The event loop is like a manager who checks:
    • “Is the Call Stack (where code runs) empty?”
    • If yes, it takes a task from a queue and puts it into the Call Stack to run.
  • It connects everything: synchronous code, Web APIs, and the queues.

Web APIs

  • These are features provided by the browser (not JavaScript itself), like:
    • setTimeout, fetch, DOM events, etc.
  • When you call one, it’s sent to the Web API section to run in the background.
  • After finishing, the result (callback) is sent to a queue.

Callback Queue (Task Queue or Macro Task Queue)

  • This queue holds regular async callbacks, like:
    • setTimeout, setInterval, click, fetch.then()
  • The event loop picks from here only when the Call Stack is empty.

Microtask Queue

  • This queue is more important than the callback queue.
  • It stores microtasks, like:
    • Promise.then, Promise.catch, async/await results, queueMicrotask
  • After each task in the Call Stack finishes, the event loop:
    • Checks the Microtask Queue first and runs everything in it,
    • Then moves to the Callback (Macro) Queue.

How All Work Together (Simple Flow):

  1. JS code runs in the Call Stack.
  2. Async code (like setTimeout, fetch, Promise) goes to Web APIs.
  3. After finishing:
    • Promise results → Microtask Queue
    • setTimeout/click → Callback Queue
  4. The Event Loop:
    • Waits for Call Stack to be empty,
    • Runs all Microtasks first,
    • Then takes one task from the Callback Queue.

code-1​

console.log("1️⃣ Synchronous - Start");

// Synchronous Code
function syncFunc() {
  console.log("2️⃣ Synchronous - Inside syncFunc");
}
syncFunc();

// setTimeout (Macrotask)
setTimeout(() => {
  console.log("6️⃣ setTimeout - Macrotask");
}, 0);

// Promise (Microtask)
Promise.resolve()
  .then(() => {
    console.log("4️⃣ Promise.then - Microtask");
    return Promise.reject("Oops!");
  })
  .catch((err) => {
    console.log("5️⃣ Promise.catch - Microtask:", err);
  });

// async/await
async function asyncFunc() {
  console.log("3️⃣ async/await - Before await");
  await Promise.resolve(); // This becomes a microtask
  console.log("7️⃣ async/await - After await (Microtask)");
}
asyncFunc();

console.log("8️⃣ Synchronous - End");

Breakdown:

  • All synchronous code runs first (1️⃣ 2️⃣ 3️⃣ 8️⃣)
  • Then microtasks: Promise.then, catch, await continue (4️⃣ 5️⃣ 7️⃣)
  • Finally, macrotasks like setTimeout run (6️⃣)

code-2​

console.log("1 - Start");

async function asyncFunc() {
  console.log("2 - Inside asyncFunc");
  await Promise.resolve();
  console.log("5 - After await");
}

asyncFunc();

console.log("3 - After calling asyncFunc");

Promise.resolve().then(() => {
  console.log("4 - In Promise.then");
});

JavaScript Execution Order

  • Run all synchronous code
  • JavaScript starts running code in the Call Stack (top to bottom):

  1. console.log("1 - Start") → ✅ Prints: 1 - Start

  2. Define asyncFunc() → nothing runs yet

  3. asyncFunc() is called

    • Enters the function
    • Logs: 2 - Inside asyncFunc
    • Hits await Promise.resolve()
    • What happens here:
      • Execution pauses
      • The rest of the function (console.log("5 - After await")) is put into the microtask queue to run after the Call Stack is empty

  4. Back in main code: console.log("3 - After calling asyncFunc") → ✅ Prints: 3 - After calling asyncFunc

  5. Promise.resolve().then(...)

    • The .then() part goes into the microtask queue
